# VIM

## 启动

```
vim <文件名>   # 打开文件，不存在则新建
```



## 三种模式

- **正常模式**：启动后默认，用于导航和操作（按 `Esc` 返回）
- **插入模式**：输入文本（按 `i`/`a`/`o` 等进入）
- **命令行模式**：执行命令（按 `:` 进入）

## 正常模式下常用操作

### 光标移动

```
h j k l        ← ↓ ↑ →
w / b          下一个/上一个词首
e              词尾
0 / $          行首/行尾
gg / G         文件首/文件尾
:n             跳转到第n行（命令行模式）
Ctrl+f / Ctrl+b 向下/向上翻页
```



### 编辑操作

```
i / a          光标前/后插入
I / A          行首/行尾插入
o / O          下方/上方新行插入
x              删除光标处字符
dd             删除当前行
dw             删除到词尾
yy             复制当前行
p / P          光标后/前粘贴
u              撤销
Ctrl+r         重做
.              重复上次操作
```



### 保存与退出（命令行模式）

```
:w              保存
:q              退出（未保存则提示）
:wq 或 :x       保存并退出
:q!             强制退出不保存
ZZ（正常模式）   保存并退出
```



### 查找与替换

```
/pattern       查找 pattern（按 n/N 下一个/上一个）
: s/old/new/g  当前行替换所有 old 为 new
: %s/old/new/g 全文替换
```



### 可视模式（选择文本）

```
v               进入字符选择模式
V               进入行选择模式
Ctrl+v          进入块选择模式
选择后按 y 复制，d 删除，p 粘贴
```



### 常用命令（命令行模式）

```
:set number / :set nu      显示行号
:set nonumber              隐藏行号
:syntax on                 开启语法高亮
:help <命令>               查看帮助
```

## **一、终端快捷键删除整行**

### **1. 基础快捷键（适用于大多数终端）**

- **Ctrl + U**：删除从光标位置到**行首**的所有字符，若光标在行尾则相当于删除整行
- **Ctrl + K**：删除从光标位置到**行尾**的所有字符，若光标在行首则相当于删除整行
- **Ctrl + Shift + K**：**直接删除光标所在整行**（Windows/Linux系统）
- **Cmd + Shift + K**：**直接删除光标所在整行**（Mac系统）
- **Ctrl + A + Ctrl + K**：先移动到行首，再删除整行，确保完全清除12

### **2. 高级快捷键技巧**

- **Ctrl + X, Ctrl + E**：在bash中打开多行文本编辑器，可删除特定行后保存
- **Alt + D**：删除光标右侧的一个单词，配合其他快捷键可快速清理行内容
- **Ctrl + Y**：粘贴之前删除的内容，可用于恢复误删或移动行1115

## **二、编辑器内删除整行命令**

### **1. Vim/Vi编辑器**

- **dd**：在命令模式下删除当前光标所在整行
- **行号d**：如`5d`表示删除第5行
- **:行号d**：在底线命令模式下，输入`:5d`删除第5行
- **:1, $ d**：删除整个文件的所有行18

### **2. 其他编辑器**

- **Sublime Text**：Ctrl + K, Ctrl + Backspace（Windows/Linux）或 Cmd + K, Cmd + Backspace（Mac）删除行首到光标内容
- **UltraEdit**：Ctrl + E 可以删除光标所在行
- **VS Code**：Ctrl + Shift + K（Windows/Linux）或 Cmd + Shift + K（Mac）一键删整行411

## **三、脚本化删除整行命令**

### **1. Sed命令（最常用）**

bash



```
sed -i '5d' filename.txt  # 删除第5行
sed -i '3,5d' filename.txt  # 删除第3到第5行
sed -i '/特定文本/d' filename.txt  # 删除包含特定文本的行
```

- **-i选项**：直接修改原始文件
- **多行删除**：`sed -i '5,10d' file.txt`删除第5到第10行212

### **2. Awk命令**

bash



```
awk 'NR!=5' filename.txt > temp.txt && mv temp.txt filename.txt  # 删除第5行
awk '!/特定文本/' filename.txt > temp.txt && mv temp.txt filename.txt  # 删除包含特定文本的行
```

- **NR变量**：记录行号，`NR!=5`表示不等于第5行
- **临时文件**：先输出到临时文件，再替换原文件712

### **3. Grep命令**

bash



```
grep -v '特定文本' filename.txt > temp.txt && mv temp.txt filename.txt  # 删除包含特定文本的行
```

- **-v选项**：反向选择，排除匹配的行12

## **四、实用技巧与注意事项**

### **1. 安全操作建议**

- **操作不可逆**：删除后无法恢复，请确保已备份重要文件
- **先测试再执行**：建议先不加`-i`选项测试效果，确认无误后再执行
- **谨慎使用**：特别是`sed -i`直接修改文件时，避免误删关键数据111

### **2. 高效技巧**

- **多行删除**：`for i in 2 4; do sed -i "${i}d" filename.txt; done`可删除不连续多行
- **组合使用**：`Ctrl + A`移动到行首 + `Ctrl + K`删除整行，确保完全清除
- **历史命令**：`history -d <行号>`可删除命令历史中的特定行1213

### **3. 系统差异**

- **Windows/Linux**：主要使用Ctrl键组合快捷键
- **Mac**：将Ctrl替换为Cmd键，如Cmd + Shift + K
- **终端差异**：不同终端可能支持不同快捷键，建议检查终端设置1517

在 Docker 中，将**正在运行的容器**打包成**自定义镜像**，主要有两种方法：

### 方法一：使用 `docker commit`（最快，适合临时保存）

这是最简单的方法，直接把当前容器的状态 “快照” 成一个新镜像。

#### 步骤：

1. **查看容器 ID**

   先找到你要打包的容器：

   ```
   docker ps -a
   ```

   记下 `CONTAINER ID`（例如 `abc123`）。

2. **（可选）停止容器**

   为了保证数据一致性，建议先停止容器：

   ```
   docker stop abc123
   ```

   

3. **提交为镜像**

   使用 `docker commit` 创建镜像：

   ```
   # 语法：docker commit <容器ID> <镜像名>:<标签>
   docker commit abc123 my-tf-musa:v1.0
   ```

   

4. **验证镜像**

   查看新创建的镜像：

   ```
   docker images
   ```

   

   你会看到 `my-tf-musa:v1.0` 已经在列表中了。

   

------

### 方法二：使用 `Dockerfile`（推荐，适合正式发布）

`docker commit` 虽然快，但镜像会比较臃肿，且无法追溯修改历史。**推荐使用 `Dockerfile`**，通过代码定义镜像，可重复、可版本控制。

#### 步骤：

1. **创建 `Dockerfile`**

   在任意目录下创建一个名为 `Dockerfile` 的文件：

   ```
   # 1. 基于基础镜像（例如你之前用的 TensorFlow/MUSA 镜像）
   FROM tensorflow/tensorflow:2.6.1-musa
   
   # 2. 执行你的配置命令（安装依赖、复制代码等）
   RUN pip install numpy pandas \
       && apt-get update && apt-get install -y git
   
   # 3. 复制你的项目代码到镜像中
   COPY ./wukong /workspace/wukong
   
   # 4. 设置工作目录
   WORKDIR /workspace/wukong
   
   # 5. （可选）设置启动命令
   CMD ["python", "exp/debug_tensorflow.py"]
   ```

   

2. **构建镜像**

   在 `Dockerfile` 所在目录执行：

   ```
   # 语法：docker build -t <镜像名>:<标签> .
   docker build -t my-tf-musa:v2.0 .
   ```

   

   > 注意最后的 `.` 表示当前目录。
   >
   > 

   

3. **验证并运行**

   ```
   # 查看镜像
   docker images
   
   # 运行新镜像
   docker run -it --gpus all my-tf-musa:v2.0
   ```

   

------

### 两种方法对比



|     特性     |     `docker commit`      |    `Dockerfile`    |
| :----------: | :----------------------: | :----------------: |
|   **速度**   |      快（直接快照）      | 稍慢（需重新构建） |
| **镜像体积** | 大（包含缓存和垃圾文件） |   小（可优化层）   |
| **可维护性** |    差（无法追溯修改）    |  好（代码即文档）  |
| **适用场景** |  临时调试、快速保存环境  | 正式发布、团队协作 |

------

### 针对你场景的建议

如果你是想**保存当前调试好的 MUSA/TensorFlow 环境**：

1. 先用 `docker commit` 快速保存一个版本（防止环境丢了）。
2. 稳定后，把安装步骤整理成 `Dockerfile`，方便后续复用和分享。

在服务器 A 制作好镜像后，要在服务器 B 拉取，主要有 **3 种常用方法**，根据你的网络环境（公网 / 内网）和镜像大小选择即可：

------

### 方法一：使用私有 Docker Registry（推荐，适合内网 / 团队）

这是最规范的方法，适合频繁在多服务器间同步镜像。

#### 步骤：

1. **在服务器 A（或任意内网服务器）启动私有 Registry**

   ```
   # 拉取官方 Registry 镜像并启动（端口 5000）
   docker run -d -p 5000:5000 --name private-registry registry:2
   ```

   

2. **在服务器 A 给镜像打标签**

   假设你的私有 Registry 地址是 `服务器A的IP:5000`，镜像名是 `my-tf-musa:v1.0`：

   ```
   # 语法：docker tag <原镜像名> <Registry地址>/<镜像名>:<标签>
   docker tag my-tf-musa:v1.0 192.168.1.100:5000/my-tf-musa:v1.0
   ```

   

   > 注意：把 `192.168.1.100` 换成服务器 A 的实际内网 IP。
   >
   > 

   

3. **（关键）配置 Docker 信任 HTTP  Registry**

   默认 Docker 只信任 HTTPS，内网 Registry 通常是 HTTP，需要在 **服务器 A 和 B** 都配置：

   

   ```
   # 编辑 /etc/docker/daemon.json
   vim /etc/docker/daemon.json
   ```

   

   添加以下内容（保存并退出）：

   

   ```
   {
     "insecure-registries": ["192.168.1.100:5000"]
   }
   ```

   

   重启 Docker 生效：

   

   ```
   systemctl restart docker
   ```

   

4. **服务器 A 推送镜像到私有 Registry**

   ```
   docker push 192.168.1.100:5000/my-tf-musa:v1.0
   ```

   

5. **服务器 B 拉取镜像**

   ```
   docker pull 192.168.1.100:5000/my-tf-musa:v1.0
   ```

   

------

### 方法二：`docker save` + `docker load`（适合小镜像 / 无网络）

如果镜像不大（<10GB），或者两台服务器无法直接网络通信，直接用文件传输最简单。

#### 步骤：

1. **服务器 A 将镜像导出为 tar 文件**

   ```
   # 语法：docker save -o <文件名.tar> <镜像名>:<标签>
   docker save -o my-tf-musa-v1.0.tar my-tf-musa:v1.0
   ```

   

2. **将 tar 文件传输到服务器 B**

   使用 `scp`（或其他文件传输工具）：

   ```
   # 在服务器 A 执行（或在服务器 B 拉取）
   scp my-tf-musa-v1.0.tar user@服务器B的IP:/path/to/save/
   ```

   

3. **服务器 B 加载镜像**

   ```
   # 语法：docker load -i <文件名.tar>
   docker load -i my-tf-musa-v1.0.tar
   ```

   

4. **验证镜像**

   ```
   docker images
   ```

   

------

### 方法三：使用公共镜像仓库（如 Docker Hub，适合公开镜像）

如果镜像不涉及隐私，也可以用 Docker Hub 等公共仓库中转。

#### 步骤：

1. **注册 Docker Hub 账号**

   去 https://hub.docker.com 注册一个账号。

   

2. **服务器 A 登录 Docker Hub**

   ```
   docker login
   # 输入你的 Docker Hub 用户名和密码
   ```

   

3. **给镜像打标签（格式：用户名 / 镜像名：标签）**

   ```
   docker tag my-tf-musa:v1.0 your-docker-username/my-tf-musa:v1.0
   ```

4. **推送镜像到 Docker Hub**

   ```
   docker push your-docker-username/my-tf-musa:v1.0
   ```

5. **服务器 B 拉取镜像**

   ```
   # （可选）登录 Docker Hub（如果是私有镜像）
   docker login
   
   # 拉取
   docker pull your-docker-username/my-tf-musa:v1.0
   ```

   

------

### 针对你场景的建议

- 如果是**内网环境、镜像较大（如 MUSA/TensorFlow 环境）**：优先用 **方法一（私有 Registry）**，一次搭建，长期使用。
- 如果是**临时同步、镜像较小**：直接用 **方法二（save+load）**，最快最简单。

# **如何 Fork 项目并提交 Pull Request (PR)**

如果你希望对某个开源项目进行改进，并最终提交 PR（Pull Request）给原项目，可以按照以下步骤操作。

## **步骤 1：Fork 远程仓库**

1. **访问原始仓库**
   - 在浏览器打开该项目的 GitHub 仓库（例如 `https://github.com/xxx/xxx`）。
2. **点击 Fork 按钮**
   - 点击右上角的 **Fork** 按钮。
   - 这将会在你的 GitHub 账户中创建该仓库的副本。
3. **确认 Fork 成功**
   - Fork 后，你的 GitHub 账户下会出现这个仓库，URL 类似于：`https://github.com/your-username/project-name`。
   - 此时该副本属于你，你可以自由修改代码，而不会影响原始仓库。

## **步骤 2：克隆 Fork 到本地**

1. **克隆你自己的 Fork**

   bash

   

   ```
   git clone git@github.com:your-username/project-name.git
   cd project-name
   ```

   这将在本地创建 `project-name` 目录，并拉取你的 Fork 代码。

2. **检查远程仓库**

   bash

   

   ```
   git remote -v
   ```

   你应该看到 `origin` 指向你的 Fork 仓库。

3. **添加 upstream（原始仓库）**
   为了同步原始项目的更新，需要手动添加原始仓库地址：

   bash

   

   ```
   git remote add upstream git@github.com:original-owner/project-name.git
   ```

   现在你的本地仓库有两个远程仓库：

   - `origin` → 你的 Fork（你自己的 project-name 仓库）
   - `upstream` → 原始仓库（官方 project-name 仓库）

4. **再次确认远程仓库**

   ```
   git remote -v
   ```

   输出应包含 `origin` 和 `upstream` 的 fetch/push 地址。

## **步骤 3：创建新分支进行开发**

> **注意**：不要直接在 `main` 分支修改，而是创建一个新的开发分支。

1. **获取原始仓库的最新代码**

   ```
   git fetch upstream
   ```

2. **基于 upstream/main 创建新分支**

   ```
   git checkout -b dien upstream/master
   ```

   - `feature-branch` 是你的新功能分支名称（建议改为实际功能名，如 `fix-login-bug`）。
   - 确保代码是基于官方最新的 `main` 分支。

3. **进行代码修改并提交**

   bash

   

   ```
   git add .
   git commit -m "add dien"
   ```

4. **推送你的修改到 GitHub**

   bash

   

   ```
   git push origin feature-branch
   ```

## **步骤 4：创建 Pull Request (PR)**

1. **访问你的 GitHub 仓库**

   - 打开浏览器访问 `https://github.com/your-username/project-name`。

2. **点击 "Compare & pull request"**

   - GitHub 会自动检测你刚刚推送的新分支 `feature-branch`。
   - 它会提示你是否要向 `original-owner/project-name` 提交 PR。

3. **填写 PR 详情**

   - **Title（标题）**: 简要说明你做了什么（如：修复登录 bug）。

   - **Description（描述）**: 详细描述你的更改。

     > 示例：修复了登录功能的一个 bug，现在用户可以正常登录，而不会出现 500 错误。修改了 auth.py 代码，添加了 try-catch 逻辑。

4. **提交 PR**

   - 点击 **"Create Pull Request"** 按钮。

## **步骤 5：等待原仓库维护者审核**

你的 Pull Request 会进入原项目的 PR 列表，维护者可能会：

- ✅ **直接合并** 你的 PR 🎉
- 🔄 **请求修改**（你可以继续在 `feature-branch` 提交新代码并推送）
- ❌ **关闭 PR**（如果他们不接受你的修改）

## **步骤 6：保持你的 Fork 仓库最新**

如果原始项目有新代码，你应该同步它以避免冲突：

bash



```
git fetch upstream
git merge upstream/main
git push origin main
```

------

## **⚠️ 常见问题：SSH 端口 22 连接失败**

如果你的网络环境无法通过 SSH（端口 22）连接 GitHub，可以改用 HTTPS。

1. **检查当前 Git 远程仓库 URL**

   bash

   

   ```
   git remote -v
   ```

   如果看到的是 SSH URL (`git@github.com:...`)。

2. **改为 HTTPS**

   bash

   

   ```
   git remote set-url origin https://github.com/your-username/project-name.git
   ```

3. **使用 HTTPS 方式推送**

   bash

   

   ```
   git push origin main
   ```

> **注意**：如果 GitHub 要求密码，你需要使用 **个人访问令牌（PAT）**，而不是 GitHub 账户密码。

# ONNX

可以把它理解为深度学习模型的**通用文件格式**，就像图像领域的 JPEG 或音频领域的 MP3 一样。它的核心目标是解决深度学习框架之间的“孤岛”问题。

下面从几个方面来详细拆解：

### 1. 为什么需要 ONNX？—— 解决框架锁定的问题

在 ONNX 出现之前，深度学习生态是这样的：

- **PyTorch** 训练出的模型（`.pth` 文件）通常只能在 PyTorch 环境里运行。
- **TensorFlow** 训练出的模型（`.pb` 文件）通常只能在 TensorFlow 环境里运行。
- 如果你想在生产环境部署一个模型，可能发现某个框架的推理引擎速度更快，或者某个硬件（如特定的加速芯片）只支持特定的框架。这就导致了“训练环境”和“部署环境”不匹配的矛盾。

**ONNX 的角色就像一个桥梁或翻译器：**

1. **训练阶段**：你可以使用你最喜欢的、最灵活的框架（比如 PyTorch 来做研究）。
2. **导出阶段**：将这个训练好的模型（计算图和权重）**标准化**地表述出来，也就是转换成 `.onnx` 格式的文件。这个过程就像把一本中文书准确地翻译成“世界语”。
3. **部署阶段**：任何支持 ONNX 格式的平台或推理引擎（最著名的是微软的 **ONNX Runtime**）都可以读取这个 `.onnx` 文件，并把它高效地运行起来，无论底层的硬件是 CPU、GPU 还是专用的 AI 芯片。

### 2. ONNX 的核心组成

一个 ONNX 模型文件内部主要包含两部分信息：

- **模型结构（计算图）**：
  - 定义了模型的“骨架”。它由一系列**节点（Nodes）**组成，每个节点代表一个算子（Operator），比如卷积（Conv）、矩阵乘法（Gemm）、激活函数（Relu）等。
  - 这些节点通过**张量（Tensors）**连接，描述了数据（即张量）在网络中流动的路径。这就是所谓的“计算图”。
- **模型权重（参数）**：
  - 定义了模型的“记忆”。这是神经网络在训练过程中学习到的所有数值，比如卷积核的权重和偏置项。这些也以张量的形式存储在 ONNX 文件中。
- **模型元数据**：
  - 定义了模型的版本、作者、使用的算子集版本等信息，方便管理和追溯。

### 3. 从 TensorFlow 模型导出为 ONNX 的过程（正在进行的操作）

当你看到“程序正在把 TensorFlow 模型导出成 ONNX 格式”时，背后大致发生了这些事情：

1. **加载原始模型**：程序首先会使用 TensorFlow 加载你已经训练好的模型（可能是 `SavedModel` 格式、`.h5` 格式或冻结的 `.pb` 图）。
2. **图遍历与转换**：这是最核心也是最耗时的部分。转换工具（通常是 `tf2onnx` 库）会逐层、逐操作地遍历 TensorFlow 的计算图。
3. **算子映射**：对于遇到的每一个 TensorFlow 算子（比如 `tf.nn.conv2d`），转换器会在 ONNX 的算子标准集中寻找一个功能完全等价的算子（比如 `Conv`）。由于不同框架的算子定义和细节可能存在细微差别，这个过程需要非常精确的映射和参数调整。
4. **常量折叠与优化**：转换器会尝试对一些静态的、可以预先计算的部分进行计算，简化计算图。同时，将 TensorFlow 的变量（Variables）转换为 ONNX 中的常量张量（即模型权重）。
5. **构建 ONNX 图**：转换器使用映射后的算子、连接关系和权重，在内存中构建出一个全新的、符合 ONNX 规范的模型图。
6. **验证与保存**：转换完成后，通常还会用 ONNX Runtime 对生成的模型进行一次推理，并将结果与原始 TensorFlow 模型的推理结果进行比对，以确保转换过程没有引入误差。验证通过后，最终将模型保存为一个 `.onnx` 文件。

### 4. 转换成 ONNX 格式的价值

为什么即使需要花时间转换，大家还是愿意这么做？

- **极致的推理性能（ONNX Runtime）**：ONNX Runtime 是一个专门为 ONNX 模型设计的高性能推理引擎。它会对模型进行深度图优化，比如：
  - **算子融合**：将多个连续的细小操作（如 `Conv + BN + Relu`）合并成一个大的、更高效的内核（kernel）来执行，减少内存读写和内核启动的开销。
  - **内存复用**：智能地规划张量的内存分配，减少内存占用和分配开销。
  - **硬件加速**：它可以无缝对接各种硬件平台的专用库，如 NVIDIA GPU 的 CUDA 和 TensorRT、Intel CPU 的 OpenVINO、以及各种移动端和嵌入式端的 NPU 等。你只需一个 ONNX 模型，ONNX Runtime 就能自动调用底层的硬件加速能力。
- **跨平台部署**：从云端服务器到手机（iOS/Android），从树莓派到 Windows 桌面应用，ONNX Runtime 都有对应的运行时库。你可以“一次转换，到处运行”。
- **框架互操作性**：比如，你用 PyTorch 训练了一个新模型，但你的生产环境是 TensorFlow 的生态。ONNX 可以帮你把 PyTorch 模型无缝地“喂”给 TensorFlow Serving 进行部署。

总结一下：你正在做的“把 TensorFlow 模型导出成 ONNX 格式”，本质上是在把你的模型从一种专有的“方言”（TensorFlow），转换成为一种通用的“世界语”（ONNX）。虽然转换需要一点时间，但它为后续的跨平台部署和性能优化打开了大门。



# screen

`screen` 是 Linux 下的终端复用工具，核心作用是**创建独立会话、后台保活任务、多窗口管理、远程断连不中断**，适合跑长时间脚本、远程服务器操作。

`screen` 是 Linux 下的终端复用工具，核心作用是**创建独立会话、后台保活任务、多窗口管理、远程断连不中断**，适合跑长时间脚本、远程服务器操作。

### 一、安装（如未预装）
```bash
# Debian/Ubuntu
sudo apt update && sudo apt install screen

# CentOS/RHEL
sudo yum install screen

# Fedora
sudo dnf install screen
```

### 二、基础操作（最常用）
#### 1. 创建会话
```bash
# 匿名会话（不推荐，难管理）
screen

# 命名会话（推荐，方便识别）
screen -S my_task
```
进入后和普通终端一样，可执行任意命令。

#### 2. 分离会话（后台保活）
在 screen 会话内按：
**`Ctrl + a` → 松开 → 按 `d`**
提示 `[detached from ...]`，回到原终端，会话后台继续运行。

#### 3. 查看所有会话
```bash
screen -ls
# 输出示例：
# There is a screen on:
# 12345.my_task    (Detached)
# 1 Socket in /run/screen/S-xxx.
```
- `Attached`：正在被连接
- `Detached`：后台挂起

#### 4. 重新连接会话
```bash
# 按名称连接
screen -r my_task

# 按ID连接（多个会话时用）
screen -r 12345

# 强制断开并重新连接（解决Attached占用）
screen -dr my_task
```

#### 5. 关闭/杀死会话
- 会话内直接退出：`exit` 或 `Ctrl + d`
- 外部杀死会话：
```bash
screen -S my_task -X quit
```

---

### 三、多窗口管理（会话内）
所有操作以 `Ctrl + a` 为前缀：
- 新建窗口：`Ctrl + a` → `c`
- 下一个窗口：`Ctrl + a` → `n`
- 上一个窗口：`Ctrl + a` → `p`
- 窗口列表（选择切换）：`Ctrl + a` → `"`
- 关闭当前窗口：`exit` 或 `Ctrl + d`

---

### 四、分屏（高级）
- 垂直分屏（左右）：`Ctrl + a` → `|`（Shift + \）
- 水平分屏（上下）：`Ctrl + a` → `S`（大写）
- 切换分屏：`Ctrl + a` → `Tab`
- 关闭当前分屏：`Ctrl + a` → `X`

---

### 五、常用进阶命令
```bash
# 后台创建会话并直接执行命令（不进入）
screen -dmS long_task "python train.py"

# 开启日志（记录会话输出）
screen -L -S log_session

# 共享会话（多人协作）
screen -x my_task
```

---

### 六、典型场景示例
1. 跑长时间训练脚本，断网不中断
```bash
screen -S train
python train.py
# Ctrl + a d 分离
# 下次登录：screen -r train
```

2. 服务器多任务并行
```bash
screen -S server
# Ctrl + a c → 开新窗口跑服务1
# Ctrl + a c → 开新窗口跑服务2
# Ctrl + a n/p 切换
```
# screen 常用命令速查表
> 核心说明：所有快捷键操作均以 **Ctrl + a** 为前缀（先同时按下Ctrl和a，松开后再按后续按键）；命令行操作直接在终端执行。

| 快捷键 | 功能说明 |

### 新手快速上手示例（最常用场景）

3. 按 `Ctrl + a` 再按 `d` 分离会话，任务后台继续运行
4. 下次登录服务器，执行 `screen -r my_run` 即可回到任务界面





# tensorboard在内网怎么进行


## 步骤 1：在**远程服务器**（宿主机，不是容器里）执行这行命令

**获取你的 Docker 容器内网 IP**（复制直接运行）：

```
docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' 98eb0f0208be
```

运行后会输出一串 IP，比如

```
172.17.0.2
```

## 步骤 2：修改你**本地电脑**的 SSH 转发命令（核心！）

把你之前的错误命令：

```
ssh -L 6006:localhost:6006 mccxadmin@worker33016
```

**替换成这个**（把 `容器IP` 换成你刚才查到的）：

```
ssh -L 6006:容器IP:6006 mccxadmin@worker33016
```

### 举个例子（直接照抄改 IP）：

```
ssh -L 6006:172.17.0.2:6006 mccxadmin@worker33016
```

------

## 步骤 3：保持容器内 TensorBoard 运行

确保你的容器里一直开着这个命令（**不要关**）：

```
tensorboard --logdir logs/ --bind_all
```

------

## 步骤 4：本地浏览器打开

```
http://localhost:6006
```
