# 第 1 章 为什么要系统学习 GPU 内存管理

## 1.1 本章目标

这一章不急着进入 CUDA API，也不急着讲显存页表、内存池或者 profiler。它先回答一个更基础的问题：为什么 GPU 内存管理值得单独、系统地学习。

很多人在接触 GPU 时，最先形成的印象通常是：

- GPU 核心多
- GPU 并行强
- GPU 算得快
- 只要把计算搬到 GPU 上，就会明显加速

这套理解并不完全错误，但它只看到了 GPU 的"算力侧"，没有看到 GPU 在真实系统中更常见的限制因素：内存。

从工程角度看，GPU 程序经常不是先被"算力"卡住，而是先被下面这些因素卡住：

- 显存容量不够
- 显存带宽用不满
- 访问模式太差
- CPU 和 GPU 之间搬运数据太慢
- 内存分配与释放开销太高
- 隐式同步打断流水线
- 内存碎片导致表面空闲、实际不可用

因此，如果你想真正掌握 GPU，不管你将来更偏向深度学习训练、推理、通用 CUDA 开发，还是图形渲染，都绕不过内存管理。

本章的目标有三个：

1. 让你知道 GPU 内存管理为什么是核心问题，而不是边角知识。
2. 让你对后面会反复出现的几个关键词先有直觉：容量、带宽、延迟、访问模式、分配开销。
3. 让你带着清晰的问题进入后续章节，而不是在一堆术语里迷路。

---

## 1.2 为什么很多 GPU 程序不是先被算力限制

### 先看一个直觉

假设你有一台很强的 GPU，它有很高的 FLOPS，理论算力非常惊人。你写了一个程序，把大量计算都放到了 GPU 上，结果运行后发现：

- GPU 利用率并不高
- 程序整体还是不够快
- 显存很紧张
- 数据拷贝时间很长
- 某些 kernel 很慢，但算术操作并不复杂

这时候问题往往不在"算得不够快"，而在"数据没准备好，或者数据访问不高效"。

这是 GPU 编程里非常重要的一条基本事实：

**算力只决定上限，内存系统决定你能不能接近上限。**

### 为什么会这样

GPU 之所以强，是因为它能同时调度大量线程去做相似的工作。可这些线程不是凭空工作的，它们要读数据、写结果、访问中间状态。如果数据不能快速、规律、批量地送到计算单元面前，再强的计算单元也只能等待。

这就像一个大型工厂：

- 机器很多，工位很多，这是"算力"
- 原材料供应、流水线运输、仓库调度，这是"内存系统"

如果原材料供应不上，机器再多也没有用。

### 现实中的典型瓶颈

在 GPU 程序里，最常见的瓶颈通常落在这些地方：

#### 1. 容量瓶颈

你的数据、模型、中间结果放不下。

典型现象：

- 训练时 batch size 稍微调大就 OOM
- 推理时上下文一长显存就爆
- 图形程序里纹理、几何资源一多就超预算

#### 2. 带宽瓶颈

数据放得下，但 GPU 读写显存的速度不够，导致算力发挥不出来。

典型现象：

- kernel 的算术并不重，但运行时间很长
- GPU 的 ALU 没有忙起来，内存通路先饱和了
- 增加一点计算反而几乎不影响总时间，说明程序本来就是被内存访问拖住

#### 3. 访问模式瓶颈

不是"没有带宽"，而是"没把带宽用好"。

典型现象：

- 同样大小的数据，连续访问快，随机访问慢很多
- 数据结构稍微换一种布局，性能差距巨大
- 同样读显存，访问是否合并、是否对齐，结果差异很大

#### 4. 传输瓶颈

GPU 的计算速度再高，也经常被数据搬运拖慢。

典型现象：

- 每次计算前都要拷一次数据到 GPU
- 每次计算后又把结果拷回 CPU
- 小块数据来回搬很多次，整体时间全耗在传输上

#### 5. 分配器与同步瓶颈

不是算慢，也不是搬慢，而是内存管理本身就让程序发抖。

典型现象：

- 频繁 `cudaMalloc` / `cudaFree`
- 某些 API 看起来只是"释放内存"，却突然让程序卡住
- 长时间运行后性能变差，出现碎片化和抖动

## 1.3 你要先分清五个关键词

GPU 内存管理里最容易混乱的一点，是把很多不同问题都笼统叫成"内存问题"。实际上，这些问题的本质完全不同。

后面的全文都会围绕下面五个关键词展开。你现在先建立一个最初步的直觉即可。

### 1. 容量

容量就是"能放多少"。

在 GPU 语境里，最常见的是显存容量，也就是 VRAM 的大小。它直接决定：

- 模型是否能装下
- batch size 能有多大
- 缓存能保留多少
- 是否需要分块、offload、量化或重计算

容量问题最直观，因为它通常会直接报错，比如 OOM。但也最容易让人误判，因为"没放满"不代表"还能继续分配"。

### 2. 带宽

带宽就是"单位时间能运多少数据"。

GPU 显存带宽通常非常高，但这不代表程序一定能吃满。你可以把它理解为一条很宽的高速公路：

- 路很宽，不代表车就一定跑得顺
- 如果车流组织混乱，或者上下匝道太频繁，实际通行效率会很差

带宽问题的核心不是"有没有路"，而是"你是否在高效地用路"。

### 3. 延迟

延迟就是"一次访问从发起到拿到结果要多久"。

显存访问的延迟通常远高于寄存器和片上共享存储。GPU 的一个重要设计思想是：**靠大量并发去隐藏延迟**。也就是说，它不一定让一次访问特别快，而是让很多线程交替工作，在有人等数据时让别人先算。

因此，GPU 面对延迟的方式，和 CPU 很不一样。CPU 常常追求单次访问尽量快，GPU 更常追求整体吞吐尽量高。

### 4. 访问模式

访问模式指的是：线程如何组织对数据的读写。

这是 GPU 内存管理里最常被低估，但影响极大的因素。

同样读 1GB 数据：

- 连续、对齐、合并访问，可能很快
- 跨步、随机、零散访问，可能慢很多

很多"显存带宽利用率低"的问题，真正根因不是硬件差，而是访问模式差。

### 5. 分配开销

很多人以为内存管理只在程序开始时做一次，之后就结束了。真实工程里，分配与释放可能贯穿程序全程，而每一次分配都不是零成本。

在 GPU 上，这些成本尤其值得注意，因为它们可能涉及：

- 运行时与驱动交互
- 虚拟地址管理
- 物理页分配与映射
- 分配器元数据维护
- 与已有工作流的同步

所以，频繁地申请和释放小块 GPU 内存，常常是一种坏味道。

## 1.4 为什么"程序能跑"不等于"GPU 用得对"

这是 GPU 学习中非常重要的一次认知转变。

在 CPU 编程里，很多时候只要程序逻辑正确、复杂度合理、缓存没特别离谱，性能通常还能接受。但在 GPU 上，很多程序即使"能跑"，依然可能离合理的性能和资源利用率相差很远。

### 一个常见误区

很多人第一次写 GPU 程序时，会这样想：

- 我把 CPU 代码搬到 GPU 上了
- 线程也开起来了
- 数据也传过去了
- 结果也算对了
- 所以应该已经差不多了

这通常只是刚开始。

因为 GPU 上"算对"和"跑好"之间，往往隔着很多层约束：

- 数据布局是否适合并行线程访问
- 内存访问是否合并
- 中间 buffer 是否频繁申请释放
- 是否发生了不必要的主机-设备同步
- 计算和传输是否能重叠
- 是否被显存容量或碎片卡住
- 是否因为寄存器/shared memory 使用过多，压低了并发度

### GPU 更像"系统优化问题"

你可以把 GPU 程序理解成一个完整的数据流系统，而不是一个单纯的算术函数。

这个系统至少包含下面几件事：

1. 数据从哪里来
2. 数据如何进入 GPU
3. 数据在 GPU 内部存在哪里
4. 线程如何访问这些数据
5. 中间结果如何复用
6. 什么时候释放资源
7. 如何避免传输、分配、同步成为主瓶颈

因此，GPU 内存管理不是"会几个 API"这么简单，而是一种系统设计能力。

## 1.5 常见问题总览：你之后会不断碰到什么

在正式进入总图和底层原理之前，先把你以后最常碰到的问题列出来。这样你在学后面的内容时，会知道每个知识点是为了解决什么。

### 1. 显存没满，为什么还 OOM

这通常不是单纯容量不足，而可能是：

- 内存碎片
- 可用连续块不足
- 框架缓存分配器行为
- 某个时刻峰值内存过高
- 还有未完成的操作持有某些内存块

### 2. GPU 利用率不高，为什么程序还是慢

这类问题很常见，可能的原因包括：

- 数据传输拖住了
- kernel 太碎，调度和启动成本高
- 访问模式差，带宽利用低
- CPU 端在等待
- 分配释放频繁引入同步

### 3. 异步 API 为什么还是会卡

因为"调用异步"不等于"整个链路异步"。

常见原因包括：

- 使用了 pageable memory 而不是 pinned memory
- 某个操作需要等待前面的工作完成
- 某些分配或释放会触发同步
- stream 之间存在依赖

### 4. 框架为什么不主动释放显存

很多框架会使用缓存分配器，原因不是懒，而是为了减少昂贵的系统级分配与释放。

这会带来一个表面现象：

- `nvidia-smi` 看起来显存一直很高
- 但框架内部其实可能已经把部分块标记为可复用

### 5. 为什么改个 batch size，显存变化会特别大

因为显存消耗并不只来自模型参数，还包括：

- 激活值
- 梯度
- 优化器状态
- 临时 workspace
- 通信 buffer
- 推理中的 KV cache

这些量有些与 batch size 线性相关，有些与序列长度、层数、头数等共同相关。

### 6. 为什么同样的算法，数据布局一变就快很多

因为 GPU 尤其依赖：

- 连续访问
- 对齐访问
- 合并访问
- 更好的局部性
- 更少的 bank conflict

这说明内存管理不是"只管容量"，也包括"如何组织数据"。

## 1.6 本教程的学习路径

这篇教程会按"从底层到顶层"的顺序展开。

### 第一步：先建立总图

你需要先知道 GPU 内存系统有哪些层：

- 硬件层
- 地址空间和页映射层
- CUDA 运行时层
- 分配器和框架层
- 上层应用场景层

如果一开始没有这张图，后面每个概念都会显得很碎。

### 第二步：理解 CPU 和 GPU 的思维差异

如果你还在用 CPU 的默认直觉理解 GPU，那么很多概念会看得似懂非懂。你必须先完成思维切换：GPU 不主要追求单次低延迟，而主要追求整体高吞吐。

### 第三步：进入硬件和系统底层

包括：

- 寄存器、shared memory、cache、global memory
- 地址空间、页表、UVA、UVM
- 分配、映射、迁移、回收

这是全文的物理基础。

### 第四步：进入 CUDA 运行时和工程机制

包括：

- `cudaMalloc` / `cudaFree`
- `cudaMemcpyAsync`
- pinned memory
- `cudaMallocAsync`
- memory pool
- 缓存分配器与碎片

这是全文的实践基础。

### 第五步：进入具体场景

包括：

- 深度学习训练
- 深度学习推理
- 通用 CUDA/C++
- 图形与渲染
- 多 GPU / 多进程 / 多租户

你会看到同一套底层原理在不同场景里如何变成不同的问题。

### 第六步：形成排障与优化能力

最终你不仅要知道概念，还要能回答：

- 这是容量问题还是带宽问题
- 是访问模式差，还是分配器在抖
- 是传输阻塞，还是同步点太多
- 是真泄漏，还是缓存持有
- 该优先做哪一类优化

## 1.7 本章小结

这一章最重要的不是记住术语，而是建立一个正确的学习出发点：

1. GPU 程序经常不是先被算力卡住，而是先被内存系统卡住。
2. 内存问题至少分成容量、带宽、延迟、访问模式、分配开销这几类，不能混为一谈。
3. "程序能跑"远不代表"GPU 用得对"，因为 GPU 性能高度依赖数据布局、访问路径和资源生命周期。
4. 学 GPU 内存管理，本质上是在学一个完整系统如何把数据高效地送到计算单元面前，并在全过程中控制容量、带宽、同步和复用。

下一章我们会先画出一张总图，把 GPU 内存系统到底由哪些层组成讲清楚。这样后面无论讲显存、页表、UVA、`cudaMalloc`、碎片还是深度学习框架分配器，你都能知道它们各自位于整条链路的哪个位置。

# 第 2 章 总图：GPU 内存系统到底由哪些层组成

## 2.1 本章目标

上一章回答了"为什么要学 GPU 内存管理"。这一章开始回答"GPU 内存系统到底是什么"。

初学者最容易遇到的问题不是概念太少，而是概念太散。会听到很多名词：

- 显存
- VRAM
- global memory
- shared memory
- L2
- pinned memory
- pageable memory
- UVA
- UVM
- `cudaMalloc`
- memory pool
- caching allocator
- stream
- DMA
- PCIe
- NVLink

这些名词如果一个一个孤立地去记，很快就会混乱。真正有效的方式，是先建立一张总图，再把每个名词放回总图中的位置。

本章的目标是：

1. 给出 GPU 内存系统的全局结构。
2. 区分硬件层、系统层、运行时层、框架层、应用层。
3. 给出 GPU 内存"从申请到释放"的完整生命周期总览。
4. 建立后面分析所有问题的统一分类框架。

这一章不追求细节讲透，而追求"定位准确"。你读完后，应该能说出每一类问题属于哪一层，每一个关键概念在整条链路中起什么作用。

## 2.2 先看一张总图

可以先用一个简化的文字图把整体框架搭起来。

```text
应用/框架层
  ├─ 深度学习框架（PyTorch / TensorFlow）
  ├─ 推理引擎
  ├─ CUDA/C++ 应用
  └─ 图形/渲染引擎

运行时层
  ├─ CUDA Runtime API
  ├─ CUDA Driver API
  ├─ stream / event
  ├─ cudaMalloc / cudaMemcpy
  └─ cudaMallocAsync / memory pool

系统层
  ├─ 进程与 GPU context
  ├─ 虚拟地址空间
  ├─ 页表与映射
  ├─ UVA / UVM
  ├─ DMA
  ├─ IOMMU / BAR
  └─ PCIe / NVLink

GPU 硬件内存层次
  ├─ 寄存器
  ├─ shared memory
  ├─ L1 / texture / constant cache
  ├─ L2 cache
  └─ global memory / VRAM

CPU 侧
  ├─ CPU 寄存器 / cache / DRAM
  ├─ pageable host memory
  └─ pinned host memory
```

这张图表达了一个非常重要的事实：

**GPU 内存管理不是"只有显存"这一件事，而是从 CPU 内存、总线、地址空间、运行时、硬件缓存一直到上层框架的一整条链路。**

很多问题之所以难，是因为它们跨层出现。例如：

- 框架里的一次张量分配，最终可能落到 CUDA 分配器和 GPU 虚拟地址映射上
- 一次看似普通的异步拷贝，是否真正异步，取决于主机内存是否 pinned
- 一次 OOM 可能来自框架缓存策略，也可能来自驱动层分配失败，还可能来自碎片和生命周期重叠

所以你要学会用"分层"的方式看问题。

---

## 2.3 第一层：硬件层

最底层是 GPU 自己的硬件存储层次。这一层决定的是：

- 数据在哪里存
- 访问速度大概如何
- 哪些资源最稀缺
- 线程并发会受到哪些存储资源约束

这一层主要包括：

- 寄存器
- shared memory
- L1 cache
- L2 cache
- global memory，也就是通常说的显存
- 某些专用只读缓存，例如 constant cache 或 texture cache

### 为什么这一层重要

因为你最终写的所有 CUDA kernel，都要在这些层次上读写数据。无论上层 API 多么抽象，真正影响执行效率的，始终是：

- 数据是不是落在便宜的层次
- 访问是不是符合硬件偏好
- 某类资源是不是成了并发瓶颈

举例来说：

- 寄存器很快，但数量有限
- shared memory 很快，但每个 SM 容量有限
- 显存容量大，但访问代价高
- L2 是共享资源，容易成为很多线程竞争的焦点

后面第 4 章会详细展开这一层。

---

## 2.4 第二层：系统层

系统层解决的不是"算得快不快"，而是"数据到底怎么被看见、被搬运、被映射"。

这一层主要包括：

- 进程和 GPU context
- 设备虚拟地址空间
- 页表和地址翻译
- 物理页分配与映射
- DMA
- IOMMU
- BAR
- PCIe / NVLink
- UVA
- UVM

### 这一层到底在干什么

如果你在程序里拿到了一个设备指针，这个指针并不意味着"有一块永远简单连续的物理显存"。它通常意味着：

1. 某个 GPU context 中保留了一段虚拟地址空间
2. 某些物理页被分配出来
3. 虚拟地址与物理页之间建立了映射
4. 后续的访问通过地址翻译和硬件通路完成

同理，如果你要把数据从 CPU 搬到 GPU，也不是"瞬间复制"这么简单，它背后依赖：

- 主机内存是否允许 DMA 高效读取
- 传输路径走 PCIe 还是 NVLink
- 拷贝请求是同步还是异步
- 是否需要 staging
- 有没有 page migration 或 page fault

### 为什么很多人会低估系统层

因为 CUDA API 把很多复杂性藏了起来。你写的是：

```cpp
cudaMalloc(&ptr, size);
cudaMemcpy(ptr, host, size, cudaMemcpyHostToDevice);
```

但底层实际涉及的是：

- context
- 地址空间
- 分配器
- 页映射
- DMA 路径
- 同步规则

如果不了解系统层，就很容易把很多现象误判成"GPU 不稳定"或者"CUDA 很玄学"。

---

## 2.5 第三层：运行时层

运行时层是你最常直接接触的一层。它负责把底层复杂机制包装成编程接口。

在 CUDA 体系下，这一层通常包括：

- CUDA Runtime API
- CUDA Driver API
- stream
- event
- `cudaMalloc`
- `cudaFree`
- `cudaMemcpy`
- `cudaMemcpyAsync`
- `cudaMallocManaged`
- `cudaMallocAsync`
- memory pool

### 这一层的作用

运行时层的作用可以概括为三件事：

1. 给你提供编程接口
2. 管理与驱动和 context 的交互
3. 规定操作之间的同步、顺序和资源语义

### 为什么这一层重要

因为你在工程里感受到的大多数"内存管理行为"，都先通过这一层表现出来。例如：

- 为什么 `cudaFree` 可能卡住
- 为什么 `cudaMemcpyAsync` 有时并不真正异步
- 为什么 `cudaMallocAsync` 可以减少某些同步和碎片问题
- 为什么 stream 的存在会影响内存复用

运行时层是"底层机制"和"上层应用行为"之间的桥梁。

---

## 2.6 第四层：框架与分配器层

如果你做的是深度学习、推理服务或者复杂 CUDA 应用，通常不会每次都直接调用最原始的 CUDA 分配接口，而是会通过某种上层分配器。

这一层常见的对象包括：

- 深度学习框架的 caching allocator
- 推理引擎的 arena 或 buffer pool
- 图形引擎的资源堆和驻留管理
- 自定义应用中的 memory pool、ring buffer、staging allocator

### 为什么这一层不可忽视

因为现实工程中，真正决定"显存看起来怎么用""为什么不释放""为什么会碎片""为什么长时间运行变抖"的，经常不是 CUDA 最底层 API，而是这一层的分配策略。

比如深度学习框架通常会：

- 预先申请较大的显存块
- 内部切分成小块给张量使用
- 张量释放后不立刻归还给 CUDA
- 保留这些块以备下次复用

这样做的好处是：

- 减少 `cudaMalloc` / `cudaFree` 次数
- 降低系统级分配开销
- 避免频繁同步
- 提高整体稳定吞吐

但代价是：

- 显存占用表面上一直较高
- 碎片问题可能更明显
- 更难从系统层直接判断"到底是不是泄漏"

---

## 2.7 第五层：应用层

最上层就是你的具体应用场景。不同场景会以完全不同的方式使用同一套 GPU 内存基础设施。

### 深度学习训练

重点通常是：

- 参数
- 梯度
- 优化器状态
- 激活值
- 中间 workspace
- 多卡通信 buffer

训练的核心问题是：**峰值显存和生命周期重叠**。

### 深度学习推理

重点通常是：

- 权重常驻
- 中间 buffer 复用
- KV cache
- 动态 batch
- 长时间服务稳定性

推理的核心问题是：**稳定复用和动态请求带来的波动**。

### 通用 CUDA/C++

重点通常是：

- 输入输出缓冲区
- staging buffer
- 双缓冲/多缓冲
- 分块与流水线
- 多 stream 协同

核心问题是：**如何把数据流设计得顺畅**。

### 图形与渲染

重点通常是：

- 纹理
- buffer
- render target
- 上传路径
- 驻留预算
- 流式资源加载

核心问题是：**资源生命周期和显存预算控制**。

这说明，同一套底层原理，在不同应用中会长成不同的"表面问题"。

---

## 2.8 GPU 内存的一条完整生命周期

现在我们把各层连起来，先看一遍一块 GPU 内存从"出生"到"消失"的完整链路。

### 第一步：申请

应用或框架请求一块 GPU 内存。

可能的入口有：

- 直接调用 `cudaMalloc`
- 通过框架张量创建触发缓存分配器
- 从 memory pool 中切出一块已有空间

### 第二步：地址空间准备

运行时和驱动会在 GPU context 相关的地址空间里处理这一请求，可能涉及：

- 保留虚拟地址范围
- 查找或创建分配器元数据
- 准备映射关系

### 第三步：物理页分配与映射

系统为这块虚拟地址范围关联实际物理显存页，形成可访问关系。

这时你在应用层才真正"得到一块可用设备内存"。

### 第四步：数据进入

如果这块内存需要装入数据，通常会涉及：

- H2D 拷贝
- pinned memory 或 pageable memory 差异
- DMA
- stream 和异步调度

### 第五步：kernel 访问

GPU 线程开始访问这块内存。此时性能由硬件层和访问模式共同决定：

- 是进寄存器、shared memory 还是直接走显存
- 是否 coalesced
- 是否命中 cache
- 是否有 bank conflict
- 是否因为资源占用压低 occupancy

### 第六步：复用

任务结束后，这块内存不一定立刻归还系统。更常见的是：

- 返回给框架缓存分配器
- 留在 pool 中等下次复用
- 在服务系统中长期保留为工作缓冲区

### 第七步：回收

只有在特定时机，这块内存才会真正被归还到底层系统分配器。

这时才可能释放映射、更新元数据、归还物理页。

### 这一条链路的重要性

后面整篇教程其实都可以看成是在反复展开这七步：

1. 申请
2. 准备地址空间
3. 映射
4. 迁移/拷贝
5. 访问
6. 复用
7. 回收

只要你脑中一直保留这条主链路，遇到任何新问题都更容易定位。

---

## 2.9 四类最常见的内存问题

为了后续分析方便，我们把 GPU 内存问题统一归成四大类。几乎所有具体问题都能先归到其中之一。

### 第一类：容量问题

核心问题是：放不下。

表现包括：

- OOM
- batch size 上不去
- 长上下文推理撑爆显存
- 图形资源超预算

常见原因包括：

- 真实数据量就是太大
- 某时刻峰值太高
- 缓存太多
- 内存生命周期重叠太多
- 碎片导致无法满足大块分配

### 第二类：带宽问题

核心问题是：运得不够快。

表现包括：

- kernel 执行时间受显存读写主导
- 算力空着，内存通路先饱和
- 算法复杂度不高，但吞吐依然上不去

常见原因包括：

- 数据量本身巨大
- 访存密度高
- 缓存命中率低
- 读写模式难以复用数据

### 第三类：访问模式问题

核心问题是：用错了内存系统。

表现包括：

- 随机访问导致事务数暴涨
- warp 内线程访问不合并
- bank conflict
- AoS 布局不适合并行访问
- 访存对齐差

这一类问题和带宽问题相关，但不相同。带宽问题是"路不够"，访问模式问题往往是"你没把路走对"。

### 第四类：分配器与同步问题

核心问题是：内存管理本身拖慢程序。

表现包括：

- 频繁分配释放造成抖动
- 某些释放操作卡顿
- 异步链路被意外同步打断
- 长时间运行后碎片加重

这一类问题最容易被忽略，因为它常常不表现为"显式的计算慢"，而是表现为"不稳定"。

---

## 2.10 从"看到现象"到"归到哪一层"

后续学 GPU 内存管理时，一个非常重要的习惯是：看到现象，不要立刻乱猜；先归层。

下面举几个例子。

### 例子 1：`nvidia-smi` 显示显存一直很高

先别急着说"泄漏"。先问：

- 是框架缓存分配器持有吗
- 是不是还有长生命周期 buffer
- 是否存在未完成工作占用
- 是否已经归还到框架池，但没有归还给驱动

这通常要先看框架层和分配器层。

### 例子 2：`cudaMemcpyAsync` 没有带来预期中的重叠

先问：

- 主机内存是不是 pinned
- stream 是否独立
- copy engine 是否可用
- 是否存在前后依赖
- 是否有隐式同步点

这通常涉及系统层和运行时层。

### 例子 3：kernel 很慢，但算术不复杂

先问：

- 是否带宽受限
- 是否访问不合并
- 是否缓存命中差
- 是否 shared memory 用法不当
- 是否 occupancy 太低

这通常涉及硬件层和访问模式层。

### 例子 4：显存还有空闲，但分配失败

先问：

- 是否碎片化
- 是否需要较大连续块
- 是否某些块在别的 stream 中仍不可回收
- 框架池中是否有不可合并的空洞

这通常涉及分配器层和生命周期管理。

这个"先归层"的习惯，会决定你后面的学习效率和排障效率。

---

## 2.11 为什么要用"生命周期"而不是"静态容量"思考

初学者常常把显存想成一个固定大小的盒子，程序只是在往里面塞东西。但工程上的真实问题，更多发生在"时间维度"上。

同样 16GB 显存：

- 一种程序可能稳定运行
- 另一种程序可能频繁 OOM 或抖动

差别不一定在于总量，而在于生命周期管理。

你要关心的不只是：

- 总共有多少数据

还要关心：

- 哪些数据必须常驻
- 哪些只是短暂中间结果
- 哪些可以复用同一块空间
- 哪些不同阶段其实不该同时活着
- 哪些操作完成前内存不能回收

这就是为什么 GPU 内存管理经常要谈：

- 峰值内存
- 生命周期重叠
- 缓冲区复用
- 分块计算
- 流水线
- memory pool

从这个角度看，"内存管理"本质上是在管理**时间上的资源占用形状**，而不只是管理空间大小。

## 2.12 本章小结

这一章最重要的工作，是建立 GPU 内存系统的"全图"。

你现在应该有了三个核心认识：

1. GPU 内存管理不是只有显存，而是从 CPU 内存、传输链路、地址空间、运行时、分配器到应用策略的一整套系统。
2. 任何具体问题都可以先归到几类核心瓶颈：容量、带宽、访问模式、分配器与同步。
3. 一块 GPU 内存从申请、映射、拷贝、访问、复用到回收，贯穿了整条软件和硬件栈。

后面每一章其实都只是把这张图中的某一部分放大来看。

下一章我们会先做一次最关键的思维切换：解释 CPU 和 GPU 的内存思维为什么根本不同。只有完成这一步，后面的很多 GPU 访存特性才会真正变得"合理"。

# 第 3 章 CPU 与 GPU 的内存思维为什么根本不同

## 3.1 本章目标

如果说上一章是给出了地图，这一章就是帮你换一副眼镜。

很多人在学习 GPU 时，最大的障碍并不是公式、API 或者硬件术语，而是他们在用 CPU 的直觉理解 GPU。这会导致很多 GPU 特性看起来"不自然"：

- 为什么 GPU 这么怕随机访问
- 为什么数据布局影响这么大
- 为什么同样是内存访问，GPU 对 coalescing 如此敏感
- 为什么大量并发线程并不能自动把问题解决
- 为什么"只要逻辑正确就行"的写法在 GPU 上经常很慢

这一章的目标是：

1. 讲清 CPU 和 GPU 在设计目标上的根本差异。
2. 解释低延迟思维和高吞吐思维的区别。
3. 说明为什么 GPU 对规则访问、批量处理和数据布局异常敏感。
4. 为后面理解 GPU 内存层次、coalescing、occupancy、带宽瓶颈打下思维基础。

如果这一章真正理解了，后面很多概念都会顺很多。

---

## 3.2 CPU 和 GPU 的目标从一开始就不一样

### CPU 想解决什么问题

CPU 是为通用计算设计的。它面对的是各种复杂任务：

- 操作系统调度
- 文件系统
- 网络协议栈
- 浏览器逻辑
- 数据库事务
- 分支复杂的业务代码
- 低并发但控制流复杂的程序

因此，CPU 的设计偏好通常是：

- 单线程性能强
- 单次访问延迟尽量低
- 分支预测能力强
- 缓存层次丰富
- 控制流灵活
- 对不规则工作负载更友好

### GPU 想解决什么问题

GPU 最初服务于图形渲染，后来扩展到通用并行计算。它最擅长的是：

- 大量相似计算
- 规则的数据并行
- 高吞吐处理
- 用很多线程同时推进工作

因此，GPU 的设计偏好通常是：

- 并行线程极多
- 单个线程相对轻量
- 更强调吞吐而不是单次低延迟
- 依赖规则访问和批量访问
- 通过线程切换隐藏延迟

### 一个粗略但很有用的类比

可以把 CPU 和 GPU 想成两类完全不同的工厂。

CPU 更像一支高技能小队：

- 人不多
- 每个人都很强
- 擅长处理复杂、变化多的任务
- 遇到突发情况也能灵活应对

GPU 更像一条超大规模流水线：

- 工位非常多
- 每个工位的动作更简单
- 最适合大量重复、规则的任务
- 最怕原料供应混乱或流程不整齐

这个类比虽然简化，但足够帮助你建立最关键的直觉。

---

## 3.3 低延迟思维和高吞吐思维

### CPU 更看重低延迟

CPU 经常追求的是：

- 一次操作尽快完成
- 一个线程尽快往前推进
- 尽量减少单个请求的等待时间

所以 CPU 会投入大量硬件成本去降低单次访问延迟，例如：

- 更复杂的缓存
- 更激进的预测
- 更强的乱序执行
- 更高的控制流灵活性

### GPU 更看重高吞吐

GPU 更常关心：

- 单位时间内总共完成多少工作
- 有多少线程能持续推进
- 能否把硬件管线喂饱
- 能否把内存等待隐藏在别的线程工作后面

GPU 不一定让每个线程都特别快，但它努力让整个芯片持续忙碌。

### 为什么这会影响内存理解

这意味着 GPU 面对内存延迟时，策略不是主要去"缩短每一次等待"，而是"在等待时让别人先上"。

也就是说，GPU 常常通过下面方式隐藏访存延迟：

- 同时挂起很多线程
- 某些线程等数据时，切去执行别的线程
- 借助大量并发保持计算单元不闲着

这就是为什么后面会出现 `occupancy`、`latency hiding` 这些概念。它们本质上都和 GPU 的高吞吐设计有关。

---

## 3.4 为什么 GPU 特别依赖规则访问

你可以把 GPU 想成一个特别擅长处理"整齐队列"的系统。

如果很多线程：

- 做的是相似操作
- 访问的是相邻地址
- 控制流也相近

那么 GPU 的效率会很高。

反过来，如果很多线程：

- 走不同分支
- 访问零散地址
- 各自处理完全不同的数据形状

GPU 的效率通常就会明显下降。

### 原因有两个

#### 1. 线程组织方式决定了它偏爱整齐工作

GPU 通常以 warp 之类的执行粒度组织线程。一个 warp 中的线程在硬件层面往往会协同执行。如果它们访问的是一段相邻地址，硬件就更容易把这些访问合并成少量高效事务。

如果每个线程都跳到不同位置去读，事务数就会上升，通路利用率就下降。

#### 2. GPU 依赖规模化并行来摊薄成本

GPU 上很多优化都建立在"批量化"的前提上：

- 批量发起访存
- 批量切换线程
- 批量使用 cache line
- 批量利用带宽

这意味着零散、不规则、不可预测的访问会破坏这种规模化优势。

---

## 3.5 为什么大量线程并不能自动解决问题

这是一个常见误区：GPU 线程很多，所以是不是问题自然就会被"并行"掩盖掉？

答案是否定的。

大量线程只能在某些条件下帮助你隐藏延迟，但它们不能修复本质上的访问低效。

### 线程多，并不等于访问就高效

如果每个线程都在做低效访问，例如：

- 随机读显存
- 访问跨度很大
- 访问不对齐
- 访问模式无法合并

那么线程再多，也只是"很多线程一起低效"。

### 线程多，也不等于资源无限

大量线程需要竞争有限资源，例如：

- 寄存器
- shared memory
- cache
- 显存带宽
- 调度槽位

如果一个 kernel 占用了太多寄存器或 shared memory，那么同时能驻留的线程块数就会减少，并发度反而下降。

### 线程多，还可能让问题放大

当访问模式差时，线程越多，越容易让：

- 内存事务爆炸
- cache 压力增大
- L2 冲突加剧
- 带宽更快打满

所以"大量线程"不是万能药。GPU 能高效运行的前提，是这些线程以硬件友好的方式工作。

---

## 3.6 CPU 上自然的写法，为什么在 GPU 上常常低效

这一节非常关键，因为它解释了很多"从 CPU 移植到 GPU 的代码为什么不理想"。

### 1. 指针追逐

在 CPU 上，链表、树、图等指针结构虽然可能不如数组快，但常常还能接受，尤其当工作集不大时。

在 GPU 上，指针追逐的问题会明显放大：

- 不同线程可能跳到完全不同地址
- 访问缺乏连续性
- cache 效果差
- 难以合并访存

所以很多 GPU 程序更偏好：

- 扁平数组
- 索引而不是指针
- 结构化批量布局

### 2. 小对象大量分配

在 CPU 上，小对象分配虽然也有成本，但成熟分配器通常能相对平稳地处理。

在 GPU 上，如果你频繁申请释放很多小块设备内存，问题会更明显：

- 分配调用更贵
- 更容易引入同步
- 更容易产生碎片
- 更难稳定复用

这就是为什么 GPU 工程里特别强调：

- 预分配
- 池化
- arena
- buffer 复用

### 3. 随机访问

CPU 有强大的缓存和复杂乱序机制，在一些随机访问场景里还能维持一定效率。

GPU 更怕大量随机访问，因为这会破坏：

- 合并访问
- 局部性
- 带宽利用
- cache 利用

### 4. 零散控制流

CPU 擅长复杂控制流，分支代价相对可控。

GPU 的并行执行更偏向"大家一起做类似的事"。如果同一批线程走不同路径，会造成执行效率下降。虽然这不是纯内存问题，但它经常和内存访问模式一起出现，使整体性能更差。

### 5. 面向对象式的数据布局

在 CPU 上，面向对象结构、层层封装、对象之间互相引用，通常是常见且可维护的写法。

在 GPU 上，如果这种风格导致数据分散在内存各处，就会显著伤害访问效率。GPU 往往更偏好：

- 紧凑布局
- 可批处理布局
- 可预测步长布局
- 面向数组而不是面向对象的访问方式

这并不是说 GPU 上不能抽象，而是说抽象不能破坏数据局部性和访问规律。

---

## 3.7 为什么 GPU 程序经常要"为了内存改算法"

这也是 CPU 经验迁移到 GPU 时很容易忽略的一点。

在 CPU 上，我们常说算法复杂度最重要，例如从 `O(n^2)` 改到 `O(n log n)` 往往是决定性的。

在 GPU 上，复杂度当然仍然重要，但很多时候，**访问模式会决定一个复杂度不错的算法是否真的跑得快**。

这就导致 GPU 上常见一种现象：

- 算法理论上没问题
- 但要为了更好的内存访问重排计算顺序
- 或者把数据切成 tile
- 或者把中间结果暂存到 shared memory
- 或者牺牲一点重复计算来换更好的局部性

也就是说，GPU 上的算法设计，经常不是"纯算术最优"，而是"算术、访存、并行度三者之间的折中最优"。

这对深度学习算子、矩阵运算、图像卷积、排序、稀疏计算都成立。

---

## 3.8 一种更适合 GPU 的思维方式

如果你想更自然地理解后面的内容，建议逐步建立下面这套 GPU 思维。

### 1. 不只问"算什么"，还要问"数据怎么走"

写一个 kernel 或设计一个系统时，不要只想：

- 输入是什么
- 输出是什么
- 公式是什么

还要想：

- 数据从哪里来
- 放在哪一层
- 是否能连续访问
- 是否能被复用
- 是否值得搬到 shared memory
- 是否会产生过多事务

### 2. 不只问"对不对"，还要问"是否整齐"

在 GPU 上，整齐往往意味着高效。所谓整齐，包括：

- 相邻线程访问相邻数据
- 相似线程做相似工作
- 缓冲区生命周期可预测
- 分配和回收可以批量化
- 数据布局适合批处理

### 3. 不只问"单次快不快"，还要问"整体吞吐顺不顺"

GPU 优化常常不是把一个点做到极致，而是让整条流水线尽量顺：

- 传输和计算是否能重叠
- 中间 buffer 是否能复用
- kernel 是否太碎
- 是否有不必要同步
- 是否总有线程可执行

### 4. 不只问"内存够不够"，还要问"生命周期如何重叠"

显存预算不是静态表格，而是时间曲线。你需要思考：

- 哪些对象必须同时存在
- 哪些对象可分阶段复用
- 哪些对象可以通过分块降低峰值
- 哪些缓存该常驻，哪些该按需生成

---

## 3.9 一个对比案例：CPU 友好 vs GPU 友好

我们用一个简化例子帮助建立直觉。

假设要处理一百万个元素。

### CPU 友好的想法

你可能会写成：

- 每个元素是一个对象
- 对象中有多个字段
- 某些字段是指针，指向其它结构
- 处理时根据条件跳来跳去

这在 CPU 上未必不可接受，尤其如果业务逻辑复杂、开发效率优先。

### GPU 友好的想法

在 GPU 上更常见的思路会是：

- 把对象摊平为多个连续数组
- 同一类字段放在一起
- 用索引代替指针
- 把相似工作按批组织
- 尽量让一个 warp 的线程访问相邻地址
- 尽量避免线程之间走完全不同的控制路径

两者并不是谁"更高级"，而是谁更符合底层硬件的偏好。

这就是为什么 GPU 编程经常要求你重构数据组织方式，而不是简单地把 CPU 代码翻译一遍。

---

## 3.10 本章小结

这一章最核心的结论有五条。

1. CPU 和 GPU 的设计目标根本不同。
CPU 更强调低延迟、灵活控制和单线程能力；GPU 更强调大规模并行和高吞吐。

2. GPU 面对内存延迟的方式，不是主要去降低单次访问时间，而是用大量线程和快速切换去隐藏等待。

3. GPU 对规则访问、连续布局和批量处理高度敏感，因为这决定了访存能否合并、带宽能否吃满、并行度能否真正发挥。

4. CPU 上自然的编程风格，例如指针追逐、小对象频繁分配、随机访问、复杂对象布局，在 GPU 上往往会带来显著性能问题。

5. GPU 编程本质上不是只写"计算逻辑"，而是在设计一条数据如何被高效组织、搬运、访问和复用的流水线。

到这里，你已经完成了最重要的一次思维切换。

下一章会继续往下走，进入 GPU 真正的硬件内存层次：寄存器、shared memory、L1/L2、global memory 各自是什么，它们为什么快慢不同，又是如何共同决定 kernel 性能上限的。

# 第 4 章 GPU 硬件内存层次：从寄存器到显存

## 4.1 本章目标

前面两章已经建立了两件事：

1. GPU 内存管理是整条系统链路中的核心问题。
2. 理解 GPU，必须先从 CPU 思维切换到 GPU 思维。

这一章开始进入更"硬"的部分：GPU 自己内部的内存层次。

很多初学者说"GPU 内存"，其实心里只想到"显存"。但这远远不够。对一个 kernel 来说，数据可能出现在很多不同层次：

- 寄存器
- shared memory
- L1 cache
- L2 cache
- global memory / VRAM
- 某些只读缓存，如 constant cache、texture cache

这些层次的差别，决定了三件非常重要的事：

1. 数据访问到底快不快
2. 线程并发到底能开多大
3. 一个 kernel 为什么会在某个资源上先撞墙

本章的目标是：

1. 建立 GPU 硬件内存层次的全图。
2. 解释每一层的容量、速度、作用和可见范围。
3. 讲清 occupancy、latency hiding、资源约束和性能之间的关系。
4. 为后面理解 coalescing、shared memory 优化、带宽瓶颈打下底层基础。

---

## 4.2 GPU 内存层次的整体直觉

可以先把 GPU 内存层次粗略理解成这样一张图：

```text
线程私有，最快，最少
  └─ 寄存器

线程块共享，片上，较快，容量有限
  └─ shared memory

硬件自动管理的片上/片外缓存
  ├─ L1 cache
  ├─ texture / constant cache
  └─ L2 cache

容量大但访问代价高
  └─ global memory / VRAM
```

如果只看直觉，可以记成一句话：

**越靠近计算单元，通常越快、越小、越稀缺；越远离计算单元，通常越慢、越大、越便宜。**

这和 CPU 的多级缓存思路有相似之处，但 GPU 的重点不同：

- CPU 更强调让单线程尽量少等待
- GPU 更强调让大批线程持续有活可干

因此，GPU 的内存层次不只是"加速访问"，还深度参与并行度管理。

---

## 4.3 寄存器：最快，但也是最稀缺的资源

### 直觉理解

寄存器可以理解为"离计算单元最近的临时工作台"。

一个线程在执行时，如果某个局部变量、循环计数器、中间结果能放进寄存器，那么访问代价通常是最低的。对 kernel 来说，寄存器是最理想的存储位置。

### 寄存器的几个关键特征

- 速度最快
- 通常线程私有
- 容量极小
- 数量固定，是非常稀缺的硬件资源

这里最重要的一点不是"寄存器快"，而是"寄存器非常有限"。

### 为什么寄存器会影响并发

GPU 上一个 SM 能同时驻留多少线程块、多少 warp，不只受线程数限制，还受寄存器总量限制。

例如，假设某个 kernel 中每个线程用了很多寄存器，那么即使线程块规模不大，也可能因为总寄存器预算不够，导致同一时间只能放很少的线程块上来执行。

这会直接影响 occupancy，也就是单位时间内能挂在硬件上的活跃线程数量。

### 一个重要结论

**寄存器用得越多，不一定越快。**

因为更多寄存器意味着：

- 单个线程状态更"豪华"
- 但同一时刻能并行驻留的线程更少

如果并发度因此下降太多，GPU 隐藏延迟的能力就会下降，整体反而可能变慢。

### 工程含义

优化时不能只看"某个变量是否进寄存器"，还要看：

- 寄存器使用量是否把 occupancy 压得过低
- 是否值得为了更多并发而让一部分值落到别的层次
- 编译器生成的寄存器使用是否过高

---

## 4.4 local memory：名字像本地，实际常常不"本地"

这是 GPU 学习里一个非常容易误解的概念。

### 为什么它叫 local memory

"local"在这里通常指的是"线程私有可见"，而不是"物理上很近"。

这和很多人直觉不同。很多人第一次看到 local memory，会以为它是某种接近寄存器的高速线程私有存储。通常不是这样。

### 它为什么会出现

当线程需要的私有数据太多，寄存器放不下时，编译器可能把部分内容溢出到 local memory。常见情形包括：

- 局部数组太大
- 寄存器压力过大
- 某些变量难以保持在寄存器中

### 为什么要特别警惕

因为 local memory 在很多实现中，底层访问路径会接近 global memory，也就是显存路径，而不是寄存器路径。

这意味着：

- 它虽然在线程语义上是"私有"
- 但访问代价可能远高于你直觉中的"本地临时变量"

### 工程含义

如果你发现一个看似简单的 kernel 很慢，其中一个可能原因就是：

- 编译器发生了 register spilling
- 某些变量落到了 local memory
- 导致实际访问变成昂贵的显存式访问

因此，"线程私有"不等于"便宜"。

---

## 4.5 shared memory：线程块内共享的片上存储

### 直觉理解

如果说寄存器是每个线程自己的小工作台，那么 shared memory 就像线程块内部共享的一张大工作台。

同一个线程块内的线程可以把数据放进 shared memory，然后彼此读取和复用。这让它特别适合：

- 协作计算
- 数据重用
- 分块算法
- 临时缓存热点数据

### shared memory 的关键特点

- 速度通常远快于显存
- 可被同一个线程块中的线程共享
- 容量有限
- 由程序员显式使用和管理
- 生命周期通常和线程块绑定

### 为什么 shared memory 很重要

因为 GPU 的一个常见优化套路就是：

1. 先把会被多次访问的数据，从 global memory 搬进 shared memory
2. 在线程块内部多次复用
3. 减少重复访问显存

这在矩阵乘、卷积、图像处理、tile-based 算法中非常常见。

### shared memory 不是免费午餐

虽然它快，但有几个代价：

- 容量有限
- 用多了会压低 occupancy
- 需要程序显式同步
- 访问不当会出现 bank conflict

所以它不是"越多越好"，而是"该用时精准地用"。

---

## 4.6 bank conflict：shared memory 里的典型陷阱

### 先用直觉理解

可以把 shared memory 想成被拆成多个 bank 的并行小通道。理想情况下，不同线程访问不同 bank，就能并行进行。

但如果多个线程同时访问同一个 bank 的不同位置，就可能产生冲突，导致原本可以并行的访问被串行化一部分，这就是 bank conflict。

### 为什么它重要

很多人第一次优化 shared memory 时，只想到：

- shared memory 比 global memory 快
- 所以搬进去肯定更快

这并不总成立。如果访问模式导致严重 bank conflict，那么 shared memory 的收益会明显下降。

### 工程含义

使用 shared memory 时，要特别关心：

- 数据在 shared memory 中的排布方式
- 线程索引和数据索引的关系
- 是否需要 padding
- 是否有固定跨步访问导致大量线程撞到同一个 bank

这说明 GPU 内存优化不仅是"用哪一层"，还包括"在这一层里如何布局"。

---

## 4.7 L1 cache、texture cache、constant cache：硬件帮你做的一部分缓存

### L1 cache

L1 cache 是离执行单元较近的缓存层之一，用来缓存最近或局部访问的数据。不同 GPU 架构的具体组织方式会变化，但核心作用可以粗略理解为：

- 降低一部分访存延迟
- 提升局部性好的访问效率
- 减轻更下层存储的压力

不过，GPU 的 L1 不应该被理解成"像 CPU L1 那样可以兜住一切"。GPU 的总体设计仍然更依赖大规模并行和访问规律。

### constant cache

constant memory / constant cache 通常适合：

- 小规模只读数据
- 许多线程反复读取同样的常量
- 参数表、固定配置等

如果一个 warp 中很多线程访问的是同一个常量位置，这类只读缓存通常非常高效。

### texture cache

texture 路径最早来自图形处理，后来也被用于某些通用只读访问场景。它对某些空间局部性良好的只读访问模式很有帮助，尤其在图像、采样类问题中经常出现。

### 工程含义

这些缓存的共同点是：

- 它们试图自动提高一部分访问效率
- 但前提仍然是访问模式足够友好

你不能指望"有缓存"就自动救活糟糕的数据布局。GPU 缓存更像是锦上添花，而不是替代访问优化。

---

## 4.8 L2 cache：整个 GPU 共享的重要缓冲层

### 直觉理解

L2 cache 可以看成 GPU 范围内更大、共享的一层缓存。它不属于单个线程块，也不只服务某个局部执行单元，而是服务更广范围的访存活动。

### 它的重要性

L2 的作用通常包括：

- 缓存 global memory 的部分热点数据
- 降低对 VRAM 的直接访问压力
- 在多个 SM 之间起到某种共享缓冲作用

### 为什么 L2 很关键

很多 global memory 访问是否真的"贵到极致"，很大程度上取决于：

- 是否命中 L2
- 访问模式是否产生复用
- 多个线程块是否在读相同或邻近的数据

如果访问完全随机，工作集很大，局部性又差，那么 L2 的帮助就有限。

### 工程含义

L2 让你理解一个重要事实：

**global memory 访问不一定每次都直达最慢路径，但是否能受益于缓存，取决于你的访问模式是否值得被缓存。**

---

## 4.9 global memory / VRAM：容量大，但不是"随便用都行"

### 它是什么

global memory 通常就是程序最常接触到的"设备内存"，底层通常对应显存 VRAM。

你通过 `cudaMalloc` 拿到的很多设备指针，本质上最终都是在使用这一层。

### 它的特点

- 容量最大
- 对所有合适范围内的线程都可见
- 相比寄存器和 shared memory，访问延迟高得多
- 访问性能高度依赖模式和带宽利用

### 为什么它不是"只要有就行"

很多初学者认为：

- 数据能放进显存就行
- 算法主要由算术复杂度决定

这在 GPU 上通常不成立。对 global memory 来说，关键不是只有"能不能放下"，还包括：

- 是否频繁访问
- 是否访问合并
- 是否有局部性
- 是否反复读写本可复用的数据
- 是否把热点数据留在了更高层次

### 一个核心事实

**global memory 的性能不是一个常数，而是一个由访问模式决定的结果。**

同样是访问显存：

- 合并、连续、对齐访问，可能接近理想带宽
- 零散、随机、跨步访问，可能浪费大量事务

所以显存不是"最后的答案"，而是"必须谨慎使用的大容量后备层"。

---

## 4.10 occupancy：为什么并发能力和内存资源绑在一起

### 定义的直觉版本

occupancy 可以粗略理解为：某个时刻，一个 SM 上实际能维持多少活跃线程或 warp，与理论上限相比的比例。

它反映的不是"程序快不快"，而是"硬件同时挂了多少活"。

### 为什么它和内存层次有关

occupancy 受很多资源共同限制，其中和内存层次最相关的就是：

- 每线程寄存器使用量
- 每线程块 shared memory 使用量

如果一个 kernel：

- 每个线程用很多寄存器
- 每个线程块占很多 shared memory

那么同一时刻能驻留的线程块数量就会减少，活跃 warp 数也会减少。

### 为什么 occupancy 重要

GPU 需要靠大量活跃线程来隐藏延迟。如果 occupancy 太低，那么当一批线程因为等待内存而停住时，硬件能切换去执行的其它线程也不够多，延迟就更容易暴露出来。

### 但 occupancy 不是越高越好

这点非常关键。

更高 occupancy 有时确实有帮助，但它不是最终目标。因为：

- 为了提高 occupancy，你可能被迫减少寄存器使用
- 这可能导致更多 spilling
- 或者 shared memory 太少，无法实现数据复用
- 反而让每个线程的执行效率下降

因此，occupancy 更像一个"约束与平衡问题"，不是单一追求的分数。

---

## 4.11 latency hiding：GPU 如何"假装不怕慢内存"

### 为什么显存访问慢，但 GPU 还能快

这是 GPU 体系结构中一个特别核心的点。

显存访问很慢，远慢于寄存器和 shared memory，但 GPU 并没有因此彻底变慢。原因在于，它采用的是"隐藏延迟"而不是"完全消灭延迟"的策略。

### 具体直觉

当一个 warp 发起访存后，如果数据还没回来，这个 warp 可以先挂起。调度器会去执行别的、已经准备好的 warp。只要这种"别的 warp"足够多，计算单元就不会闲下来。

这就是 latency hiding。

### 成功隐藏延迟的前提

想让 GPU 隐藏延迟，通常需要：

- 足够多的活跃 warp
- 不是所有 warp 都同时卡在同类等待上
- 有足够独立工作可切换
- 内存系统没有差到把所有线程都一起堵死

### 工程含义

这解释了为什么：

- 并发度很重要
- 寄存器和 shared memory 用量会影响性能
- 某些访存密集型 kernel 需要更多活跃线程来掩盖等待

也解释了为什么当 occupancy 太低时，显存延迟会暴露得特别明显。

---

## 4.12 带宽和延迟不是一回事

这两个概念经常被混淆，但必须严格分开。

### 延迟

延迟回答的是：

- 一次访问要等多久

它更像"一个包裹从发货到送到我手里要多长时间"。

### 带宽

带宽回答的是：

- 单位时间能运多少数据

它更像"这条物流线路一小时能送多少吨货"。

### 为什么要区分

一个系统可能：

- 单次访问延迟很高
- 但只要请求足够多、组织得足够好，总吞吐仍然很高

这正是 GPU 的常见特点。

### GPU 内存优化里两者分别意味着什么

如果你在优化延迟，通常会关注：

- 是否使用更近层次的存储
- 是否提高 occupancy 去隐藏等待
- 是否减少同步和阻塞

如果你在优化带宽，通常会关注：

- 是否访问合并
- 是否减少不必要流量
- 是否提高数据复用
- 是否降低事务浪费

两者相关，但不是一回事。

---

## 4.13 一个简单的层次总结表

可以先建立一个非严格但非常有用的印象表：

| 层次 | 速度 | 容量 | 可见范围 | 典型特点 |
|---|---|---:|---|---|
| 寄存器 | 最快 | 最小 | 单线程 | 线程私有，极稀缺 |
| shared memory | 很快 | 小 | 线程块内 | 显式管理，适合复用 |
| L1/只读缓存 | 较快 | 小 | 局部硬件范围 | 自动缓存，依赖局部性 |
| L2 | 中等偏快 | 较大 | 全 GPU | 全局共享缓存层 |
| global memory / VRAM | 最慢之一 | 最大 | 广泛可见 | 容量大，访问模式决定性能 |

这个表不追求架构细节绝对精确，但足够帮助你建立后续分析的方向。

---

## 4.14 本章小结

这一章最重要的结论有六条。

1. GPU 内存不是只有显存，而是一个分层系统。
2. 越靠近计算单元的层次通常越快，但容量越小、资源越稀缺。
3. 寄存器和 shared memory 不仅影响访问速度，也直接影响并发度和 occupancy。
4. local memory 虽然在线程语义上私有，但性能上往往不能被当成便宜存储。
5. global memory 容量大，但性能高度依赖访问模式，不能把它当成"随便读都一样快"的统一大仓库。
6. GPU 的一个核心策略是通过高并发去隐藏内存延迟，因此资源占用、并发能力和访存效率始终绑在一起。

下一章我们从硬件层再往上走一层，进入系统视角：显存、地址空间、页表、UVA、UVM 到底是什么。那一章会回答一个常被忽略的问题：为什么 GPU 内存管理不是"申请一块显存"这么简单，而是地址空间、映射和迁移问题。

# 第 5 章 显存不是一块大数组：地址空间、页表与虚拟内存

## 5.1 本章目标

上一章我们从 GPU 硬件内部看了内存层次，重点是：

- 数据放在哪一层
- 哪一层快
- 哪一层稀缺
- 为什么资源会影响并发和性能

这一章开始切换到系统视角，回答另一个非常重要的问题：

**一块"GPU 内存"到底是怎么被系统组织和看见的？**

很多初学者把 GPU 显存理解成"设备上的一块连续大数组"，程序申请一块空间后，GPU 就直接读写它。这个直觉太粗糙了。

真实情况更接近下面这套链路：

1. 有虚拟地址空间
2. 有物理显存页
3. 有地址映射关系
4. 有 CPU 与 GPU 之间的数据通路
5. 有 DMA、IOMMU、BAR、UVA、UVM 等机制在背后协作

本章的目标是：

1. 解释 VRAM、主机内存、地址空间和页映射的基本关系。
2. 讲清 UVA 和 UVM 的意义与边界。
3. 解释 pinned memory 和 pageable memory 的差别。
4. 为后面理解 `cudaMalloc`、`cudaMemcpyAsync`、统一内存迁移和异步传输打基础。

---

## 5.2 先区分三件事：主机内存、显存、地址空间

### 1. 主机内存

主机内存通常指 CPU 所在系统的 DRAM，也就是你平时理解的"系统内存"或 RAM。

CPU 程序默认最常直接使用这一层。操作系统会管理它的虚拟地址空间、物理页、分页、缓存等。

### 2. 显存

显存通常指 GPU 设备上的 VRAM。它是 GPU 主要的大容量设备内存，供 GPU 执行 kernel、存放张量、纹理、缓冲区等使用。

### 3. 地址空间

地址空间不是"某块物理内存"，而是"程序看到的地址视图"。

这是关键点。

一个指针首先属于某个地址空间视图。这个地址到底映射到哪里：

- 可能是主机物理页
- 可能是设备物理显存页
- 可能还需要迁移、映射或转换后才能实际访问

### 为什么这三者要分开

因为很多 GPU 内存相关问题，本质上不是"物理有没有空间"，而是：

- 这个地址属于谁的视图
- 谁可以访问它
- 访问路径经过什么
- 虚拟地址和物理页如何建立关系
- 访问时是否需要迁移或拷贝

---

## 5.3 虚拟地址：为什么程序看到的不是真实物理位置

### CPU 世界里你已经在用虚拟地址

其实你在普通 CPU 编程里，早就习惯了虚拟地址。你写 C/C++ 时拿到的指针，一般也不是"真实物理地址"，而是进程虚拟地址空间中的地址。

操作系统通过页表把这些虚拟地址映射到物理页。

### GPU 世界里也类似

GPU 编程中，很多设备指针本质上也要通过某种地址空间机制来解释。它们不是简单等价于"某颗显存芯片上的裸物理位置"。

这意味着：

- 设备内存也可能通过虚拟地址来管理
- 运行时和驱动可能需要保留地址区间
- 物理显存页和虚拟地址之间要建立映射

### 这样做的好处

虚拟地址的好处包括：

- 简化编程模型
- 让地址管理更灵活
- 方便重映射和页迁移
- 让 CPU 和 GPU 之间的统一地址视图成为可能
- 支持更复杂的内存管理策略

所以，GPU 内存管理并不是在"裸金属显存上直接切块"那么简单。

---

## 5.4 页与页表：不是整块管理，而是按页组织

### 什么是页

页可以理解为内存管理中的基本块单位。无论是 CPU 内存还是 GPU 相关地址映射，通常都不会逐字节管理，而会按固定大小的页来组织。

### 什么是页表

页表记录了虚拟地址页到物理页的映射关系。

粗略理解就是：

- 程序看到的是虚拟地址
- 硬件或系统要根据页表找到实际物理位置

### GPU 里为什么也要有这套机制

因为设备侧同样需要解决：

- 如何组织大规模内存
- 如何支持地址空间抽象
- 如何让不同对象映射到合适位置
- 如何支持统一地址视图和迁移

### 工程意义

理解页和映射后，你会更容易明白：

- 为什么分配不只是"拿块内存"那么简单
- 为什么统一内存会涉及 page migration
- 为什么某些访问会触发 page fault
- 为什么碎片化和可用连续块是不同问题

---

## 5.5 UVA：统一虚拟地址空间是什么

### 先说直觉

UVA，Unified Virtual Addressing，可以粗略理解为：

**让 CPU 和 GPU 在编程视角上尽量共享一个统一的虚拟地址体系。**

这并不等于 CPU 和 GPU 从此完全没有边界，也不等于任何地址谁都能无成本直接访问。它更像是：

- 统一"地址表达方式"
- 降低主机地址和设备地址之间的割裂感
- 为更高级的管理机制打基础

### 它解决了什么问题

没有统一虚拟地址时，主机指针和设备指针的世界可能更割裂，程序和运行时需要更明确地区分两套地址体系。

有了 UVA 后，很多情况下可以：

- 更一致地识别地址属于哪一侧
- 让某些 API 交互更自然
- 为 peer access、managed memory 等机制铺路

### 它没有解决什么问题

UVA 不等于：

- 自动消除数据拷贝
- 自动让所有访问都高效
- 自动解决 CPU/GPU 一致性和迁移问题

所以不要把 UVA 理解成"从此主机和设备就是一块内存"。它更像"统一视图"，不是"统一物理现实"。

---

## 5.6 pinned memory 和 pageable memory：异步传输的关键分界线

这是 GPU 编程里一个必须吃透的概念。

### pageable memory 是什么

这是普通主机内存的默认状态。它可以被操作系统分页、迁移、换出，物理位置并不总是稳定固定。

从 CPU 程序角度，这通常没问题，因为虚拟内存系统会帮你处理。

### pinned memory 是什么

pinned memory 又叫 page-locked memory，可以理解为：

- 这块主机内存的物理页被"钉住"
- 不会随意被操作系统移动或换出
- 更适合 DMA 直接访问

### 为什么 pinned memory 很重要

GPU 做主机到设备传输时，常常依赖 DMA。DMA 想高效工作，有个重要前提：数据源或目标的物理位置要足够稳定、可直接访问。

如果主机内存是 pageable 的，那么运行时往往需要额外步骤，例如：

- 先把数据临时搬到一个适合 DMA 的固定区
- 再由 DMA 传到 GPU

这会增加额外开销，也会削弱异步传输效果。

### 一个非常实用的结论

**真正高效的异步 H2D / D2H 传输，通常离不开 pinned memory。**

这也是为什么很多人调用了 `cudaMemcpyAsync`，却发现效果不明显，原因往往不是"异步 API 失效"，而是数据源本身不满足高效异步传输条件。

### pinned memory 的代价

它也不是越多越好。因为：

- 它会增加主机内存管理压力
- 被钉住的页对操作系统更不灵活
- 过量使用可能影响系统整体行为

所以 pinned memory 是一种高性能资源，不是普通内存的无脑替代品。

---

## 5.7 DMA：为什么数据可以不靠 CPU 一点点搬

### 直觉理解

DMA，Direct Memory Access，可以理解为：

- 某些数据传输可以不由 CPU 核心自己逐字节搬运
- 而由专门的数据通路和控制机制完成

### 为什么它重要

CPU 和 GPU 之间的大块数据传输，如果全靠 CPU 自己搬，成本会非常高，也会占用大量 CPU 资源。

DMA 的意义在于：

- 让传输更高效
- 让 CPU 不必全程亲自参与数据逐字节移动
- 为异步传输和计算重叠提供基础

### 它和 pinned memory 的关系

DMA 想高效读取主机内存，通常更需要主机侧提供稳定、固定的物理页，也就是 pinned memory。

因此这两者经常一起出现。

---

## 5.8 IOMMU 与 BAR：你不必深挖实现，但要知道它们在系统中做什么

### IOMMU

可以把 IOMMU 粗略理解成：

- 面向 I/O 设备访问的一套地址转换和保护机制

它帮助设备侧访问主机内存时，也能有更灵活和更安全的地址转换能力。

### BAR

BAR 可以粗略理解为设备暴露给系统的一部分地址映射窗口，帮助 CPU 或系统软件与设备资源建立访问关系。

### 为什么只需要理解到这个程度

在一篇以 GPU 内存管理为主线的教程里，这些机制不必展开到操作系统内核或总线协议级别。但你必须知道：

- GPU 和主机之间的访问不是"天然直接无缝"
- 中间有设备访问路径、地址转换和系统级机制在支撑
- 这解释了为什么很多看似简单的内存访问，背后其实是复杂系统协作

---

## 5.9 UVM / managed memory：更方便，但不是没有代价

### 先说目标

UVM，Unified Virtual Memory，或者通常说的 managed memory，目标是进一步降低 CPU 和 GPU 内存管理的割裂感。

理想效果是：

- 程序员不必总是手动决定什么时候 `cudaMemcpy`
- 数据可以按需在 CPU 和 GPU 之间迁移
- 访问模型更统一

### 为什么它看起来很吸引人

因为它简化了编程心智：

- 你少写很多显式拷贝代码
- 更容易写原型
- 更容易管理复杂数据结构
- 更容易让程序先"跑起来"

### 真正代价在哪里

代价不在于"它不好用"，而在于"它把复杂性从代码层转移到了运行时行为层"。

这会带来几个问题：

- 访问时可能触发按需页迁移
- 可能出现 page fault
- 迁移时机会变得不那么可预测
- 程序延迟可能抖动
- 在高性能路径中，自动迁移可能不如手动管理高效

### 一个典型误区

很多人会以为：

- 统一内存 = 不用管内存了

更准确的说法应该是：

- 统一内存 = 你少管了一部分显式操作，但必须理解背后的迁移和访问代价

---

## 5.10 page migration：数据不是总在你以为的地方

### 直觉理解

如果一段 managed memory 当前驻留在 CPU 侧，而 GPU 开始访问它，那么系统可能需要把对应页面迁移到 GPU 更适合访问的位置。

反过来，如果 GPU 上的数据随后又被 CPU 高强度访问，也可能发生反向迁移。

### 为什么它重要

因为它告诉你一个很重要的现实：

**在统一内存模型下，"地址能访问"不等于"数据已经在最合适的地方"。**

地址视图可以统一，但物理驻留位置仍然是动态的。

### 工程后果

如果程序访问模式非常稳定、迁移可预测，那么统一内存可能表现不错。

但如果程序：

- CPU/GPU 交替访问同一批页面
- 访问范围跳跃大
- 工作集大于设备合适容量
- 迁移节奏频繁

那么 page migration 会成为明显成本。

---

## 5.11 page fault：GPU 也可能"缺页"

### 直觉理解

page fault 可以理解为：

- 当前访问的虚拟页没有按照当前访问者所需的方式就绪
- 系统需要先处理映射、迁移或调页
- 然后访问才能继续

### 为什么这在 GPU 里值得关注

很多人习惯了 CPU 上 page fault 的概念，但不太会把它和 GPU 访问联系起来。实际上，在统一内存或更复杂的虚拟内存场景下，GPU 侧同样可能因为某页尚未就位而触发处理流程。

### 工程后果

这意味着某些访问延迟可能不是平滑稳定的，而是"突然慢一下"。在吞吐型程序里，这种不可预测性尤其值得警惕。

所以统一内存虽然省代码，但它可能把"显式 copy 的可见成本"变成"运行时 page fault 的隐式成本"。

---

## 5.12 "统一视图"不等于"统一性能"

这是本章最重要的提醒之一。

无论是 UVA 还是 UVM，它们都在努力减少编程视角上的割裂。但你必须记住：

- 统一地址，不等于统一物理位置
- 统一访问方式，不等于统一成本
- 统一编程模型，不等于统一性能结果

也就是说：

- 你可以在逻辑上把某块数据当成"统一可访问"
- 但工程上仍然要关心它到底驻留在哪、什么时候迁移、访问路径经过什么、是否适合当前工作负载

这一点会在后面讲 `cudaMemcpyAsync`、`cudaMallocManaged`、`cudaMemPrefetchAsync` 时不断出现。

---

## 5.13 本章小结

这一章最重要的结论有七条。

1. GPU 显存不是"简单的一块大数组"，而是处在地址空间、页映射和系统访问路径中的资源。
2. 程序看到的地址首先是虚拟地址视图，不应直接等同于裸物理位置。
3. UVA 统一的是地址表达和视图，不是物理现实和访问成本。
4. pinned memory 是高效 DMA 和真正异步主机设备传输的重要前提。
5. UVM / managed memory 让编程更方便，但会引入迁移、缺页和不可预测延迟等代价。
6. page migration 和 page fault 告诉我们：数据"能被访问"不代表"已经在最合适的地方"。
7. GPU 内存管理从一开始就是系统问题，而不只是 API 问题。

下一章我们会把这些系统概念继续往前推进，落到一个大家最熟悉的入口上：`cudaMalloc`。我们会回答，一次 CUDA 内存分配到底发生了什么，为什么分配和释放会贵，为什么 `cudaFree` 常常和同步问题联系在一起。

# 第 6 章 一次内存分配到底发生了什么：从 `cudaMalloc` 讲起

## 6.1 本章目标

前面几章已经把 GPU 内存管理的基础背景搭起来了：

- 第 4 章讲硬件层次
- 第 5 章讲地址空间、页映射、UVA/UVM

这一章开始把视角聚焦到一个最常见、最具体的入口上：`cudaMalloc`

很多人对它的理解停留在：

- 它和 `malloc` 差不多
- 只是分配的是 GPU 内存
- 用完 `cudaFree` 就行

这个理解太浅了。工程上关于显存分配的很多现象，比如：

- 为什么分配慢
- 为什么释放可能卡住
- 为什么频繁小块申请会让程序发抖
- 为什么框架喜欢缓存显存而不立刻释放
- 为什么 `cudaMallocAsync` 会出现

都和 `cudaMalloc` 背后真正做的事情有关。

本章的目标是：

1. 用系统视角解释一次设备内存分配的大致过程。
2. 讲清运行时、驱动、context、虚拟地址、物理页之间的关系。
3. 解释为什么分配和释放通常都不是零成本。
4. 为后面理解内存池、缓存分配器和异步分配器打基础。

---

## 6.2 从 API 表面看，`cudaMalloc` 做了什么

从程序员视角，`cudaMalloc` 很简单：

```cpp
float* ptr = nullptr;
cudaMalloc(&ptr, n * sizeof(float));
```

看起来你只是"向 GPU 申请了一块内存，并得到了一个指针"。

但这个指针背后至少隐含了几层语义：

1. 这块内存属于某个 CUDA context
2. 它在设备地址空间中有一个有效地址范围
3. 底层已经为它准备了可用的存储资源
4. 后续 kernel 可以拿这个地址进行访问
5. 运行时和驱动已经记录了相关分配状态

所以，`cudaMalloc` 不是单纯地"在显存数组上切一刀"，而是一次系统级资源建立过程。

---

## 6.3 CUDA Runtime、Driver、Context：分配不是孤立事件

### Runtime 和 Driver 的关系

通常你直接调用的是 Runtime API，比如 `cudaMalloc`。但 Runtime 并不是最终执行一切细节的最低层，它还要和更底层的 driver 机制协作。

粗略理解可以是：

- Runtime 提供更直接、更易用的编程接口
- Driver 负责更贴近底层资源和设备管理的部分

### Context 的作用

context 可以粗略理解为：

- 进程在某个 GPU 设备上的执行与资源环境
- 它持有与设备交互所需的大量状态
- 内存分配、stream、模块、事件等很多资源都与 context 关联

这意味着一块设备内存不是"脱离环境独立存在"的，它通常归属于某个 context 的资源体系。

### 为什么这很重要

因为很多分配行为、同步行为、资源生命周期行为，都不能脱离 context 理解。

这也解释了为什么：

- 初始化 CUDA 环境本身就有成本
- 某些操作第一次调用会更慢
- 内存与 stream、kernel、模块并不是完全无关的孤立对象

---

## 6.4 一次设备分配的大致路径

虽然具体实现会因 CUDA 版本、驱动和架构而变化，但从教学视角，可以把一次 `cudaMalloc` 粗略理解成下面几步。

### 第一步：参数检查与上下文准备

运行时先要确认：

- 当前设备和 context 是否可用
- 请求大小是否合法
- 当前环境是否已经初始化

有时，第一次调用 CUDA 相关 API 会触发额外初始化成本。

### 第二步：在设备地址空间中准备可用范围

系统需要为这次分配准备一段可用的设备虚拟地址空间。你最终拿到的指针，首先是这个地址空间中的一个可用地址。

### 第三步：申请底层物理显存页

接着要有真实的存储资源作为后盾。也就是说，底层需要有对应的物理显存页可供使用。

### 第四步：建立虚拟地址到物理页的映射

只有地址还不够，还要让这个地址真正对应到那批物理页，后续 GPU 访问时才能正确寻址。

### 第五步：更新分配器和运行时元数据

运行时和驱动还要记录：

- 这块内存的大小
- 所属 context
- 分配状态
- 可能的对齐和池化信息
- 后续释放时需要用到的元信息

### 第六步：返回设备指针

到这一步，程序才拿到一个可用的设备指针。

这个过程哪怕你不研究底层实现细节，也应该明白：它比普通堆上取一小块内存复杂得多。

---

## 6.5 为什么 `cudaMalloc` 往往比你想象中贵

### 1. 它涉及系统级资源管理

`cudaMalloc` 不是简单的用户态指针偏移操作。它通常涉及运行时、驱动、context、地址空间、底层显存资源和元数据管理。

### 2. 它可能需要和设备状态协调

GPU 不是一块静止仓库，而是在持续执行工作。新的分配可能需要和已有资源状态、地址空间状态协同。

### 3. 它可能不是纯本地小操作

有些成本体现在：

- 驱动交互
- 页映射建立
- 资源表维护
- 对齐与管理开销

### 4. 频繁调用会把固定开销放大

如果你只在程序开始时分配几块大 buffer，这些成本往往还可以接受。

但如果你在热路径中频繁分配很多小块，那么这些固定管理成本就会被无限放大，导致：

- 吞吐下降
- 延迟抖动
- 程序表现不稳定

这也是工程上极少建议"边算边频繁 `cudaMalloc`"的原因。

---

## 6.6 为什么 `cudaFree` 也可能很贵

初学者常以为：

- 分配贵可以理解
- 释放总该便宜吧

但在 GPU 上，释放未必便宜，甚至某些时候更麻烦。

### 原因一：系统要确认这块内存是否真的可以回收

如果这块设备内存还在被未完成的 kernel、拷贝或其它异步操作使用，那么系统不能立刻粗暴回收，否则会出错。

### 原因二：释放可能需要和执行流做同步协调

这就是 `cudaFree` 常和"隐式同步"联系在一起的原因之一。

粗略理解就是：

- 你在主机端说"这块内存不要了"
- 但设备端可能还有排队中的工作尚未彻底结束
- 系统必须确保不会回收一块仍在使用的内存

### 原因三：释放不仅是还回去，还要更新很多状态

包括：

- 分配器元数据
- 地址映射状态
- 可复用块的管理
- 可能的碎片合并逻辑

### 工程后果

这意味着：

- `cudaFree` 不应被当成廉价操作
- 在性能敏感路径中频繁 free 往往很糟
- 这正是内存池和缓存分配器存在的重要理由之一

---

## 6.7 隐式同步：为什么"只是释放内存"也会卡住

### 先说直觉

GPU 编程的很多操作是异步的。你在 CPU 上发起了操作，不代表 GPU 立刻做完。

于是当你要释放一块内存时，系统必须回答一个问题：

**这块内存后面还会不会被设备上的未完成工作用到？**

如果答案还不确定，就可能需要等待。

### 这类等待为什么危险

因为它打破了你的流水线节奏。

你原本可能希望：

- CPU 持续提交任务
- GPU 持续执行任务
- 两边尽量并行

但一次隐式同步可能突然把这条流水线打断，让 CPU 等 GPU，或者让后续分配/执行被迫串起来。

### 工程上的直接后果

这就是为什么实际工程中常见建议是：

- 避免频繁使用 `cudaFree`
- 尽量把分配释放从热路径移开
- 使用缓存分配器或 pool
- 必要时用 stream-ordered 的异步分配机制

---

## 6.8 为什么频繁小块分配特别容易出问题

这几乎是 GPU 内存管理里最常见的坏味道之一。

### 先看问题本质

频繁小块分配的问题不是只有一个，而是四个问题叠加：

1. 固定管理开销大
2. 更容易产生碎片
3. 更容易触发同步协调
4. 更难稳定复用和调优

### 为什么碎片更容易出现

如果内存不断以不同大小申请和释放，显存空间就可能逐渐形成很多空洞：

- 总空闲量看起来不少
- 但大块连续可用区变少
- 某次大分配就可能失败

### 为什么这在长期运行服务中特别明显

训练脚本可能运行一阵就结束，但推理服务、图形引擎、长期在线系统会持续进行分配和回收。这时小块动态分配更容易积累碎片和抖动。

### 工程含义

频繁小块分配通常说明你的程序需要重新设计：

- 是否能预分配
- 是否能池化
- 是否能按阶段复用
- 是否能统一分配粒度
- 是否能用 arena / slab / ring buffer

---

## 6.9 为什么框架和工程系统不喜欢"申请了就立刻还"

到这里你就能理解一个工程上的常见现象：

- 框架分配了显存
- 张量释放后并不立刻把显存还给底层
- `nvidia-smi` 看起来显存占用还很高

这并不一定是错误，更可能是一种设计选择。

### 原因很直接

如果每次对象生命周期结束都马上走到底层 `cudaFree`：

- 释放可能触发同步
- 下次再申请又要重新走昂贵路径
- 整体抖动很大

所以框架更常做的是：

1. 从系统拿较大的块
2. 内部切分给对象使用
3. 对象释放后只回收到框架池里
4. 以后再复用，而不是立刻归还底层

### 这带来的好处

- 降低系统级分配频率
- 避免大量同步
- 提高吞吐稳定性
- 更容易控制生命周期

### 带来的代价

- 表观显存占用高
- 更难从外部直接判断真实"活跃用量"
- 碎片管理变成框架自己的责任

这也是为什么后面要专门讲内存池和缓存分配器。

---

## 6.10 `cudaMemGetInfo` 能告诉你什么，不能告诉你什么

在工程里，很多人会想知道"显存还剩多少"，于是会看类似 `cudaMemGetInfo` 这样的信息。

### 它能告诉你的

粗略上，它能帮助你了解：

- 当前设备层面还有多少可用显存
- 总显存大致是多少

### 它不能完整告诉你的

它通常不能直接完整回答：

- 框架池里有多少块已保留但可复用
- 碎片化程度到底如何
- 某个时刻峰值到底有多高
- 某次分配失败是否因为缺总量还是缺连续块
- 某些异步操作是否仍持有尚未释放的内存

### 工程含义

所以你不能只看一个"剩余显存数字"就下结论。GPU 内存问题常常必须结合：

- 框架层统计
- 生命周期分析
- 分配轨迹
- profiler 时间线
- 服务运行时模式

来一起看。

---

## 6.11 为什么 `cudaMallocAsync` 会出现

虽然这一章不展开讲它，但现在已经可以先建立直觉。

既然传统 `cudaMalloc` / `cudaFree` 存在这些问题：

- 分配释放成本高
- 释放常牵扯同步
- 热路径使用不友好
- 碎片和复用压力大

那么更合理的方向自然是：

- 更好地池化
- 更好地和 stream 顺序结合
- 更少全局同步
- 更高效地复用块

这正是 `cudaMallocAsync` 和 memory pool 这类机制要解决的问题。

所以新分配器机制不是"功能变花哨"，而是对旧痛点的直接回应。

---

## 6.12 一个完整的心智模型：`cudaMalloc` 其实在做什么

你现在可以把 `cudaMalloc` 压缩成一个更准确的理解：

**它不是单纯"拿一块显存"，而是在某个 context 下，协调运行时、驱动、设备地址空间和物理显存资源，建立一段后续可安全访问的设备内存对象。**

而 `cudaFree` 则不是单纯"把这块显存删掉"，而是：

**在确保不破坏现有异步执行和访问正确性的前提下，回收这段设备内存对应的资源与元数据。**

只要你记住这个版本，后面关于同步、碎片、缓存池、异步分配的内容就都会变得顺理成章。

## 6.13 本章小结

这一章最重要的结论有八条。

1. `cudaMalloc` 不是简单的"设备版 malloc"，而是一次系统级资源建立过程。
2. 一次分配通常涉及 Runtime、Driver、Context、地址空间、物理页和映射关系。
3. 设备指针首先是地址空间中的可用地址，不应被简单理解成裸物理显存位置。
4. `cudaMalloc` 往往比想象中贵，因为它涉及多层管理和资源维护。
5. `cudaFree` 也可能很贵，尤其因为它常常要和未完成的异步工作做安全协调。
6. 释放可能引入隐式同步，这会打断流水线并造成抖动。
7. 频繁小块分配释放是 GPU 程序中的常见坏味道，会放大管理成本并加重碎片问题。
8. 这正是缓存分配器、内存池和 `cudaMallocAsync` 存在的根本原因。

# 第 7 章 数据如何在 CPU 与 GPU 之间流动

## 7.1 本章目标

前面几章已经建立了两个关键基础：

1. GPU 内部有分层的硬件内存结构。
2. GPU 内存管理背后有地址空间、页映射、context、分配器等系统机制。

但真实程序还有另一条同样重要的主线：**数据并不是天然就在 GPU 上的，它通常需要从 CPU 一侧进入 GPU，又可能在计算后返回 CPU，或者在 GPU 之间继续流动。**

因此，哪怕 kernel 写得很好，如果数据流设计不好，程序仍然可能很慢。很多工程瓶颈并不是出在计算本身，而是出在：

- H2D 传输太慢
- D2H 传输阻塞
- 异步拷贝没有真正异步
- 小块传输太多
- 传输与计算没有重叠
- 多 GPU 间的数据交换代价高

本章的目标是：

1. 讲清 CPU 与 GPU 之间的数据传输路径。
2. 解释 H2D、D2H、D2D、P2P 的基本区别。
3. 讲清 DMA、PCIe、NVLink、copy engine、stream 与异步传输的关系。
4. 解释为什么 pinned memory 是高效传输的关键条件。
5. 为后面理解流水线、重叠执行、服务系统优化打基础。

---

## 7.2 先建立一条最基本的数据路径

先看最常见的一条路径：

```text
CPU 侧数据
  -> 主机内存
  -> 通过总线传输
  -> GPU 显存
  -> kernel 计算
  -> 结果可能再传回主机内存
```

这条路径看起来简单，但里面实际包含了多层机制：

- 主机内存是 pageable 还是 pinned
- 传输走 PCIe 还是 NVLink
- 是否由 DMA 执行搬运
- 调用是同步还是异步
- 是否和某个 stream 绑定
- 是否能与 kernel 重叠执行
- 是否有额外 staging
- 是否涉及多卡 peer access

所以"把数据传到 GPU"并不是单一动作，而是一套系统流程。

---

## 7.3 三类最常见的数据传输：H2D、D2H、D2D

### 1. H2D：Host to Device

这是从主机到设备的传输，也就是把 CPU 一侧的数据送到 GPU 显存。

常见场景：

- 把训练输入送到 GPU
- 把参数、纹理、缓冲区上传到设备
- 把预处理后的数据输入到 kernel

### 2. D2H：Device to Host

这是从设备到主机的传输，也就是把 GPU 结果取回 CPU。

常见场景：

- 把计算结果给 CPU 后续处理
- 调试时读取中间结果
- 推理服务将输出回传给 CPU 侧业务逻辑
- 图形或计算结果回读

### 3. D2D：Device to Device

设备到设备传输可以有两种常见语境：

- 同一块 GPU 内不同设备缓冲区之间的拷贝
- 不同 GPU 之间的数据传输

前者通常仍发生在设备侧，只是搬运路径和调度方式不同。后者则可能涉及跨 GPU 通路、P2P 能力、NVLink 或 PCIe 拓扑。

### 一个重要认识

这些传输不是"附属小事"，它们经常决定程序整体吞吐。尤其当 kernel 很快时，数据搬运甚至可能成为主瓶颈。

---

## 7.4 PCIe 与 NVLink：数据通路不是都一样

### PCIe 是什么

PCIe 可以粗略理解为 CPU、GPU 和其他设备之间最常见的高速互连通道。很多服务器和工作站中的 GPU 数据传输，默认都要经过 PCIe。

### 它的重要特征

- 通用
- 广泛存在
- 带宽不错，但远低于 GPU 片内和显存内部通路
- 延迟和带宽都比"GPU 本地读显存"差很多

### NVLink 是什么

NVLink 是更偏向高性能 GPU 间互连或 CPU-GPU 高速互联的技术。并不是所有机器都有，但在高端多 GPU 系统里很重要。

### 它和 PCIe 的直觉差别

可以先这样理解：

- PCIe 是广泛存在、通用但相对"外部"的通路
- NVLink 更像专门为高带宽低开销 GPU 协作设计的高速通路

### 为什么这很重要

因为你必须建立一个很强的直觉：

**GPU 本地显存访问、CPU-GPU 传输、GPU-GPU 传输，它们的代价根本不是一个数量级。**

这意味着：

- 不要轻易把大量数据在 CPU 和 GPU 之间来回折返
- 不要把跨设备传输当成"几乎免费"
- 设计系统时要把通路代价纳入整体考虑

---

## 7.5 DMA：为什么 CPU 不需要亲自逐字节搬运

前一章已经提到过 DMA，这里把它放回传输链路里理解。

### 先用直觉看

如果每次 CPU 和 GPU 之间传输数据，都由 CPU 核心自己一字节一字节拷贝，那成本会非常高，也会严重浪费 CPU 时间。

DMA 的意义就在于：

- 让数据搬运尽量由专门机制完成
- 减轻 CPU 的直接参与负担
- 为异步传输和计算重叠创造条件

### 为什么 DMA 不能脱离主机内存性质来理解

DMA 想高效读取主机内存，通常要求：

- 数据所在主机页足够稳定
- 不会在传输中被随意迁移
- 设备能直接可靠访问

这就是 pinned memory 会变得关键的原因。

### 工程含义

很多时候我们说"把数据异步传到 GPU"，背后真正依赖的是：

- 有 DMA
- 主机侧内存可被高效 DMA 访问
- 运行时和 stream 调度允许并行推进

---

## 7.6 同步拷贝与异步拷贝：API 看起来只差一个词，行为差很多

### 同步拷贝的直觉

同步拷贝可以粗略理解为：

- 主机发起拷贝
- 调用在语义上要等到某种完成条件满足后才返回或继续依赖后续结果

这类方式编程直觉更简单，但容易让 CPU 和 GPU 之间形成串行流水。

### 异步拷贝的直觉

异步拷贝则更偏向：

- 主机发起请求后尽快返回
- 实际搬运在设备和运行时体系中继续进行
- 后续是否真正能与别的工作重叠，要看很多条件

### 为什么"异步"不等于"自动加速"

这是最常见误区之一。

调用异步 API 后，想真正获得收益，还要满足：

- 数据源或目标内存适合异步搬运
- 没有立即用同步方式去等待结果
- stream 和执行依赖关系允许重叠
- 硬件上有相应的 copy engine / 资源支持
- 传输规模和组织方式足够合理

所以"异步"只是能力，不是结果。

---

## 7.7 `cudaMemcpy` 与 `cudaMemcpyAsync` 的思维差别

### `cudaMemcpy`

最直觉的理解是：

- 我要把一块数据从 A 拷到 B
- 并且我接受更直接、偏同步的控制语义

它简单，但在高性能路径里容易把数据搬运和计算串起来。

### `cudaMemcpyAsync`

更像是在说：

- 我把这次拷贝请求提交给某个执行流
- 希望它按这个流的顺序和其它工作一起调度
- 并尽量与别的工作形成重叠

### 真正的关键不在函数名，而在依赖图

这两个接口的真正差异，不只是"一个堵塞、一个不堵塞"，而是：

- 任务如何进入执行图
- 与 stream 中其它操作如何排序
- CPU 何时能继续提交工作
- 设备上拷贝和计算是否能并行

所以你理解 `cudaMemcpyAsync` 时，不要只盯着"异步"两个字，而要看它在整条流水线中的位置。

---

## 7.8 为什么 pinned memory 对异步传输几乎是基础条件

这一点值得单独强调。

### 如果主机内存是 pageable

那么系统通常不能直接把它视为最理想的 DMA 数据源。可能发生的情况包括：

- 先临时复制到 pinned staging 区
- 再发起真正的设备传输
- 或者其它额外中间步骤

这会带来：

- 更高延迟
- 更多 CPU 侧参与
- 难以真正形成高效异步链路

### 如果主机内存是 pinned

则更容易做到：

- 设备直接高效读取或写回
- 减少额外 staging
- 更真实地发挥异步传输能力

### 工程上的一个实用判断

如果你调用了 `cudaMemcpyAsync`，但观察不到预期中的拷贝与计算重叠，第一批应该检查的问题里，几乎一定包括：

- 主机内存是不是 pinned
- 数据准备方式是否破坏了异步链路

---

## 7.9 stream：为什么传输顺序和计算顺序不是天然固定的

### 先用直觉理解 stream

stream 可以粗略理解为 GPU 工作提交的一条有序队列。

在同一条 stream 内：

- 任务通常按顺序推进
- 前后操作有天然顺序关系

在不同 stream 之间：

- 如果没有显式依赖，操作可能并发推进
- 是否真的并发，还要看硬件资源和具体条件

### 为什么传输和 stream 绑定后很重要

因为一旦拷贝进入某个 stream，它就不只是"搬运数据"而已，而是成为整条执行图的一部分：

- 它可能先于某个 kernel
- 也可能和另一个 stream 中的 kernel 重叠
- 还可能因为事件依赖而等待

所以 stream 是理解异步拷贝、流水化和重叠执行的核心抽象之一。

---

## 7.10 copy engine：为什么拷贝和计算有时能同时发生

### 直觉理解

GPU 里除了计算单元，通常还有专门处理数据搬运的硬件资源，可以粗略理解为 copy engine。

如果硬件和执行条件允许，那么：

- 某些数据拷贝可以由 copy engine 推进
- 同时计算单元在执行 kernel
- 从而实现"传输和计算重叠"

### 为什么有时做不到

不是每次调用异步拷贝都一定会发生重叠。原因可能包括：

- 没有合适的 stream 并行关系
- 数据源/目标条件不合适
- 硬件资源不足
- 依赖关系让某些操作必须等前面完成
- 程序很快又同步等待了

### 工程含义

"计算与传输重叠"不是一句口号，而是需要满足：

1. 有独立工作可重叠
2. 有合适资源可重叠
3. 有正确调度关系可重叠
4. 没有人为同步把它打断

---

## 7.11 计算与传输重叠：真正高性能流水线的起点

### 一个理想场景

假设你有一批一批的数据要处理。理想状态不是：

1. 先把一整批拷进去
2. 等拷完
3. 再算
4. 等算完
5. 再拷出来
6. 再开始下一批

而是尽可能形成流水线：

- 当前批次在计算
- 下一批次正在拷入
- 上一批次结果正在拷出

### 为什么这样更好

因为它让：

- copy engine 有活干
- 计算单元也有活干
- CPU 端也可以提前准备下一批数据

整个系统不再是"所有阶段串行排队"，而是"多个阶段同时推进"。

### 这就是 GPU 工程为什么总在讲 pipeline

很多看似复杂的流式系统优化，最终目标其实都很朴素：

**不要让昂贵硬件闲着。**

---

## 7.12 小块传输为什么特别亏

很多人知道"大量小 kernel 不好"，但容易忽略"大量小传输也不好"。

### 原因一：固定开销占比太高

每次传输不只是纯数据流量，还包含：

- 调用开销
- 调度开销
- 协议和管理开销
- 同步关系处理

当数据块太小时，这些固定成本占比会很高。

### 原因二：难以充分利用带宽

总线和 DMA 更擅长处理足够大的、连续的数据流。小块零散传输往往不能有效打满链路。

### 原因三：更难形成高效重叠

如果数据被切得过碎，计算与传输之间的节奏也会更混乱，流水线不容易稳定。

### 工程含义

这就是为什么高性能 GPU 程序常常会：

- 批量化搬运
- 合并小请求
- 用 staging buffer 聚合数据
- 以 chunk/tile 为单位组织工作

---

## 7.13 NUMA：为什么在多路 CPU 服务器上，传输还和"离谁近"有关

### 先说直觉

在多路 CPU 的服务器里，不同 CPU socket 往往各自连接一部分内存和 I/O 设备。于是"主机内存"不再是一个对所有设备完全等价的池子。

### 为什么这会影响 GPU 传输

如果 GPU 更靠近某个 CPU/socket，而你的 pinned host buffer 却分配在另一个更远的 NUMA 节点上，那么数据路径可能更绕，代价更高。

### 工程后果

在高性能服务器上，数据预处理线程、主机缓冲区分配位置、GPU 拓扑，最好尽量协同考虑。否则程序表面上只是做 H2D 传输，实际却绕了更长的系统路径。

这说明数据传输优化有时不仅是 CUDA 问题，也是系统拓扑问题。

---

## 7.14 P2P：GPU 和 GPU 能不能直接互相访问

### 什么是 P2P

P2P，peer-to-peer，可以粗略理解为：

- 一块 GPU 能直接访问或更直接地拷贝到另一块 GPU 的内存
- 减少绕回 CPU 主机内存的需要

### 为什么它重要

如果没有更直接的 GPU-GPU 路径，那么跨卡交换数据可能要走：

- GPU A -> Host
- Host -> GPU B

这会增加额外流量和延迟。

### 有了 P2P 的好处

- 路径更短
- 更少 CPU 侧参与
- 对多 GPU 并行更友好

### 但不要想得太理想化

P2P 是否有效、性能如何，还受很多因素影响：

- 硬件拓扑
- GPU 是否支持直接 peer access
- 是否有 NVLink
- PCIe 拓扑是否友好

所以它不是"所有 GPU 默认都完全直连"。

---

## 7.15 数据传输中的几个常见误区

### 误区 1：GPU 很快，所以传输不值一提

错。很多程序里，传输才是主要瓶颈，尤其在：

- 计算量不大
- 输入输出频繁
- 小批量推理
- CPU 预处理复杂
- 多卡同步频繁

的场景中更明显。

### 误区 2：用了异步 API 就一定重叠了

错。异步只是提交方式，不代表最终一定并发。还要看 pinned memory、stream、依赖关系和硬件资源。

### 误区 3：传输成本只和数据大小有关

错。还和以下因素强相关：

- 是否连续
- 是否被切得太碎
- 是否需要 staging
- 是否存在 NUMA 远距访问
- 是否有同步点打断流水线

### 误区 4：CPU 和 GPU 来回拷也没关系，反正都很快

错。CPU-GPU 之间的通路远比 GPU 内部显存路径贵得多。设计时应尽量减少来回折返。

---

## 7.16 一个统一的传输心智模型

你现在可以把 CPU-GPU 传输理解成这样一条链：

1. 数据先存在某种主机内存中
2. 主机内存是否 pinned 会影响 DMA 效率
3. 拷贝请求被提交到运行时与 stream
4. 数据通过 PCIe 或 NVLink 等通路搬运
5. copy engine 与调度机制决定能否和计算重叠
6. 后续 kernel 在设备侧消费这些数据
7. 整个系统性能取决于搬运是否批量、规律、可重叠

一旦记住这条链，很多现象就不再玄学。

---

## 7.17 本章小结

这一章最重要的结论有八条。

1. GPU 程序的性能不仅由计算决定，也经常由数据传输决定。
2. H2D、D2H、D2D、P2P 是不同类型的数据路径，代价和适用场景不一样。
3. PCIe 和 NVLink 等通路的性能远不能和 GPU 本地显存访问混为一谈。
4. DMA 是高效搬运的基础，而 pinned memory 是高效 DMA 访问主机内存的重要前提。
5. `cudaMemcpyAsync` 的价值不在"名字叫异步"，而在它能进入 stream 执行图并参与重叠调度。
6. 计算与传输重叠需要正确的数据条件、执行依赖和硬件资源，不能想当然。
7. 小块传输通常很亏，因为固定开销和带宽浪费都会放大。
8. 在多 GPU 或多路服务器环境中，拓扑、NUMA 和 peer access 也会显著影响传输效率。

下一章我们进入 GPU 内存性能里最关键的一部分：访问模式。你会看到，很多"显存慢"的问题，本质上不是显存本身慢，而是线程访问方式没有顺着 GPU 的硬件偏好来组织。

# 第 8 章 访问模式决定一切：coalescing、局部性与冲突

## 8.1 本章目标

前面几章已经讲了三件基础事实：

1. GPU 有分层的硬件内存结构。
2. 设备内存管理涉及地址空间、映射和传输路径。
3. CPU 和 GPU 之间的数据搬运常常是系统瓶颈。

这一章开始进入 GPU 内存性能里最关键、最决定性的部分之一：**访问模式。**

很多新手会把性能问题归因于"显存太慢"。这个说法只对了一半。更准确地说是：

**显存的理论能力很强，但程序是否能接近这种能力，极大取决于访问模式。**

同样访问 1GB 数据：

- 一种线程组织方式可能非常快
- 另一种方式可能慢很多

算法没有变，数据量没有变，甚至硬件没有变，差别只在"线程怎样访问数据"。

本章的目标是：

1. 解释 coalescing 为什么是 GPU 访存优化的核心概念。
2. 讲清连续访问、跨步访问、随机访问的代价差异。
3. 解释事务、对齐、cache line、局部性这些概念的直觉。
4. 说明 shared memory 中的 bank conflict 和全局内存访问问题是两类不同但相似的冲突。
5. 让你理解：内存管理不只包括"分配多少"，更包括"数据如何布局和被访问"。

---

## 8.2 为什么访问模式会决定性能

### 先建立直觉

GPU 擅长的是：

- 大量线程同时工作
- 规则访问
- 批量利用带宽

因此，当一批线程访问的是一批相邻或规则的数据时，硬件更容易把这些访问合并成较少、较大的内存事务。

而当每个线程都跳到完全不同的位置去访问时：

- 事务数会增加
- 带宽利用变差
- cache 效果变差
- 硬件需要处理更多零散请求

### 一个简化比喻

想象一群工人去仓库取货。

- 如果他们要取的是相邻货架上的连续箱子，仓库系统很容易批量调度
- 如果每个人都跑向不同楼层、不同角落取一小件东西，整体效率会显著下降

GPU 访存也是一样。问题不只是"取多少货"，更是"取货路径是否整齐"。

---

## 8.3 warp 是理解访存模式的起点

### 为什么总要提 warp

GPU 通常不会把每个线程完全孤立地看待，而是以 warp 这样的执行粒度组织线程。一个 warp 中的线程在硬件层面会被一起调度、一起推进。

### 这和访存有什么关系

因为当一个 warp 发起访存时，硬件会观察这一批线程要访问哪些地址，再决定如何形成内存事务。

这意味着：

- 如果 warp 中线程访问的是一段紧邻地址，硬件就更容易高效处理
- 如果 warp 中线程访问散得很开，硬件就需要拆成更多事务

### 一个非常重要的结果

**GPU 访存性能往往不是看"单个线程怎么访问"，而是看"一组线程合起来怎么访问"。**

这是很多 CPU 思维迁移不过来的地方。CPU 常更多关心单线程局部性，GPU 则非常关心一批线程的协同性。

---

## 8.4 coalescing：什么叫合并访问

### 先用一句话讲

coalescing 可以粗略理解为：

**让一组线程的多个访存请求尽可能合并成更少、更高效的底层内存事务。**

### 为什么它这么重要

因为 GPU 的显存带宽很大，但前提是你要以硬件喜欢的方式去发起请求。如果一个 warp 原本可以用少量事务完成读取，却因为地址分散变成很多事务，那么：

- 带宽利用率会下降
- 额外事务管理成本会上升
- cache 和总线压力会增加

### 一个直观例子

假设一个 warp 中的线程要读取一个连续数组中的连续元素：

```cpp
a[tid]
```

如果 `tid` 与线程组织方式对应得好，那么相邻线程读相邻元素，访问通常就更容易被合并。

但如果访问变成：

```cpp
a[tid * stride]
```

而 `stride` 很大，那么线程读到的地址就会分散开，事务数可能显著增加。

### 工程含义

很多 GPU 优化的第一原则就是：

- 让相邻线程尽量访问相邻地址

这听起来简单，但它会深刻影响：

- 数据布局设计
- 线程索引设计
- tile 组织方式
- 张量内存排布
- 稀疏结构的编码方式

---

## 8.5 连续访问、跨步访问、随机访问：代价为什么差这么大

### 1. 连续访问

连续访问是 GPU 最喜欢的一类模式。它通常具有：

- 地址相邻
- 易于合并事务
- 更容易利用 cache line
- 更容易打满带宽

这是访存优化的理想状态。

### 2. 跨步访问

跨步访问指线程按某个固定步长隔着取数据。

例如：

```cpp
a[tid * 2]
a[tid * 4]
a[tid * stride]
```

这种模式有时仍有规律，但规律不一定足够"紧"。如果步长太大，会导致：

- 一个 warp 的访问散落在更大范围
- 事务合并效果下降
- 局部性变差

### 3. 随机访问

随机访问最不友好，因为：

- 线程访问地址不可预测
- 事务难合并
- cache 复用差
- L2 压力大
- 带宽利用差

### 这三者的本质差异

不是"数学上访问了多少元素"的差异，而是：

- 这些元素在地址空间里排布得是否适合一组线程共同取用

这说明 GPU 访存优化的核心不只是算法逻辑，还包括数据排列与线程映射。

---

## 8.6 transaction：为什么不是"访问一次元素就只做一次内存动作"

这是理解 GPU 访存效率的关键概念。

### 直觉理解

当线程发起对显存的访问时，硬件通常不是按"一个线程一个元素一次最小读取"来工作，而是按某种更粗粒度的事务来搬运数据。

也就是说：

- 线程请求的是几个字节
- 硬件真正执行的，可能是某种固定粒度的内存 transaction

### 为什么这很重要

因为如果一个 warp 的访问集中在同一小段区域里，硬件可能用较少事务就满足请求。

但如果地址分散，那么即使总共读取的数据量没变，也可能需要更多事务。

### 工程后果

访存效率常常取决于：

- 需要多少事务
- 每个事务中有多少数据是真正有用的
- 有没有因为分散和不对齐而浪费很多流量

所以 GPU 访存优化常常在做一件事：

**减少事务数量，提高每次事务的有效载荷。**

---

## 8.7 对齐：为什么"差一点点"也可能影响事务组织

### 先说直觉

对齐可以理解为：

- 数据地址是否落在硬件和事务组织比较舒服的边界上

如果访问对齐良好，事务组织更自然；如果访问起点偏斜，就可能让原本一笔能做完的事拆成更多笔。

### 为什么 GPU 上尤其值得关注

因为 GPU 访问常常是成组发生的。一点点偏移，影响的不是一个线程，而可能是一整组线程的事务组织方式。

### 工程含义

这会影响：

- buffer 起始地址设计
- 结构体字段布局
- pitch 分配
- 二维/三维数据存放方式
- tile 的边界处理

对齐不是"底层冷知识"，而是实际能影响事务数量和吞吐的条件。

---

## 8.8 cache line 与局部性：缓存不是万能，但局部性依然重要

### 什么是局部性

局部性可以粗略分成两类：

- 时间局部性：刚访问过的数据很快还会再访问
- 空间局部性：刚访问的位置附近很快也会被访问

### GPU 上为什么仍然讲局部性

虽然 GPU 的缓存设计理念和 CPU 不完全一样，但缓存仍然存在，L1/L2 仍然会对性能产生明显影响。

如果访问具有较好的局部性，那么：

- 更容易命中缓存
- 更少直接打到显存
- 更容易减少总流量

### 但不要过度依赖缓存

GPU 的缓存通常不能替代良好的访问组织。对于很多高吞吐场景，缓存更像帮你吃掉一部分重复访问，而不是帮你兜底一个彻底混乱的访问模式。

所以正确顺序通常是：

1. 先把访问组织成尽量合理
2. 再让缓存锦上添花

而不是反过来。

---

## 8.9 AoS 与 SoA：为什么数据结构设计直接影响访存效率

### AoS：Array of Structures

例如：

```cpp
struct Particle {
    float x, y, z;
    float vx, vy, vz;
};
Particle particles[N];
```

这是很多 CPU 程序里自然的写法，因为它贴近对象语义。

### SoA：Structure of Arrays

例如：

```cpp
struct Particles {
    float x[N], y[N], z[N];
    float vx[N], vy[N], vz[N];
};
```

这是更偏 GPU 友好的写法之一，因为如果一个 warp 想同时读取所有粒子的 `x`，那么访问会更连续。

### 为什么 GPU 经常更喜欢 SoA

因为 GPU 常常是"很多线程同时处理很多对象的同一字段"。在这种模式下，SoA 更容易让相邻线程访问相邻地址。

### 但不是机械地"永远 SoA 更好"

实际情况还要看：

- 你的 kernel 每次主要访问哪些字段
- 字段是否总是一起被用到
- 是否存在复合布局更优的折中方式

真正该记住的是：

**数据结构设计就是 GPU 内存管理和性能设计的一部分。**

---

## 8.10 shared memory 的 bank conflict 与 global memory 的不合并访问，有什么相似之处

### 它们不是同一个问题

- global memory 的问题主要是事务合并、带宽利用、cache 行为
- shared memory 的问题主要是 bank 访问冲突

### 但它们的内核逻辑很像

相似点在于：

- 硬件都希望一组线程以可并行、规则的方式访问存储
- 如果访问组织方式不好，就会把原本可并行的访问部分串行化或拆碎

### 为什么要把它们放一起理解

因为这有助于建立一个更统一的访存思维：

**不管是显存还是 shared memory，真正重要的都是"线程组访问与硬件组织方式是否匹配"。**

这会让你后面面对不同层级优化时，思路更一致。

---

## 8.11 稀疏结构、图结构和不规则数据为什么更难跑快

这一节很重要，因为它解释了"为什么 GPU 并不天然适合一切数据结构"。

### 原因一：地址分散

图、树、链表、稀疏邻接结构通常容易带来：

- 指针追逐
- 间接索引
- 线程访问不同位置
- 局部性差

### 原因二：工作负载不均匀

不同线程可能处理的数据量和访问路径差异很大，导致：

- 分支发散
- 访存模式更混乱
- 缓存效果更差

### 原因三：难以构造漂亮的合并访问

对于规则网格、矩阵、图像，往往更容易构造相邻线程访问相邻地址。

而对不规则结构，这种整齐性本来就难得多。

### 工程含义

所以在 GPU 上处理稀疏或图结构时，经常要：

- 重排数据
- 压缩编码
- 重新编号
- 分块
- 改变遍历顺序

这些操作很多看起来像"算法预处理"，本质上其实是在修复内存访问模式。

---

## 8.12 图像、矩阵、张量为什么往往更适合 GPU

这并不是因为这些任务"数学上高级"，而是因为它们天然更容易满足 GPU 的偏好：

- 数据通常是规则网格
- 邻近元素在地址上往往也邻近
- 容易批处理
- 容易构造 tile
- 更容易使用 shared memory 做复用
- 更容易形成 coalesced access

这就是为什么 GPU 最初擅长图形，后来又特别适合矩阵、深度学习张量计算。

它们在底层上有一个共同点：**结构规则。**

---

## 8.13 一个统一案例：同样是遍历数组，为什么索引设计很关键

假设你要处理一个二维数组。

### 一种较友好的方式

让线程映射顺着内存布局方向展开，例如：

- 相邻线程处理相邻列元素
- 地址连续
- 更容易合并

### 一种不友好的方式

让线程以大跨步方向访问，例如：

- 相邻线程处理跨很远的元素
- 每个线程都要跳大步
- 事务难合并

### 工程结论

很多 kernel 优化并不需要换算法，而只需要：

- 调整线程到数据的映射
- 调整数据存储顺序
- 调整 tile 方向
- 调整张量布局

这些看似只是"索引写法变化"，但常常决定最终性能档次。

---

## 8.14 常见误区

### 误区 1：总读写量一样，性能应该差不多

错。GPU 上总流量相同，不代表事务数量相同，不代表带宽利用相同，也不代表 cache 行为相同。

### 误区 2：缓存会自动修复坏访问模式

错。缓存能缓解一部分问题，但通常不能挽救彻底不规则、不可合并的访问。

### 误区 3：shared memory 一上就会快

错。如果：

- bank conflict 严重
- 数据复用并不高
- 额外同步太多
- occupancy 被压低

那么 shared memory 反而可能得不偿失。

### 误区 4：GPU 慢就是算力不够

错。很多时候不是算力不够，而是显存访问方式没有把算力喂饱。

---

## 8.15 一个统一的访存心智模型

你可以把 GPU 访存优化压缩成下面几个问题：

1. 相邻线程是否访问相邻地址
2. 一组线程的访问是否容易被合并
3. 数据是否对齐良好
4. 是否存在良好局部性
5. 热点数据是否值得搬到 shared memory
6. shared memory 中是否会出现 bank conflict
7. 数据结构是否天然适合并行访问
8. 是否可以通过重排布局降低事务数

只要始终沿着这八个问题思考，大多数 GPU 访存问题都能找到方向。

---

## 8.16 本章小结

这一章最重要的结论有九条。

1. GPU 显存性能高度依赖访问模式，而不是只由硬件带宽数字决定。
2. GPU 常以 warp 等线程组为单位表现访存行为，因此要看"一组线程合起来怎么访问"。
3. coalescing 的核心是让多个线程请求尽量合并成更少、更高效的事务。
4. 连续访问通常最友好，跨步访问次之，随机访问通常最差。
5. 对齐和事务组织方式会直接影响底层流量和带宽利用。
6. 局部性仍然重要，但缓存更多是锦上添花，不能代替良好访问组织。
7. AoS、SoA 等数据布局选择会显著影响 GPU 访存效率。
8. global memory 的不合并访问和 shared memory 的 bank conflict，本质上都体现了"线程组访问不符合硬件组织方式"。
9. 很多 GPU 优化看似在改索引、改布局，本质上是在修复访存模式。

下一章我们会把前面这些底层机制和访问原则，收束到 CUDA 提供的内存管理接口上。也就是：`cudaMalloc`、`cudaMemcpy`、`cudaHostAlloc`、`cudaMallocManaged`、`cudaMallocAsync` 这些 API，到底该如何从"系统角色"的角度去理解，而不是机械记忆。

---

# 第 9 章 CUDA 内存 API 全景：接口、分层与适用场景

## 9.1 本章目标

前面几章已经把 GPU 内存管理的底层逻辑讲清楚了：

- 第 4 章讲硬件层次
- 第 5 章讲地址空间、页表、UVA/UVM
- 第 6 章讲分配路径和 `cudaMalloc`
- 第 7 章讲 CPU-GPU 传输路径
- 第 8 章讲访存模式与 coalescing

到这里，你已经具备了理解 CUDA 内存 API 的底层视角。现在再看 API，就不应该把它们当成一堆离散函数名，而应该把它们看成：

- 对哪一类内存对象进行操作
- 解决哪一类系统问题
- 带来什么成本和语义
- 更适合什么场景

本章的目标是：

1. 从"角色"而不是"列表"来整理 CUDA 内存相关 API。
2. 区分设备内存、主机固定页内存、统一内存、异步分配和内存池接口。
3. 说明传统接口与新接口分别在解决什么问题。
4. 帮你形成一套 API 选择逻辑，而不是死记函数。

---

## 9.2 先按职责给 API 分层

先把常见 CUDA 内存 API 分成五大类：

### 第一类：设备内存分配与释放

典型接口：

- `cudaMalloc`
- `cudaFree`

它们解决的问题是：

- 在设备侧分配和回收显存对象

### 第二类：设备与主机之间的数据搬运

典型接口：

- `cudaMemcpy`
- `cudaMemcpyAsync`
- `cudaMemcpy2D`
- `cudaMemcpy3D`

它们解决的问题是：

- 在不同地址空间或不同设备对象之间搬运数据

### 第三类：主机侧固定页内存

典型接口：

- `cudaHostAlloc`
- `cudaHostRegister`
- `cudaFreeHost`

它们解决的问题是：

- 为高效 DMA 和异步传输准备 pinned host memory

### 第四类：统一内存

典型接口：

- `cudaMallocManaged`
- `cudaMemPrefetchAsync`
- `cudaMemAdvise`

它们解决的问题是：

- 降低显式管理主机/设备分界的复杂度
- 允许按需迁移和统一地址访问

### 第五类：异步分配与内存池

典型接口：

- `cudaMallocAsync`
- `cudaFreeAsync`
- memory pool 相关接口

它们解决的问题是：

- 改善传统分配释放路径中的同步和复用问题
- 让分配更适合高吞吐异步工作流

只要先记住这个分类，后面的 API 就不容易乱。

---

## 9.3 Runtime API 与 Driver API：为什么通常先从 Runtime 理解

### Runtime API

这是大多数开发者最直接使用的一层。例如：

- `cudaMalloc`
- `cudaMemcpy`
- `cudaHostAlloc`

它更易用，抽象更直接，适合大多数应用和框架开发。

### Driver API

Driver API 更底层，通常提供更细粒度的控制。对于建立内存管理心智模型，它很重要；但对大多数学习路径来说，先掌握 Runtime API 的语义和系统后果更关键。

### 工程上的理解方式

你不必一开始就把 Driver API 全记住。更重要的是知道：

- Runtime API 背后并不是"魔法"
- 它依然需要借助更底层驱动能力来完成资源管理
- 这也是为什么某些 Runtime 调用会体现出明显系统成本

---

## 9.4 `cudaMalloc` / `cudaFree`：设备显存的基础接口

### 它们的角色

这组接口是最基础的设备显存分配与释放入口。

- `cudaMalloc`：申请设备内存
- `cudaFree`：释放设备内存

### 适用场景

它们适合：

- 结构简单的程序
- 初始化阶段分配大块缓冲区
- 对底层显存对象有显式控制需求的场景

### 不适合的场景

它们不太适合：

- 热路径中频繁小块分配
- 高吞吐服务中的频繁动态生命周期对象
- 需要精细异步流水化的工作负载

### 为什么

因为我们前面已经讲过：

- 分配不便宜
- 释放可能涉及隐式同步
- 频繁调用会放大开销和碎片风险

### 正确心态

把 `cudaMalloc` 理解为：

- 低层基础能力
- 不是最适合所有热路径的默认选择

---

## 9.5 `cudaMallocPitch` / `cudaMalloc3D`：为什么二维和三维数据有专门接口

### 为什么会有它们

二维、三维数据在实际内存中不一定总是按最理想的简单线性布局使用。尤其在图像、矩阵、体数据等场景中，行对齐和步长组织会影响访问效率。

### `pitch` 的直觉

可以把 pitch 理解为：

- 每一行在内存中的实际步长
- 它可能大于"这一行数据本身的字节数"
- 目的是帮助满足对齐和访问组织要求

### 为什么这有意义

如果二维数据每行直接紧贴存放，不一定最适合底层访问和对齐。引入 pitch 后，运行时可以为你提供更符合某些硬件偏好的布局。

### 工程含义

这类接口提醒你一个重要事实：

**内存分配不是只有"多大"这个维度，还有"怎么排布"这个维度。**

---

## 9.6 `cudaMemcpy` 系列：搬运 API 不是附属，而是主角之一

### 这组 API 在做什么

它们负责在不同内存空间或设备对象之间搬运数据。

典型包括：

- H2D
- D2H
- D2D
- 二维/三维数据拷贝

### 为什么它们是核心接口

因为很多程序不是算得慢，而是搬得慢。尤其在以下场景：

- 小 batch
- 数据预处理在 CPU
- 频繁跨设备交换
- 图像/视频流式处理
- 多卡训练和推理

### 如何理解这组 API

理解重点不在"函数长什么样"，而在：

- 数据源和目标分别在哪里
- 是否需要 pinned memory
- 是否希望进入 stream 异步调度
- 是否有二维/三维步长需求
- 搬运是否会成为主瓶颈

---

## 9.7 `cudaHostAlloc` / `cudaHostRegister`：为什么主机内存也有 CUDA 接口

### `cudaHostAlloc`

它的直觉作用是：

- 直接分配一块 pinned host memory

适合你从一开始就知道：

- 这块主机内存要高频与 GPU 交互
- 希望支持高效 DMA 和异步传输

### `cudaHostRegister`

它更像是：

- 把一块已有的普通主机内存注册为固定页内存
- 让它后续更适合设备访问或高效传输

### 为什么这一组接口重要

因为它们不是"主机端的杂项功能"，而是高性能 CPU-GPU 数据路径的一部分。

### 使用时要记住的平衡

它们很有用，但 pinned memory 是高性能资源，不应无节制滥用。否则可能：

- 增加主机内存管理压力
- 影响系统弹性
- 让整体系统行为变差

---

## 9.8 `cudaMallocManaged`：统一内存入口

### 它的角色

`cudaMallocManaged` 提供的是 managed memory，也就是统一内存编程模型中的一个主要入口。

### 它适合什么

它适合：

- 原型开发
- 减少显式 copy 代码
- 更容易管理某些复杂数据结构
- 某些 CPU/GPU 共享访问模型

### 它不适合被想当然地当成"默认最优"

因为它可能引入：

- page migration
- page fault
- 不可预测延迟
- CPU/GPU 交替访问时的额外成本

### 正确理解方式

把它当成：

- 更方便的编程工具
- 但不是天然最高性能工具

---

## 9.9 `cudaMemPrefetchAsync` 与 `cudaMemAdvise`：统一内存不是只能"听天由命"

### 为什么需要这些接口

如果统一内存完全靠运行时被动按需处理，那么性能可能不稳定。于是 CUDA 提供了额外机制，让程序员给出更多"意图信息"。

### `cudaMemPrefetchAsync`

可以粗略理解为：

- 提前把某块 managed memory 迁移到更可能马上要用它的处理器侧

这样做的目的，是减少真正访问时才触发迁移和 page fault。

### `cudaMemAdvise`

它更像是在告诉运行时：

- 这块内存更可能被谁访问
- 是否偏向某类访问模式
- 是否值得按某种策略处理

### 为什么它们重要

它们说明一个事实：

**统一内存并不是"全自动、你完全别管"的模型。**

高性能使用时，仍然需要你参与引导数据驻留和迁移策略。

---

## 9.10 `cudaMallocAsync` / `cudaFreeAsync`：为什么传统接口之外需要新路线

### 这组接口在回应什么问题

前面第 6 章已经讲过传统路径的几个痛点：

- 分配释放成本不低
- `cudaFree` 容易和同步问题挂钩
- 热路径使用不友好
- 难以很好适应流式异步工作负载

### 这组接口的思路

`cudaMallocAsync` / `cudaFreeAsync` 的方向是：

- 更好地和 stream 顺序语义结合
- 更适合基于池的复用
- 减少某些全局同步压力
- 让分配释放更接近异步执行图的一部分

### 为什么这很重要

这说明现代 CUDA 的内存管理思路，已经不再只是：

- 申请一块
- 用完释放

而是更接近：

- 在异步工作流中复用和管理内存生命周期

这对训练框架、推理引擎、长期服务系统尤其重要。

---

## 9.11 memory pool：为什么"分配 API"最终会走向"池化 API"

### 先说本质

memory pool 的核心思想很简单：

- 不要每次都向系统申请和归还
- 先拿一批资源
- 在内部反复切分和复用

### 为什么 CUDA 层也会强调 pool

因为 GPU 分配器遇到的所有老问题：

- 固定分配成本
- 释放同步
- 高频动态对象
- 碎片管理

都指向同一个工程答案：**复用比反复系统分配更划算。**

### 这和框架缓存分配器的关系

深度学习框架、推理引擎里常见的 caching allocator，本质思路和 memory pool 是一致的。区别只是：

- 谁在管理池
- 粒度多细
- 生命周期如何与 stream 或图执行绑定

所以 pool 不是"高级特性"，而是现代 GPU 内存管理的自然演进方向。

---

## 9.12 该怎么选 API：一套实用决策思路

下面给你一个教学版的选择逻辑。

### 场景 1：我只需要几个长期存在的大 buffer

优先考虑：

- `cudaMalloc`
- 初始化阶段一次性分配

因为这种场景动态分配压力不大，传统接口足够直接。

### 场景 2：我需要高效 H2D / D2H 传输

优先关注：

- `cudaHostAlloc`
- `cudaHostRegister`
- `cudaMemcpyAsync`

因为真正的重点不是"拷贝函数名字"，而是主机缓冲区是否适合高效 DMA 和异步链路。

### 场景 3：我想减少手动 copy 代码，先把程序写通

可以考虑：

- `cudaMallocManaged`

但要明确它是以运行时迁移复杂性换开发便利性。

### 场景 4：我是高吞吐异步系统，频繁分配释放对象

优先考虑：

- `cudaMallocAsync`
- `cudaFreeAsync`
- memory pool
- 上层缓存分配器

因为这类系统最怕传统分配释放路径的同步和抖动。

### 场景 5：我是二维/三维规则数据场景

考虑：

- `cudaMallocPitch`
- `cudaMalloc3D`
- 对应的二维/三维 memcpy 接口

因为布局和 pitch 本身就是性能问题的一部分。

---

## 9.13 常见误区

### 误区 1：只要会 `cudaMalloc` 和 `cudaMemcpy` 就够了

错。那只够写最基本程序，不够构建高性能系统。

### 误区 2：managed memory 更高级，所以应该默认用

错。它更方便，但不一定更快，也不一定更稳定。

### 误区 3：异步分配只是"更快一点的 malloc"

错。它真正改变的是分配与执行图、stream 语义和复用策略之间的关系。

### 误区 4：主机内存接口不重要，重点都在显存

错。高性能 CPU-GPU 数据路径里，host memory 的性质经常决定传输上限。

---

## 9.14 一个统一的 API 心智模型

你现在可以把 CUDA 内存相关 API 压缩成下面这套图景：

1. 设备分配接口：决定数据在 GPU 上有没有地方住
2. 传输接口：决定数据怎么进出 GPU
3. pinned host memory 接口：决定主机侧是否适合高效搬运
4. unified memory 接口：决定是否用更统一但可能更不可预测的迁移模型
5. async allocator / pool 接口：决定高吞吐系统里内存生命周期能否更稳定、更少同步

也就是说，CUDA 内存 API 不是一个平面列表，而是一套覆盖：

- 放在哪里
- 怎么搬
- 谁来管
- 何时迁
- 怎样复用

的系统接口。

## 9.15 本章小结

这一章最重要的结论有九条。

1. CUDA 内存 API 应按职责分类理解，而不是按函数名死记。
2. `cudaMalloc` / `cudaFree` 是基础接口，但不适合所有高动态、高吞吐场景。
3. `cudaMemcpy` 系列是核心性能路径的一部分，不是附属工具。
4. `cudaHostAlloc` / `cudaHostRegister` 之所以重要，是因为 pinned host memory 直接影响 DMA 和异步传输效率。
5. `cudaMallocManaged` 提供统一内存便利，但不等于性能最优。
6. `cudaMemPrefetchAsync` 和 `cudaMemAdvise` 说明统一内存高性能使用仍需要显式引导。
7. `cudaMallocAsync` / `cudaFreeAsync` 的出现，是为了解决传统分配释放在异步工作流中的痛点。
8. memory pool 是现代 GPU 内存管理的自然方向，因为复用比频繁系统分配更稳定。
9. 选择 API 时，应该先问场景需求是什么，再决定是优先简单性、显式控制还是吞吐稳定性。

# 第 10 章 内存池与缓存分配器：为什么框架总像是"不释放显存"

## 10.1 本章目标

前面几章已经把 GPU 内存管理的底层主线讲清楚了：

- 第 4 章讲硬件层次
- 第 5 章讲地址空间、页表、UVA/UVM
- 第 6 章讲分配路径和 `cudaMalloc`
- 第 7 章讲 CPU-GPU 传输路径
- 第 8 章讲访存模式与 coalescing
- 第 9 章讲 CUDA API 体系

这一章开始进入工程世界里最容易让人困惑、也最常见的一层：**内存池和缓存分配器**。

很多人在使用深度学习框架、推理引擎或长期运行的 GPU 服务时，都会遇到这些现象：

- 张量释放了，但 `nvidia-smi` 里的显存没降
- 程序一开始分配很快，运行久了开始抖
- 明明显存还有空闲，却报 OOM
- 重启进程后问题消失，跑久了又出现
- 框架说"已分配"和"已保留"不是一回事

这些现象如果只从 `cudaMalloc` / `cudaFree` 的角度理解，会觉得很反直觉。但一旦引入内存池和缓存分配器，这些现象就会变得非常自然。

本章的目标是：

1. 解释为什么系统级 GPU 分配器通常不适合直接承载高频动态对象。
2. 讲清内存池、缓存分配器、大块切分与复用的基本思路。
3. 解释 internal fragmentation 与 external fragmentation 的区别。
4. 说明为什么"有空闲显存"仍可能分配失败。
5. 帮你区分"缓存保留显存"和"真正的内存泄漏"。

---

## 10.2 为什么系统级分配器不适合高频动态工作负载

### 先回忆第 6 章的结论

我们已经知道：

- `cudaMalloc` 不便宜
- `cudaFree` 也不便宜
- `cudaFree` 还可能引入隐式同步
- 频繁小块分配很容易带来碎片和抖动

如果一个程序只是启动时分配几块大 buffer，然后一直复用，那问题不大。但现实中的很多系统并不是这样。

### 哪些系统会频繁产生动态对象

典型包括：

- 深度学习框架中的临时张量
- 推理系统中的请求级 buffer
- 图形引擎中的资源流入流出
- 通用 CUDA 程序中的中间工作区
- 多 stream 环境下按阶段产生和回收的临时块

这些对象往往具有这些特征：

- 数量多
- 生命周期短
- 大小分布复杂
- 分配与释放频繁
- 不同流和不同阶段交错

### 如果直接每次走系统级分配会怎样

会有几个问题叠加出现：

1. 每次分配都要付出固定系统成本
2. 每次释放都可能触发额外协调
3. 显存空间会被切得越来越碎
4. 程序的延迟和吞吐变得不稳定

### 工程结论

这就是为什么大多数成熟 GPU 软件栈最终都会走向同一个答案：

**不要让高频对象生命周期直接暴露给底层系统分配器，而是要在上面加一层复用机制。**

这层复用机制，就是内存池或缓存分配器。

---

## 10.3 内存池的核心思想：先拿一大块，再内部反复复用

### 最朴素的直觉

内存池的思想其实很简单：

1. 先从底层系统申请较大的内存块
2. 在池内部把这些块切成较小部分
3. 把小部分分配给具体对象
4. 对象释放时不立刻还给系统，而是还回池中
5. 以后新的对象再复用这些块

### 为什么这能显著改善性能

因为它把"频繁系统级操作"变成了"较少系统级操作 + 大量池内操作"。

池内操作通常可以做到：

- 更轻量
- 更少同步
- 更容易根据上层模式定制
- 更容易跟踪生命周期
- 更容易做局部优化

### 这就像什么

可以把它想成仓储系统：

- 系统级分配器像是每次缺个螺丝都去供应商重新下单
- 内存池像是先批量进货，再在仓库内部按需发放

对于高频需求，后者显然更合理。

---

## 10.4 缓存分配器：为什么"释放"后显存还留在进程里

### 什么叫缓存分配器

缓存分配器是内存池思想在很多框架中的常见实现形式。它通常会做这件事：

- 当上层对象需要内存时，优先从已保留的块中分配
- 当上层对象释放时，不立刻归还到底层 CUDA，而是标记为可复用
- 在后续请求中复用这些块
- 只在必要时才向系统申请更多或归还部分保留块

### 为什么这样做

核心原因有两个：

1. 避免反复走昂贵的底层分配/释放路径
2. 尽量避免 `cudaFree` 相关同步影响

### 这就是为什么你会看到

- 框架对象已经没了
- 但进程占着的显存还在
- 下次再创建张量时分配非常快

这通常不是 bug，而是缓存分配器在工作。

### 一个重要区分

- "对象已经释放"不等于"显存已经还给 CUDA 驱动"
- "显存没有还给驱动"也不等于"这块显存不能再给框架内部复用"

这是理解框架显存统计的第一步。

---

## 10.5 内存池通常怎么工作：大块、切分、回收、合并

虽然不同框架和系统实现细节不同，但可以用一个通用模型来理解。

### 第一步：向系统申请大块

内存池往往不是按对象粒度向系统申请，而是按较大粒度拿块。原因是：

- 减少系统调用次数
- 提高后续切分灵活性
- 更适合复用

### 第二步：在池内切分成可用块

当上层需要一块内存时，池会查找某个合适的大块或空闲块，然后切出满足需求的一部分。

### 第三步：对象释放时回收到池内

对象生命周期结束后，这一块被标记为空闲，但通常还留在池中。

### 第四步：尝试合并相邻空闲块

为了减轻碎片，池往往会在条件允许时尝试合并相邻空闲块，恢复更大的可用区域。

### 第五步：必要时再向系统扩张或回收

如果池内找不到合适块，就继续向底层申请。如果保留过多、策略允许，也可能归还部分空闲块。

### 这里最重要的一点

内存池不是"只缓存不用管"，而是一个实时做平衡的系统：

- 复用效率
- 分配速度
- 内存保留量
- 碎片程度
- 同步代价

这些目标经常互相冲突。

---

## 10.6 internal fragmentation 与 external fragmentation：碎片不是一种，而是两种

这是理解"为什么有空闲显存却 OOM"的关键。

### 1. internal fragmentation：内部碎片

内部碎片指的是：

- 你分给某个对象的块，比它真正需要的大
- 多出来的那部分在这次分配中浪费了

这通常来源于：

- 分配粒度对齐
- 固定桶大小
- 向上取整策略

例如你只需要 33KB，但系统或池按 64KB 粒度给你，这里面多出来的一部分就是内部碎片。

### 2. external fragmentation：外部碎片

外部碎片指的是：

- 空闲总量看起来不少
- 但被切成很多零碎小洞
- 无法满足某次较大的连续分配请求

例如池里有很多空闲块，但每块都不够大，或者虽然总和够大，却不是一整块连续可用区。

### 为什么外部碎片更"可怕"

因为它会制造最令人困惑的现象：

- 总空闲显存看起来还行
- 但某个新请求就是失败

这也是很多"显存还没满却 OOM"的根源。

---

## 10.7 为什么"空闲显存很多"仍然可能 OOM

### 第一种情况：外部碎片

这是最典型的情况。池中还有空闲，但没有一块足够大的连续空间给新请求。

### 第二种情况：上层可见空闲与底层可分配空闲不是一回事

例如：

- 某些块在框架内部被保留但暂未用于活跃对象
- 某些块还受 stream 生命周期或异步任务影响，暂时不能安全复用
- 某些统计口径把"可复用但未归还"的内存和"真正空闲"混在一起看

### 第三种情况：峰值内存瞬间过高

程序不是静态地"当前用了多少"，而是会在某一阶段突然需要更多中间缓冲区。如果这个瞬间峰值过高，即使平均占用不高，也可能失败。

### 第四种情况：地址或分配策略层面的限制

有时问题不只是总量，还有：

- 对齐要求
- 分配粒度
- 池策略
- 大小分类桶
- 特定 stream 或 arena 的可用性

### 一个非常重要的工程认知

**能不能分配成功，不只取决于"空闲总字节数"，还取决于"空闲是如何分布、归属、对齐和可回收的"。**

---

## 10.8 为什么框架通常"保留显存不放"

### 先说误解

很多用户第一次看到深度学习框架的行为，会觉得：

- 我都删掉张量了，为什么显存不降
- 是不是框架泄漏了
- 为什么不像普通程序一样立刻还内存

### 更准确的理解

框架通常是在权衡：

- 现在还给系统，之后再申请回来会很贵
- 现在先保留，后面大概率还能复用

如果系统观察到你的程序处在"不断创建临时张量"的工作模式里，那么缓存住显存通常是合理的。

### 这样做的好处

- 下次分配更快
- 减少底层调用
- 降低同步风险
- 稳定整体吞吐

### 带来的代价

- 外部看起来显存一直占着
- 更难直接判断"活跃使用量"
- 需要框架自己承担更多碎片管理责任

### 所以要学会区分两类数字

在框架里经常要区分：

- 活跃使用量
- 已保留量
- 底层系统层面已占用量

这几个数字不是一回事。

---

## 10.9 缓存保留不是内存泄漏：两者到底怎么区分

### 真正的内存泄漏是什么

真正泄漏更接近：

- 某块内存不再需要
- 但系统中已经没有路径能正确回收它
- 它永久失去复用与释放机会

### 缓存保留是什么

缓存保留则是：

- 这块内存当前没有被活跃对象使用
- 但分配器明确知道它还在
- 可以在后续请求中再利用
- 必要时在某些策略下还能归还系统

### 为什么两者容易混淆

因为从外部看，二者共同点都是：

- 显存占用没有立刻下降

但内部语义完全不同。

### 区分方法的思路

通常要结合：

- 框架内存统计
- 活跃对象数量和生命周期
- 分配/释放趋势
- 长时间运行占用是否单调增长且不可回收
- 是否还能被后续同类请求复用

### 一个实用判断

如果显存高占用是稳定平台，并且后续对象能快速复用这些空间，这通常更像缓存策略。
如果显存占用持续只增不减，且与活跃工作量不匹配，才更像泄漏或保留失控。

---

## 10.10 为什么长期运行系统更容易暴露碎片问题

短命程序和长期服务的显存行为很不一样。

### 短命程序

例如一次训练脚本或批处理任务：

- 运行模式相对稳定
- 生命周期相对集中
- 即使有些碎片，进程结束后一切重置

### 长期运行系统

例如：

- 在线推理服务
- 图形渲染引擎
- 多租户 GPU 服务
- 长期驻留的数据处理系统

这些系统更容易遇到：

- 请求大小波动
- 生命周期交错
- 不同批次内存需求差异大
- 长时间累积的空洞
- 偶发大请求触发分配失败

### 工程后果

长期系统往往更需要：

- 更稳定的分配粒度
- 更强的池策略
- 预分配和分桶
- 定期观察碎片趋势
- 避免任意尺寸的动态分配风暴

---

## 10.11 `cudaMallocAsync` 为什么能改善一部分问题

这里不展开 API 细节，只讲它在分配器层面的意义。

### 它想解决什么

传统路径的问题在于：

- 分配释放和全局状态耦合更重
- 释放容易与全局同步扯上关系
- 高频异步流水线下不够自然

### 它改进的方向

`cudaMallocAsync` 更强调：

- 与 stream 顺序语义结合
- 更容易依赖 pool 做复用
- 减少一些粗粒度全局同步压力

### 这对工程意味着什么

它不是"更快一点的 malloc"，而是更适合：

- 异步图
- 多 stream
- 动态临时对象
- 高吞吐长时间运行系统

### 但不要误解

它也不是魔法。它可以改善某些分配释放路径问题，但不能自动消除：

- 糟糕的生命周期设计
- 失控的大小分布
- 不合理的内存峰值
- 本质上的容量不足

---

## 10.12 分配器设计其实是在做多目标平衡

一个好的缓存分配器，不是在追求某个单一指标极致，而是在多个目标间取平衡。

### 目标 1：分配快

用户一请求，就尽量快速给到块。

### 目标 2：释放不打断流水线

减少全局同步和不必要的阻塞。

### 目标 3：碎片尽量少

避免空闲总量不少但分配大块失败。

### 目标 4：保留量别太夸张

不能把显存都囤住，影响别的对象或别的系统部分。

### 目标 5：适配上层工作负载

训练、推理、图形、流式服务，它们的分配模式很不一样，分配器最好能适应这些差异。

### 所以不存在"完美分配器"

更常见的是：

- 某种策略更适合某类负载
- 另一种策略更适合另一类负载

这也是为什么不同框架和系统的显存行为会看起来不一样。

---

## 10.13 一个统一的心智模型：为什么"显存没降"不一定是坏事

你现在可以这样理解很多框架中的显存现象：

1. 底层系统分配昂贵，所以框架尽量减少直接调用
2. 框架更喜欢预留和复用，而不是频繁归还
3. 对象释放后，显存可能仍留在框架池中
4. 这些块在框架视角下是"空闲可复用"的
5. 但在系统视角下仍显示为该进程占用
6. 因此外部工具看到的显存数字，未必等于"当前活跃对象真实占用"

这是理解 GPU 框架显存管理的基本出发点。

---

## 10.14 本章小结

这一章最重要的结论有十条。

1. 系统级 GPU 分配器通常不适合直接承载高频动态对象生命周期。
2. 内存池的核心思想是：少做系统级分配，多做池内复用。
3. 缓存分配器会把释放后的块保留在进程内部，以备后续快速复用。
4. 对象释放不等于显存立即归还系统。
5. internal fragmentation 是块内浪费，external fragmentation 是空闲块分布破碎。
6. "空闲总量足够"不代表"一定能成功分配"。
7. 长期运行系统比短命程序更容易积累碎片和抖动。
8. 缓存保留显存不等于内存泄漏，二者需要通过语义和趋势区分。
9. `cudaMallocAsync` 的意义在于让分配更适配异步执行和 pool 复用，而不只是"更快一点"。
10. 显存分配器本质上是在分配速度、碎片、同步代价和保留量之间做平衡。

---

# 第 11 章 统一内存 UVM：方便背后的隐性代价

## 11.1 本章目标

前面我们已经在第 5 章提到过 UVA 和 UVM，但当时是为了建立整体地图。本章要把统一内存单独拿出来讲透。

这是因为统一内存很容易给初学者一种非常强的诱惑：

- 不想手动管 `cudaMemcpy`
- 不想显式区分主机和设备缓冲区
- 希望写起来更像"普通内存"
- 希望 CPU 和 GPU 共用一套数据对象

从编程便利性的角度看，这个方向非常合理。统一内存也确实能带来巨大的开发体验提升。

但它的问题在于：

**它并没有消灭内存管理复杂性，只是把复杂性从"你写的显式代码"转移到了"运行时迁移与页管理行为"上。**

本章的目标是：

1. 解释统一内存想解决什么问题。
2. 讲清 page migration 和 page fault 在统一内存中的意义。
3. 解释为什么统一视图不等于统一性能。
4. 说明什么时候 UVM 特别有帮助，什么时候它可能明显变慢。
5. 帮你建立对 managed memory 的正确预期。

---

## 11.2 统一内存到底想解决什么问题

### 最直接的问题

在传统 CUDA 编程中，你常常需要显式做这些事：

1. 在设备上分配内存
2. 在主机上准备数据
3. 手动把数据拷到 GPU
4. kernel 计算
5. 再手动把结果拷回 CPU

这套流程并不难，但当程序变复杂后会有几个问题：

- 代码量大
- 数据生命周期管理繁琐
- 更容易写错
- 复杂数据结构处理起来很累
- CPU/GPU 共享访问的逻辑割裂明显

### UVM 的目标

统一内存的目标就是降低这种割裂感，让程序员更像在操作"同一份逻辑数据"，而不是时刻手动维护两份位置不同的副本。

它想达到的效果是：

- 统一地址视图
- 减少显式 copy 代码
- 允许 CPU 和 GPU 以更自然方式访问同一逻辑对象
- 把部分迁移和驻留决策交给运行时

### 为什么这很有吸引力

因为它对以下场景尤其诱人：

- 原型开发
- 代码快速验证
- 数据结构较复杂、不易平铺
- CPU/GPU 交替使用数据
- 想先把程序跑通，再逐步优化

---

## 11.3 managed memory 的编程直觉：像一块"共享可见内存"

### 表面上的使用感受

用 `cudaMallocManaged` 分配后，你会得到一个看起来更统一的指针。CPU 代码可以操作它，GPU kernel 也可以使用它。

这会带来一种很自然的直觉：

- "这块内存好像大家都能直接用"
- "我不用专门准备 host buffer 和 device buffer 两份了"

### 这种直觉对不对

它在逻辑层面是对的，但在性能层面是不完整的。

因为"都能访问"并不意味着：

- 数据当前就在最适合访问者的一侧
- 下一次访问不需要迁移
- CPU 和 GPU 之间不会为了这块内存来回移动页面
- 访问成本可忽略

### 所以要建立的正确理解是

统一内存更像是：

- 一个统一的逻辑对象
- 加上一套运行时负责页面驻留、迁移和映射的机制

而不是"物理上真的所有人共享同一块没有代价的内存"。

---

## 11.4 page migration：统一内存真正的代价中心

### 直觉理解

page migration 指的是：

- 某块内存页当前主要驻留在一侧
- 另一侧开始访问它
- 于是系统把这页迁移到更适合当前访问的一侧

例如：

- 数据原本主要在 CPU 侧
- GPU 开始大量访问
- 那么相关页面可能迁移到 GPU 更方便访问的驻留位置

### 为什么这是统一内存的核心成本

因为统一内存并没有让 CPU 和 GPU 之间的物理差异消失。它只是让"何时搬、搬哪些页、谁来决定"更多由运行时接管。

所以本来你手动写的：

- 显式 H2D copy
- 显式 D2H copy

现在变成了：

- 运行时按页、按需、按访问行为去迁移

### 这就带来两个后果

1. 编程更省心
2. 运行时行为更复杂、更难预测

---

## 11.5 GPU page fault：为什么 GPU 访问也会"缺页"

### 直觉理解

在统一内存环境里，如果 GPU 访问某个页面时，发现该页尚未以当前需要的方式准备好，就可能触发类似缺页的处理过程。

这意味着：

- 访问不会立刻按理想路径完成
- 运行时需要先处理映射或迁移
- 之后这次访问才能继续

### 为什么这会影响性能

因为这种成本经常具有两个特点：

- 它不是你代码里显式写出来的
- 它可能以"突然一下慢了"的方式出现

### 这和显式 copy 的区别

显式 copy 的成本是：

- 你提前知道它在哪里发生
- 它的时间边界更清晰
- 更容易设计流水线

page fault 驱动的按需迁移则是：

- 延迟更隐式
- 更依赖访问行为
- 更可能带来抖动

所以统一内存把"显式管理复杂性"换成了"隐式运行时复杂性"。

---

## 11.6 为什么统一内存有时很方便，有时却很慢

### 方便的原因

统一内存方便，是因为它减少了程序员要显式处理的事情：

- 少写 buffer 管理代码
- 少写拷贝代码
- 更容易组织复杂对象
- 更容易让程序先"跑起来"

### 变慢的原因

它变慢通常不是因为"这个技术不好"，而是因为：

- 访问模式让迁移频繁发生
- CPU 和 GPU 交替抢同一批页面
- 工作集过大
- 页面调度和迁移时机不稳定
- 原本可以批量搬运的事情，现在变成了大量按需迁移

### 一个关键对比

显式拷贝更像：

- 你自己安排一次整批搬家

统一内存按需迁移更像：

- 谁临时要用就搬一点
- 结果可能搬很多次，节奏也未必理想

如果访问模式规则、阶段明确，显式方式往往更可控。
如果开发复杂性是主要问题，统一内存则更有吸引力。

---

## 11.7 `cudaMemPrefetchAsync`：提前迁移，减少"边跑边补票"

### 先说本质

统一内存最大的问题之一，是数据迁移可能发生在真正访问时。这样会把运行时成本插进关键路径。

`cudaMemPrefetchAsync` 的作用，就是尽量把这种代价前移：

- 在你真正要用之前
- 先把数据迁移到更可能需要它的一侧

### 为什么这很重要

因为它在一定程度上把"隐式、被动、访问触发的迁移"重新拉回到"更主动、更可控"的模式。

### 它的工程意义

它说明：

- 统一内存并不意味着你完全不需要思考数据位置
- 真正想让 UVM 用得好，仍然要对接下来的访问阶段有预判
- 你越能预测数据下一步归谁用，越能减少运行时惊喜

### 一个直观比喻

prefetch 不是让搬家消失，而是让你提前安排搬家，而不是等客人站在门口了才临时搬。

---

## 11.8 `cudaMemAdvise`：给运行时提示，不代表完全控制

### 它在做什么

`cudaMemAdvise` 更像是向运行时表达"使用意图"：

- 哪一侧更可能访问这块内存
- 这块内存的访问模式是否有某些倾向
- 是否存在偏好的驻留位置

### 为什么这类提示存在

因为运行时虽然能观察行为，但未必总能比程序员更早、更准确知道未来阶段会发生什么。

### 它能解决什么

它有助于：

- 减少一些错误迁移决策
- 提高运行时对工作负载的适配
- 改善某些访问模式下的页面驻留行为

### 它不能解决什么

它不是绝对强制控制，也不意味着：

- 你给了 hint 后就一定最优
- 可以完全不关心访问模式
- 统一内存从此没有抖动

它只是缩小"运行时不了解你意图"的信息差。

---

## 11.9 什么时候 UVM 特别有帮助

统一内存在下面这些场景通常很有价值。

### 1. 原型开发和教学

你想先把逻辑跑通，而不想一开始就被大量 buffer 和 copy 管理打断思路。

### 2. 数据结构复杂

例如存在较复杂的索引关系、层次结构、图结构、对象式组织。虽然 GPU 对这类结构未必高效，但 UVM 至少可以降低管理难度。

### 3. CPU/GPU 协作开发初期

你还在快速试验哪些阶段该放 CPU、哪些阶段该放 GPU，统一内存能降低反复重构数据通道的成本。

### 4. 对极致性能要求没那么高

有些任务更重视开发效率、可维护性和功能迭代，而不是绝对吞吐极致。

### 5. 工作集和访问行为比较温和

如果页面不会在 CPU/GPU 间剧烈来回，且整体访问阶段较清晰，统一内存也可能表现得不错。

---

## 11.10 什么时候 UVM 往往不是最优选择

### 1. 高吞吐、稳定延迟要求强的系统

例如在线推理服务、严格 SLA 的实时系统。这类场景往往更怕不可预测的 page fault 和迁移抖动。

### 2. 访问阶段非常明确、可批量搬运

如果你很清楚：

- 先 CPU 准备
- 再整批上传
- GPU 计算一大段
- 最后再整批回传

那么显式 copy 方案通常更可控。

### 3. CPU 与 GPU 高频交替访问同一批数据

这是统一内存最容易"来回搬家"的模式之一。

### 4. 工作集很大且接近设备容量边界

这时页面管理压力更大，迁移也更容易变成主要成本。

### 5. 长期运行且性能稳定性比开发便利更重要

这类系统更倾向于显式控制驻留、传输和生命周期。

---

## 11.11 统一视图不等于统一物理现实

这是本章最重要的一条结论。

统一内存带来的"统一"，主要是：

- 统一地址视图
- 统一编程感受
- 统一逻辑对象

它没有消灭的，是这些物理现实：

- CPU 内存和 GPU 显存仍然是不同层级和不同路径
- 迁移仍然要花成本
- 页面仍然要驻留在某一边
- 访问冲突和访问阶段切换仍然会带来代价

所以真正成熟的理解应该是：

**UVM 统一的是使用接口，不是性能地形。**

---

## 11.12 UVM 的正确使用心态

如果把统一内存当成"自动最优内存系统"，通常会失望。
如果把它当成"降低编程复杂度的工具，并在必要时用 prefetch 和 advise 辅助"，就更接近正确预期。

一个更成熟的心态是：

1. 先用它降低开发复杂度
2. 理解其迁移和缺页成本
3. 观察访问模式
4. 决定哪些部分继续保留统一内存
5. 哪些热点路径改回显式控制

这比一开始就"全用"或"完全不用"都更实际。

---

## 11.13 本章小结

这一章最重要的结论有九条。

1. 统一内存的目标是降低主机/设备内存管理的编程割裂感。
2. 它让编程更方便，但没有消灭 CPU 与 GPU 之间的物理差异。
3. managed memory 的核心代价来自 page migration 和 page fault。
4. 统一视图不等于统一性能，逻辑可访问不代表物理位置最优。
5. `cudaMemPrefetchAsync` 的价值在于把部分迁移从"访问时被动触发"变成"提前主动安排"。
6. `cudaMemAdvise` 是向运行时表达使用意图，而不是绝对控制。
7. UVM 特别适合原型开发、复杂数据结构和开发效率优先的场景。
8. 对高吞吐、低抖动、阶段清晰的性能敏感系统，显式管理往往更可控。
9. 使用 UVM 的正确方式不是盲目信任自动化，而是理解它把复杂性从代码层转移到了运行时行为层。

---

# 第 12 章 深度学习训练中的 GPU 内存构成

## 12.1 本章目标

前面讲的内容主要还是底层主线：

- GPU 内存有硬件层次
- 地址空间和页映射不是可忽略背景
- `cudaMalloc` / `cudaFree` 有真实成本
- 数据传输与访问模式会深刻影响性能
- CUDA API 不是平铺罗列，而是系统能力的入口

从这一章开始，我们进入场景化内容。第一个场景就是最常见的 GPU 内存需求来源之一：**深度学习训练**。

很多人知道训练吃显存，但初期理解通常比较模糊，容易把一切都归结为一句话：

- "模型大，所以显存大"

这句话不完全错，但远远不够。训练显存通常并不只来自模型参数本身，还包括：

- 梯度
- 优化器状态
- 激活值
- 临时 workspace
- 通信缓冲区
- 框架分配器保留区

如果不把这些部分分开看，就很难真正理解：

- 为什么训练比推理更吃显存
- 为什么 batch size 一变显存就大跳
- 为什么有些优化方法能省很多显存
- 为什么"同一个模型"在不同训练配置下显存差异巨大

本章的目标是：

1. 把训练显存拆成几个主要组成部分。
2. 讲清这些部分分别何时出现、为何存在、如何随配置变化。
3. 解释为什么训练的峰值显存通常比推理高很多。
4. 为下一章训练显存优化方法打基础。

---

## 12.2 训练显存不是一块，而是多个部分叠加出来的

先给出最重要的全局图。

训练时，GPU 显存通常由下面几类东西共同组成：

1. 模型参数
2. 梯度
3. 优化器状态
4. 激活值
5. 临时工作区和中间 buffer
6. 通信 buffer
7. 分配器缓存和框架保留显存

这七类里，最容易被低估的通常是：

- 激活值
- 优化器状态
- 临时工作区
- 缓存分配器保留区

### 一个关键认识

"模型参数大小"只决定了显存账单的一部分，而且往往不是最大头。尤其在训练时，真正把显存拉高的，常常是：

- 反向传播需要保存的信息
- 优化器要维护的附加状态
- batch size 带来的激活膨胀

---

## 12.3 参数：训练显存中最容易理解的一部分

### 参数是什么

参数就是模型权重本身。它们必须放在设备上，才能让前向和反向高效运行。

### 它们的显存成本如何估算

最粗略的估算方式是：

- 参数个数
- 乘以每个参数的数据类型字节数

例如：

- FP32 参数：每个 4 字节
- FP16 / BF16 参数：每个 2 字节

### 为什么它只是"基础成本"

参数通常是训练显存中最容易直接理解的部分，因为它和模型规模直接相关。

但训练里参数通常不是唯一副本，有时还会伴随：

- 主权重和低精度副本
- 优化器相关状态副本
- 分布式切分前后的不同表示

所以"参数大小 = 训练显存大小"是严重低估。

---

## 12.4 梯度：训练才有，推理没有

### 梯度为什么存在

训练和推理最大的区别之一就在于：训练需要反向传播，而反向传播要为参数计算梯度。

因此，训练时往往要为参数准备对应梯度存储。

### 它通常意味着什么

粗略上，梯度存储规模常和参数规模同阶。也就是说：

- 有多少参数
- 往往就要为其对应的梯度留出类似量级的空间

### 为什么这已经让训练明显比推理贵

推理通常只需要权重和部分中间激活，而训练额外多出了一整套梯度空间。

### 还要注意什么

梯度的实际占用还会受到：

- 数据类型
- 梯度累积策略
- 分布式训练方式
- 是否做梯度分片

等因素影响。

---

## 12.5 优化器状态：常被忽略但非常"重"的部分

### 为什么会有优化器状态

训练不是简单地"拿梯度减一下参数"。现代优化器通常会维护额外统计量，比如动量、一阶二阶矩之类的状态。

### 为什么这会很占显存

因为这些状态往往也要按参数维度存储。换句话说，参数有多大，优化器状态通常也会按类似规模增长。

### 一个非常实用的直觉

对于很多常见优化器来说，优化器状态可能和参数一样重，甚至加起来比参数本体更重。

### 为什么初学者容易忽略它

因为这些状态在代码里往往不显眼。你看到的是：

- `optimizer.step()`

但你没直接"看到"背后还维护着大规模状态张量。

### 工程后果

这也是为什么：

- 同一个模型，训练时显存远高于推理
- 更换优化器会显著影响显存
- 大模型训练里优化器状态常常成为必须分片或 offload 的对象

---

## 12.6 激活值：训练显存里最容易爆炸的一部分

### 什么是激活值

激活值可以粗略理解为：

- 前向传播各层产生的中间结果
- 这些结果在反向传播时常常还要用到

### 为什么训练中必须保留很多激活

因为反向传播不是凭空计算的。很多梯度计算需要依赖前向阶段的中间结果。因此框架通常不能在前向刚结束就把所有中间结果都扔掉。

### 为什么它常常是大头

激活值大小高度依赖：

- batch size
- 序列长度
- 特征图尺寸
- 层数
- 隐藏维度
- 是否保存额外中间结果

因此，哪怕参数本身不算夸张，只要 batch 较大、输入尺寸较高、网络较深，激活值也可能迅速成为最大的显存部分。

### 一个特别关键的认识

**训练中的显存峰值，很多时候更受激活值影响，而不是只受参数量影响。**

这也是为什么 batch size 对训练显存的影响通常极大。

---

## 12.7 临时 workspace 与中间 buffer：不是主角，但常常在关键时刻顶出峰值

### 它们是什么

除了参数、梯度、激活这些长期或半长期对象外，训练过程中还会出现很多临时工作区，例如：

- 卷积算法临时 buffer
- GEMM / attention 相关 workspace
- 归约、重排、融合算子的中间 buffer
- 框架图执行中的临时块

### 为什么它们重要

因为它们虽然不一定长期常驻，但往往会在某个阶段瞬间抬高峰值显存。

这就是为什么你会看到：

- 平均显存占用还行
- 但某个步骤突然 OOM

### 工程含义

显存问题不能只看常驻对象，还要看：

- 某一时刻哪些临时 buffer 同时存在
- 哪些算子工作区和激活重叠
- 分配器是否能有效复用这些临时块

---

## 12.8 通信 buffer：多卡训练时不能忽略的那部分

### 为什么会有通信 buffer

在多 GPU 训练中，参数、梯度或激活的某些部分常常需要跨卡同步、聚合或交换。这就需要额外的通信缓冲区。

### 它们通常什么时候重要

单卡训练时这部分可以忽略很多，但在数据并行、模型并行、流水并行中，它们会越来越重要。

### 为什么它会影响显存预算

因为这些 buffer：

- 也占显存
- 可能在峰值阶段与激活或模型状态重叠
- 常常在单卡直觉预算里被忽略

### 工程后果

多卡训练时，显存预算不能只看"单卡模型本体"，还要把通信占用算进去。

---

## 12.9 框架保留显存：为什么训练时"看上去用得更多"

### 这部分是什么

这一部分不是模型语义上的数据，而是框架分配器为了性能而保留的显存。

### 为什么训练里特别明显

训练的张量生命周期很密集：

- 前向不断产生活跃张量
- 反向又会生成大量梯度和中间块
- step 之间重复相似模式

这种模式非常适合缓存分配器复用，也意味着框架通常会保留较多显存池。

### 这对理解训练显存很重要

因为你在外部看到的显存，往往包含：

- 活跃张量
- 临时对象
- 分配器空闲但已保留的块

所以训练中的"显存占用"不是单一概念。

---

## 12.10 为什么训练通常比推理更吃显存

现在可以把这个问题系统回答了。

训练比推理更吃显存，主要不是因为"训练更复杂"这句空话，而是因为训练多了几类关键对象：

1. 需要梯度
2. 需要优化器状态
3. 需要保留反向所需激活
4. 往往有更多临时工作区
5. 多卡训练时还有通信 buffer

而推理通常只有：

- 权重
- 当前批次激活
- 一些工作区
- 在大模型场景下的 KV cache

所以训练和推理的显存账单结构根本不同。

---

## 12.11 batch size 为什么是训练显存的中心变量

### 因为很多东西随 batch 增长

参数和优化器状态通常和模型本身相关，不随 batch 成比例变化太大。
但激活值、中间结果、部分 workspace 往往会随着 batch size 增长而明显增加。

### 这意味着什么

当你增大 batch size 时，常见情况是：

- 参数几乎不变
- 梯度规模大体不变
- 优化器状态不变
- 激活大涨
- 某些临时 buffer 也增大

所以训练显存常常对 batch size 非常敏感。

### 工程上的直接结论

如果你在调训练显存预算，batch size 通常是最先要看的大旋钮之一。

---

## 12.12 静态图与动态图：内存规划能力会不同

### 动态图的特点

动态图更灵活，控制流更自然，开发体验更好，但也意味着：

- 生命周期更动态
- 运行时更难提前做最理想的全局内存规划
- 某些临时块更依赖实时分配器策略

### 静态图的特点

静态图更容易：

- 提前分析依赖关系
- 规划哪些 buffer 可复用
- 降低峰值内存
- 把某些中间对象放到同一块物理空间轮流使用

### 为什么这会影响训练显存

因为很多内存优化的本质都是生命周期分析。图越稳定、阶段越明确，就越容易做激进复用。

### 但不要简单下结论

这并不意味着动态图一定差，只是说：

- 它通常更依赖分配器和运行时策略
- 静态图通常更容易做全局最优或近似最优的内存规划

---

## 12.13 一个训练显存的统一心智模型

你现在可以把训练显存理解成三层：

### 第一层：模型常驻成本

包括：

- 参数
- 优化器状态
- 常驻梯度或相关状态

这部分更像"底座"。

### 第二层：按 batch 和输入变化的运行成本

包括：

- 激活值
- 中间结果
- attention 或特征图相关缓冲区

这部分通常是最敏感、最容易爆的部分。

### 第三层：执行过程附加成本

包括：

- workspace
- 通信 buffer
- 分配器保留区
- 临时不可复用块

这部分往往决定峰值和抖动。

### 为什么这个模型有用

因为它能帮助你迅速判断某个问题更可能来自哪里：

- 模型太大
- batch 太大
- 优化器太重
- 激活保存太多
- 临时工作区太大
- 多卡通信额外占用
- 分配器碎片或保留过多

---

## 12.14 常见误区

### 误区 1：训练显存主要就是参数

错。很多时候激活值和优化器状态更重。

### 误区 2：模型参数不大，所以显存肯定够

错。你还没算梯度、优化器状态、激活、workspace 和缓存分配器保留区。

### 误区 3：batch size 只是影响算得快慢

错。它通常直接决定训练显存里最容易膨胀的部分。

### 误区 4：OOM 就说明模型放不下

错。很多 OOM 实际发生在某个峰值阶段，是激活、临时 buffer 或碎片问题，而不是常驻参数放不下。

---

## 12.15 本章小结

这一章最重要的结论有十条。

1. 训练显存不是单一对象，而是多类对象叠加的结果。
2. 主要组成包括参数、梯度、优化器状态、激活值、workspace、通信 buffer 和分配器保留区。
3. 参数只是训练显存账单的一部分，往往不是全部甚至不是最大头。
4. 梯度是训练相对推理新增的重要成本。
5. 优化器状态经常很重，尤其在大模型训练中不可忽略。
6. 激活值通常是训练显存里最容易随 batch 和输入规模爆炸的部分。
7. 临时工作区和中间 buffer 往往决定峰值显存，而不是平均显存。
8. 多卡训练需要把通信 buffer 纳入预算。
9. 训练比推理显存更高，根本原因在于训练多了反向传播和参数更新所需的大量状态。
10. 分析训练显存时，要把常驻成本、运行成本和执行附加成本分开看。

---

# 第 13 章 训练中的内存优化：从混合精度到 ZeRO

## 13.1 本章目标

上一章我们已经把训练显存账单拆开了。现在问题变成：

**既然知道显存花在哪，工程上到底该怎么降？**

很多优化名词你可能已经听过：

- mixed precision
- activation checkpointing
- gradient accumulation
- ZeRO
- FSDP
- offload

但如果没有上一章的显存构成视角，这些方法很容易变成一堆零散技巧。真正有效的理解方式应该是：

- 它在优化哪一类显存
- 它的代价是什么
- 它适合什么场景
- 它为什么会有效

本章的目标是：

1. 把训练显存优化方法按"作用对象"分类。
2. 讲清每种方法究竟在省哪一块显存。
3. 解释这些方法各自交换掉了什么：计算、通信、稳定性、实现复杂度。
4. 建立训练 OOM 的实用排查顺序。

---

## 13.2 先建立一个总框架：训练显存优化其实是在动四类东西

从上一章可知，训练显存主要由以下部分构成：

1. 参数
2. 梯度
3. 优化器状态
4. 激活值
5. 临时 workspace 与通信 buffer

因此，训练优化方法也可以按它们作用的对象分类。

### 第一类：减少参数相关成本

常见手段：

- mixed precision
- 参数分片
- 权重量化训练中的某些策略
- offload

### 第二类：减少梯度与优化器状态成本

常见手段：

- gradient accumulation 配合更小 micro-batch
- optimizer state sharding
- ZeRO / FSDP
- CPU/NVMe offload

### 第三类：减少激活值成本

常见手段：

- activation checkpointing
- recomputation
- 更小 batch
- 更短序列
- 更小 feature map / resolution

### 第四类：减少峰值和临时额外成本

常见手段：

- 更稳定的分配策略
- 编译图优化与 buffer 复用
- kernel fusion
- 通信和计算重叠
- 更好的生命周期规划

只要你先有这个分类，任何"训练显存优化技巧"都会立刻变得可解释。

---

## 13.3 mixed precision：为什么它既是容量优化，也是带宽优化

### 它在做什么

mixed precision 的核心不是"所有东西都变低精度"，而是：

- 在保证训练稳定性的前提下
- 尽量让更多计算和存储使用更低字节宽度的数据格式
- 比如 FP16、BF16
- 同时保留某些关键部分仍以更高精度维护

### 它省了哪部分显存

最直接的是：

- 参数表示可能更小
- 梯度表示可能更小
- 激活值通常也更小
- 某些中间 buffer 更小

如果原来很多内容是 FP32，换成 16-bit 格式后，很多对象的字节数会接近减半。

### 为什么它不只是显存优化

因为数据更小，还意味着：

- 显存带宽压力下降
- cache 与寄存器承载同样"元素数"时更有利
- 传输和某些通信成本也会下降

所以 mixed precision 往往同时改善：

- 容量
- 带宽
- 吞吐

### 代价是什么

主要代价包括：

- 数值稳定性要管理
- 某些场景需要 loss scaling 或其它稳定策略
- 并不是所有层或所有状态都能无脑降精度
- 调试会更复杂一点

### 工程上的真实定位

mixed precision 往往是训练显存优化里性价比最高、最优先尝试的一类方法，因为它通常既省显存，也可能提速。

---

## 13.4 activation checkpointing：用更多计算换更少激活显存

### 它到底在做什么

activation checkpointing 的核心思想是：

- 前向传播时，不保存所有中间激活
- 只保存少数"检查点"
- 到反向传播时，如果需要某些中间结果，就重新计算一遍

### 它省了哪部分显存

主要省的是：

- 激活值

尤其对深层网络、大 batch、长序列任务，激活常常是最大头之一，因此 checkpointing 经常能显著降低峰值显存。

### 为什么会有效

因为前向产生的大量中间结果，并不是每个都必须永久保存。只要你愿意在反向时重新跑一部分前向计算，就可以把"存很多"变成"算两次"。

### 它的代价是什么

代价非常明确：

- 额外计算增加
- 训练时间可能变长
- 某些图结构下实现更复杂
- 对执行图和框架支持有一定要求

### 一个重要认识

这类优化告诉你一个 GPU 工程常见原则：

**有时重复计算比保存大量中间内存更便宜。**

这在 CPU 思维里不总是直觉，但在 GPU 训练里非常常见。

---

## 13.5 recomputation：它和 checkpointing 的关系

### 本质关系

recomputation 是更广义的说法，表示：

- 某些值不保存
- 需要时重算

activation checkpointing 是其中最典型的一种。

### 为什么单独提它

因为这个思想不只适用于标准"分段 checkpoint"，还可以出现在：

- 特定模块的局部重算
- 某些 attention 中间量不缓存、按需重建
- 编译器或框架自动做的内存-计算折中

### 工程含义

你在看任何训练优化方法时，都可以问一句：

- 它是在"少存多算"吗？

如果答案是是，那本质上它就站在 recomputation 这一类思路上。

---

## 13.6 gradient accumulation：为什么它能让你"像大 batch 一样训练"

### 它在做什么

gradient accumulation 的思路是：

- 不一次性喂一个很大的 batch
- 而是拆成多个更小的 micro-batch
- 每次 forward/backward 后先累积梯度
- 不立刻更新参数
- 等累积到一定步数后再做一次 optimizer step

### 它省了哪部分显存

它并不直接减少参数、梯度或优化器状态总规模，但它通常能减少：

- 单次前向/反向所需的激活值峰值
- 某些与 batch size 相关的中间 buffer

也就是说，它的主要作用是：

- 用更多步数换取更小的单步显存峰值

### 为什么它很常用

因为很多训练任务真正卡住的是：

- 单步 micro-batch 放不下

而不是：

- 逻辑上的总 batch 太大

通过累积梯度，你可以在显存更小的设备上，接近实现较大 batch 的训练效果。

### 它的代价是什么

- 每次参数更新之间要做更多次前后向
- 训练节奏与优化超参可能需要调整
- 多卡同步与吞吐模式可能变化
- 不一定提升墙钟时间效率

### 工程定位

当你主要是被 batch size 引起的激活峰值卡住时，gradient accumulation 是非常实用的一类方法。

---

## 13.7 optimizer state sharding：为什么优化器状态必须切

### 为什么先切优化器状态

在大模型训练中，优化器状态常常非常重，有时甚至是最难承受的常驻成本之一。

如果每张卡都完整保留：

- 全量参数
- 全量梯度
- 全量优化器状态

那么显存会非常快撞墙。

### sharding 在做什么

sharding 的基本思想是：

- 不让每张卡都保留完整副本
- 而是把参数、梯度或优化器状态拆分到多张卡上
- 每张卡只持有其中一部分

### 为什么这特别适合优化器状态

因为优化器状态是参数维度的附加常驻张量，切分后每卡压力会明显降低。

### 代价是什么

- 需要更多通信
- 逻辑更复杂
- 执行时要在"本地少存"和"跨卡取用"之间做权衡

### 这类方法的本质

本质上是在做空间换通信：

- 每卡少存一点
- 多卡之间多协调一点

---

## 13.8 ZeRO：把训练状态分散到多卡上

### 先说直觉

ZeRO 可以粗略理解为：

- 不让每张卡都存一整份训练状态
- 而是按阶段把参数、梯度、优化器状态拆开分布到不同 GPU 上

### 它为什么重要

因为在传统数据并行中，每张卡经常持有完整模型相关状态，显存利用非常不经济。ZeRO 的价值就在于：

- 把冗余副本拆掉
- 降低单卡显存压力
- 让更大模型能够训练

### 它主要在省什么

不同阶段的 ZeRO 会逐步覆盖：

- 优化器状态
- 梯度
- 参数本身

从而让单卡显存成本从"持有全量状态"变成"持有分片状态"。

### 代价是什么

- 通信更复杂
- 调度更复杂
- 性能更依赖网络与实现质量
- 某些场景吞吐可能受影响

### 一个重要理解

ZeRO 不是"凭空省显存"，而是把"每卡都全存"的模式改成"多卡协同分摊"。

---

## 13.9 FSDP：参数也做更细粒度的切分与重组

### 它的直觉

FSDP 可以粗略看作一种更系统化的全分片数据并行思路，重点在于：

- 参数本身也不总是完整常驻于每张卡
- 某些计算阶段按需聚合、计算后再释放或重新分散

### 为什么它有用

因为当模型极大时，仅仅切优化器状态和梯度还不够，参数本身也会成为单卡瓶颈。

### 它主要优化了什么

- 参数常驻占用
- 梯度和状态的复制冗余
- 每卡显存压力

### 它的代价

- 更复杂的通信调度
- 更复杂的执行时 gather/scatter 行为
- 性能表现更依赖实现、拓扑与 batch 配置

### 工程上的理解方式

FSDP 的关键不在于背名字，而在于知道它是在进一步压缩"单卡必须完整持有多少训练状态"。

---

## 13.10 offload：为什么可以把部分状态挪到 CPU 或更慢存储

### 它在做什么

offload 的核心思想是：

- 把不一定要时刻驻留在 GPU 上的部分状态
- 临时放到 CPU 内存，甚至更慢层级
- 需要时再搬回来

常见对象包括：

- 优化器状态
- 一部分参数
- 某些梯度或中间结果

### 为什么它有效

因为 GPU 显存是最稀缺、最贵的那层。只要你的系统还有：

- 较大 CPU 内存
- 可接受的传输代价
- 能设计合理的数据驻留节奏

就有机会用更慢存储换 GPU 显存空间。

### 它的代价是什么

- 传输成本上升
- 延迟更高
- 更依赖流水线与重叠设计
- 实现复杂度变高

### 它适合什么场景

通常适合：

- 模型太大，单靠 GPU 显存放不下
- 能接受吞吐下降
- 目标是"能训练起来"，而不是首先追求极致速度

---

## 13.11 graph capture 与静态内存规划：为什么"图更稳定"时更省显存

### 先说原理

如果训练步骤的执行图足够稳定，那么运行时或编译器就更容易：

- 预测对象生命周期
- 知道哪些 buffer 不会同时活着
- 让它们复用同一块内存
- 减少动态分配

### 为什么这和显存有关

因为很多显存峰值问题，本质不是对象总量太大，而是：

- 生命周期没规划好
- 本可复用的块没有复用
- 动态分配太多

### graph capture 的价值

它的一个重要收益就是：

- 提高执行形态稳定性
- 减少某些运行时抖动
- 帮助内存池和规划器做更激进复用

### 工程含义

如果你的训练步骤形态稳定，通常更容易做到：

- 更低分配开销
- 更少碎片
- 更低峰值显存

---

## 13.12 训练 OOM 时，应该先按什么顺序排查

### 第一步：先看是不是激活峰值

最常见。检查：

- batch size 是否过大
- 序列长度是否过长
- 输入分辨率是否过高
- 是否能先启用 checkpointing 或减小 micro-batch

### 第二步：看是否已经使用混合精度

如果没有，这是最优先尝试的项之一。

### 第三步：看优化器状态是否太重

尤其是大模型训练，检查：

- 当前优化器是否显存代价高
- 是否能做 sharding / ZeRO / FSDP
- 是否需要 offload

### 第四步：看是否有临时 workspace 峰值

某些算子或阶段会突然产生较大临时块，导致峰值打爆。

### 第五步：看是不是分配器碎片或缓存策略问题

如果显存统计看起来"还有"，但某些大分配失败，就要怀疑：

- 碎片
- 池内大块不足
- 生命周期交错

### 第六步：多卡场景检查通信 buffer 和并行策略

有时问题并不在模型主体，而在并行框架引入的额外开销。

---

## 13.13 一个统一视角：每种优化到底在拿什么换什么

把常见训练显存优化压缩成一个交换表，会非常清楚：

| 方法 | 主要节省 | 主要代价 |
|---|---|---|
| mixed precision | 参数/梯度/激活/带宽 | 数值稳定性管理 |
| checkpointing | 激活 | 额外计算 |
| gradient accumulation | 单步激活峰值 | 更多步数/吞吐变化 |
| optimizer sharding | 优化器状态 | 通信复杂度 |
| ZeRO / FSDP | 参数/梯度/状态冗余 | 通信与实现复杂度 |
| offload | GPU 显存 | 传输与延迟 |
| graph capture / 静态规划 | 分配与峰值 | 图形态约束 |

只要你记住"省哪块、换什么"，就不会被术语带着跑。

---

## 13.14 本章小结

这一章最重要的结论有十条。

1. 训练显存优化首先要按作用对象分类：参数、梯度、优化器状态、激活、峰值临时块。
2. mixed precision 往往是最高优先级的显存优化之一，因为它同时影响容量和带宽。
3. activation checkpointing 的本质是用额外计算换更少激活显存。
4. recomputation 是更广义的"少存多算"思路。
5. gradient accumulation 主要降低单步激活峰值，而不是消灭模型常驻成本。
6. optimizer state sharding、ZeRO、FSDP 的本质是把训练状态从"每卡全持有"改成"多卡分摊持有"。
7. offload 是用更慢层级存储换 GPU 显存空间。
8. graph capture 和静态内存规划有助于减少动态分配和峰值占用。
9. 训练 OOM 排查时，通常先看激活峰值，再看精度、优化器状态、workspace、碎片和并行策略。
10. 每种显存优化方法本质上都是一种交换：省空间，换计算、通信、传输、稳定性或实现复杂度。

下一章我们从训练切到推理。推理和训练表面上都在"跑模型"，但显存账单结构并不一样。尤其在大模型推理里，KV cache 会成为真正的核心对象。

# 第 14 章 深度学习推理中的 GPU 内存管理

## 14.1 本章目标

训练时显存管理的重点是：参数、梯度、优化器状态、激活。推理时没有梯度回传，没有优化器状态更新，显存账单结构完全不同。

本章目标是：

1. 讲清推理显存构成的独特之处。
2. 解释 KV cache 为什么在大模型推理中成为显存核心。
3. 说明推理中 batching 对显存的影响。
4. 区分单次推理、流式推理、批处理推理的显存模式差异。
5. 建立推理显存分析的心智模型。

---

## 14.2 核心结论

推理显存管理的核心结论只有一条：

**在大模型推理中，KV cache 常常比模型权重本身更占显存。**

这不是夸张。一个 70B 参数的模型，权重用 FP16 约 140GB。但如果序列长度很长、batch size 很大，KV cache 可以轻松超过这个数。

---

## 14.3 直觉理解：推理和训练的显存账单为什么不同

### 训练时的显存大头

- 模型参数
- 优化器状态
- 梯度
- 激活值

### 推理时的显存大头

- 模型参数（只读）
- KV cache（自回归生成时累积）
- 输入/输出 buffer
- 临时 workspace

推理没有梯度和优化器状态，但多了一个训练时没有的东西：**需要保存历史 token 的 key/value 张量，供后续 token 计算注意力使用。**

这就是 KV cache。

---

## 14.4 正式定义：推理显存的构成项

### 模型权重

和训练一样，模型参数需要常驻显存。推理时通常是只读的。

### KV cache

自回归 Transformer 中，每个 token 计算注意力时，需要用到所有之前 token 的 key 和 value。

为了避免重复计算，推理框架会把已生成 token 的 key/value 张量缓存起来。

KV cache 的大小取决于：

- 层数 (num_layers)
- 注意力头数 (num_heads) 或 GQA/MQA 配置
- 头维度 (head_dim)
- batch size
- 当前序列长度

### 输入输出 buffer

- 输入 token IDs
- 输出 logits / 采样结果
- 位置编码等辅助数据

### 临时 workspace

某些算子需要临时 buffer，比如 softmax、layernorm、矩阵乘的 split-k buffer 等。

### 框架开销

推理框架本身也会占用一些显存用于调度、队列、状态管理等。

---

## 14.5 底层机制：KV cache 的显存增长规律

### 自回归生成的显存特点

训练时，一个 batch 内所有 token 并行计算，激活值是"一次性"的。

推理时，生成是逐个 token 进行的：

1. 输入 prompt，计算所有 token 的 KV（prefill 阶段）
2. 生成第 1 个新 token（decode 阶段）
3. 把新 token 的 KV 追加到 cache
4. 生成第 2 个新 token
5. 继续追加...

### KV cache 的显存公式（简化版）

对标准多头注意力：

```
KV_cache_size = 2 * num_layers * num_heads * head_dim * batch_size * seq_len * sizeof(dtype)
```

其中 `2` 是因为 key 和 value 各一份。

### 用具体数字感受

假设：
- 层数 80
- 头数 64，头维度 128
- batch size 32
- 序列长度 4096
- FP16 (2 bytes)

```
KV_cache = 2 * 80 * 64 * 128 * 32 * 4096 * 2
         ≈ 343.6 GB
```

这只是一个 batch 的 KV cache。如果 batch size 更大、序列更长，这个数字会进一步膨胀。

### 为什么 GQA / MQA 能降低显存

Grouped-Query Attention (GQA) 和 Multi-Query Attention (MQA) 通过让多个查询头共享同一组 key/value 头，减少了 KV cache 的容量。

例如 MQA 中所有查询头共享 1 个 key 头和 1 个 value 头，KV cache 降低到原来的 `1/num_heads`。

这是推理优化的重要方向之一。

---

## 14.6 典型例子：对比训练和推理的显存占用

### 场景

一个 13B 参数的模型，FP16 精度。

### 训练时（单卡，batch size 4，序列长度 2048）

| 项目 | 估算大小 |
|---|---|
| 模型参数 | 26 GB |
| 梯度 | 26 GB |
| 优化器状态 (Adam) | 52 GB |
| 激活值 | 20-40 GB |
| 总计 | 124-144 GB |

### 推理时（batch size 16，序列长度 4096，已生成长度 2048）

| 项目 | 估算大小 |
|---|---|
| 模型参数 | 26 GB |
| KV cache | ~40-80 GB |
| 输入输出 buffer | 较小 |
| workspace | 数 GB |
| 总计 | 70-110 GB |

可以看到，推理时没有梯度和优化器状态，但 KV cache 成为了新的显存大户。

---

## 14.7 场景延伸：不同推理模式的显存特征

### 单次推理（single request）

- 显存占用低
- KV cache 随序列增长线性增加
- 主要瓶颈通常是延迟而非显存

### 批处理推理（offline batch）

- batch size 大，KV cache 与 batch size 成正比
- 显存容易成为瓶颈
- 吞吐优化空间大

### 流式服务（online serving）

- 同时服务多个请求，每个请求序列长度不同
- KV cache 是动态变化的
- 需要高效的内存管理来应对不同生命周期的请求
- 这是PagedAttention等技术的核心应用场景

### 长文本推理

- 序列长度极大时，KV cache 可能超过权重
- 需要特殊的 KV cache 压缩、驱逐或分层策略

---

## 14.8 常见误区

### 误区 1：推理显存一定比训练少

不一定。大 batch 长序列的推理，KV cache 可能让总显存超过训练时的单步显存。

### 误区 2：模型量化只影响权重

量化权重确实能省显存，但如果 KV cache 仍是 FP16，它可能很快成为新的瓶颈。

### 误区 3：batch size 越大越好

batch size 增大提升吞吐，但 KV cache 线性增长。存在一个"显存墙"限制最大 batch size。

### 误区 4：推理不需要关心内存碎片

流式服务中，不同请求动态创建和释放 KV cache，碎片问题可能比训练更严重。

---

## 14.9 工程建议

### 建议 1：始终把 KV cache 纳入显存预算

做推理显存规划时，不要只看模型权重，KV cache 往往是真正的变量。

### 建议 2：用公式快速估算

掌握简化版 KV cache 公式，能在设计阶段快速判断方案是否可行。

### 建议 3：关注 GQA/MQA 等结构优化

模型设计阶段就考虑推理时的 KV cache 效率，比事后优化更容易。

### 建议 4：区分 prefill 和 decode 的显存特征

- prefill 阶段：并行计算 prompt，激活值大，但 KV cache 一次性写入
- decode 阶段：逐个生成，激活值小，但 KV cache 持续增长

两阶段的优化策略不同。

### 建议 5：流式服务中预留显存余量

动态请求导致 KV cache 不可预测，不要按"平均情况"设计，要按"峰值情况"留余量。

---

## 14.10 本章小结

这一章最重要的结论有八条。

1. 推理显存构成与训练不同：没有梯度和优化器状态，但有 KV cache。
2. 在大模型长序列推理中，KV cache 常常超过模型权重成为最大显存消费者。
3. KV cache 大小与层数、头数、头维度、batch size、序列长度成正比。
4. GQA/MQA 通过共享 KV 头显著降低 KV cache 容量。
5. 单次推理、批处理推理、流式服务的显存模式差异很大。
6. 推理也要关心碎片问题，尤其是动态请求场景。
7. prefill 和 decode 阶段的显存特征不同，优化策略也不同。
8. 做推理显存预算时，必须同时考虑权重和 KV cache。

# 第 15 章 推理优化专题：量化、分页缓存与连续批处理

## 15.1 本章目标

上一章建立了推理显存的基本模型。本章深入三个最关键的推理优化方向：

1. **量化 (Quantization)**：用更低位宽表示权重和/或激活
2. **分页缓存 (PagedAttention / vLLM)**：解决 KV cache 的内存碎片和过度预留问题
3. **连续批处理 (Continuous Batching)**：提高流式服务中的吞吐和显存效率

本章目标是：

1. 讲清每种优化的核心原理。
2. 说明它们各自解决什么问题、带来什么收益、付出什么代价。
3. 解释它们之间的组合关系。
4. 建立对推理系统显存优化的完整认知。

---

## 15.2 核心结论

1. **量化**主要解决"模型权重和 KV cache 的位宽太大"问题。
2. **分页缓存**主要解决"KV cache 的内存分配效率低、碎片多、过度预留"问题。
3. **连续批处理**主要解决"请求到达时间不同导致的 batch 效率低和显存浪费"问题。
4. 三者可以组合使用，共同提升推理服务的吞吐和容量。

---

## 15.3 量化：用精度换容量

### 直觉理解

FP16 每个数占 2 字节。如果改用 INT8，每个数只占 1 字节。如果改用 INT4，只占 0.5 字节。

显存占用直接按比例下降。

### 正式定义

量化是把浮点数映射到低位宽整数表示的过程：

```
q = round((x - zero_point) / scale)
x_dequant = q * scale + zero_point
```

### 量化的层次

| 层次 | 说明 |
|---|---|
| 权重量化 (W) | 只量化模型参数 |
| 权重+激活量化 (WA) | 量化参数和激活值 |
| KV cache 量化 | 量化 KV cache |
| 全量化 | 上述全部 |

### 对显存的影响

| 精度 | 权重显存 | KV cache 显存 | 计算类型 |
|---|---|---|---|
| FP16 | 100% | 100% | FP16 |
| INT8 (W) | 50% | 100% | FP16/INT8 |
| INT8 (WA) | 50% | 50% | INT8 |
| INT4 (W) | 25% | 100% | FP16/INT8 |
| INT4 (WA) + KV INT8 | 25% | 50% | INT8/INT4 |

### 代价

- 精度损失：低位宽表达能力弱，可能降低模型质量
- 计算复杂度：需要反量化或特殊 kernel 支持
- 实现复杂度：需要校准、per-channel/per-token 策略等

### 典型例子

一个 70B 模型：
- FP16：140 GB 权重
- INT8 权重量化：70 GB 权重
- INT4 权重量化：35 GB 权重

如果同时把 KV cache 也量化到 INT8，KV cache 也能减半。

---

## 15.4 分页缓存：把 KV cache 当"虚拟内存"管理

### 想解决的问题

传统推理中，每个请求的 KV cache 通常预分配一块连续内存，大小按最大可能序列长度计算。

这导致两个问题：

1. **内部碎片**：请求实际序列远小于最大长度时，预分配的空间大量浪费。
2. **外部碎片**：不同请求释放后，留下不连续的空洞，导致后续大请求分配失败。

### 直觉理解

操作系统用"页"来管理内存，避免进程直接操作物理地址的碎片问题。

PagedAttention 把同样的思想用到 KV cache：

- 把 KV cache 切成固定大小的 block（比如 16 tokens 一块）
- 按需分配 block，而不是一次性预分配一大块
- block 不需要物理连续，通过 block table 映射逻辑位置到物理 block

### 正式定义

PagedAttention 的核心数据结构：

```
Block Table (per sequence):
  logical_block_0 -> physical_block_7
  logical_block_1 -> physical_block_3
  logical_block_2 -> physical_block_12
  ...
```

逻辑上连续的 KV cache，物理上分散在不同 block 中。

### 对显存的影响

1. **消除内部碎片**：只分配实际需要的 block
2. **减少外部碎片**：block 大小固定，回收和复用更容易
3. **支持共享**：不同序列可以共享相同的物理 block（比如 beam search、parallel sampling）

### 典型例子

假设最大序列长度 4096，block size 16：

传统方式：
- 每个请求预分配 4096 * head_dim 的连续空间
- 实际只生成 100 tokens，浪费 97.5%

PagedAttention：
- 只需要 ceil(100/16) = 7 个 block
- 浪费只有最后一个 block 内的 6 个 token 位置

### 工程意义

vLLM 基于 PagedAttention 构建，它让 GPU 显存中可以同时容纳的请求数量大幅提升，因为：

- 没有过度预留
- 碎片极少
- 支持请求间的 KV cache 共享

---

## 15.5 连续批处理：不让 GPU 闲着

### 想解决的问题

传统批处理：

1. 收集一批请求
2. 等最慢的请求完成
3. 整批释放，再收集下一批

问题：
- 快的请求要等慢的
- 新请求要等当前批完成
- GPU 在某些时刻利用率不高

### 直觉理解

连续批处理（也叫 in-flight batching 或 iterative scheduling）的核心思想：

**不等待整批完成，每当有请求完成或新请求到达，就重新组织当前 batch。**

就像 CPU 上的多任务调度，不是等一个进程跑完再跑下一个，而是不断切换。

### 对显存的影响

1. **显存利用更充分**：不会因为等待慢请求而闲置显存
2. **需要更灵活的 KV cache 管理**：请求动态进出，KV cache 生命周期不同
3. **通常配合 PagedAttention**：因为 PagedAttention 的 block 管理机制天然支持动态增删

### 典型例子

三个请求：
- A：需要 10 个迭代
- B：需要 50 个迭代
- C：需要 30 个迭代

传统批处理：
- 一起启动，等 B 完成（50 迭代）
- A 完成后 GPU 上仍保留其资源直到整批结束

连续批处理：
- A 完成于第 10 迭代，立即释放其 KV cache，新请求 D 可以加入
- C 完成于第 30 迭代，同样立即释放
- B 继续运行到第 50 迭代

GPU 始终在处理"当前活着的请求集合"，没有整批等待的浪费。

---

## 15.6 三种优化的组合关系

### 量化 + 分页缓存

- 量化降低每个 token 的 KV cache 大小
- 分页缓存提高 KV cache 的分配效率
- 组合效果：同样显存能服务更多请求

### 分页缓存 + 连续批处理

- 分页缓存提供灵活的 KV cache 增删能力
- 连续批处理需要这种灵活性来动态调整 batch 组成
- 二者天然互补，vLLM 就是典型组合

### 三者一起

- 量化减少单请求显存 footprint
- 分页缓存减少分配浪费
- 连续批处理提高时间维度的利用率
- 这是当前大模型推理服务的主流优化方向

---

## 15.7 场景延伸

### 场景 1：离线批处理

- 量化收益最大（降低峰值显存，增大 batch size）
- 分页缓存收益中等（请求长度相对均匀）
- 连续批处理收益较小（没有动态到达）

### 场景 2：在线聊天服务

- 量化收益大（服务更多并发用户）
- 分页缓存收益极大（请求长度差异大，动态进出频繁）
- 连续批处理收益极大（请求到达时间随机）

### 场景 3：代码生成服务

- 生成长度差异极大（有的几行，有的几百行）
- 分页缓存和连续批处理收益显著
- 量化需要注意：代码任务对精度更敏感

---

## 15.8 常见误区

### 误区 1：量化总是无损的

不是。低位宽必然损失精度，只是某些任务对精度不敏感。需要评估具体任务的 quality degradation。

### 误区 2：分页缓存让 KV cache 消失

没有。分页缓存解决的是分配效率问题，不是 KV cache 的容量问题。它让同样显存能装更多，但 KV cache 本身还在。

### 误区 3：连续批处理一定更好

调度本身有开销。如果请求高度同步、长度均匀，传统批处理可能更简单高效。

### 误区 4：三种优化都要上

根据场景选择。离线批处理可能只需要量化；某些场景精度敏感就不能量化。

---

## 15.9 工程建议

### 建议 1：先做显存 profiling

知道权重、KV cache、activation 各占多少，再决定优化方向。

### 建议 2：量化是"免费午餐"但有上限

INT8 权重量化通常精度损失小，INT4 需要更仔细评估。KV cache 量化相对安全。

### 建议 3：流式服务优先投资分页缓存和连续批处理

这两个技术对服务质量和吞吐的影响通常大于单纯量化。

### 建议 4：注意 kernel 支持

量化不只是存储问题，还需要对应的计算 kernel（如 CUTLASS、Marlin、AWQ kernel 等）。

### 建议 5：监控实际显存使用

优化后的系统需要监控：
- 实际 KV cache 占用
- block 利用率
- 批处理效率
- 请求排队延迟

---

## 15.10 本章小结

这一章最重要的结论有十条。

1. 量化通过降低位宽减少权重和/或 KV cache 的显存占用，代价是精度和实现复杂度。
2. 权重量化、激活量化、KV cache 量化可以独立选择，组合使用。
3. PagedAttention 把操作系统的分页思想引入 KV cache 管理，解决内部碎片和外部碎片。
4. 分页缓存让逻辑连续的 KV cache 可以物理分散，支持按需分配和共享。
5. 连续批处理让请求动态进出，避免整批等待的浪费，提高 GPU 时间利用率。
6. 分页缓存和连续批处理天然互补，前者提供灵活的空间管理，后者需要这种灵活性。
7. 三种优化可以组合，但应根据场景选择：离线批处理侧重量化，在线服务侧重分页和连续批处理。
8. 量化不是无损的，需要根据任务评估精度影响。
9. 分页缓存不减少 KV cache 的理论总量，只提高分配效率和实际利用率。
10. 推理优化的核心目标是：在显存预算内服务尽可能多的请求，同时满足延迟和精度要求。

---

# 第 16 章 通用 CUDA/C++ 开发中的内存设计模式

## 16.1 本章目标

前面几章重点讲了深度学习的显存管理。但 GPU 内存管理不仅限于深度学习。本章回到更通用的 CUDA/C++ 开发场景，讨论常见的内存设计模式。

本章目标是：

1. 介绍通用 CUDA 程序中常见的内存管理策略。
2. 讲解 RAII、内存池、环形缓冲区等模式在 GPU 上的应用。
3. 说明主机端和设备端内存管理的协同设计。
4. 建立跨场景适用的 GPU 内存设计原则。

---

## 16.2 核心结论

1. **RAII 原则同样适用于 GPU 内存**：用对象生命周期管理显存生命周期，避免泄漏。
2. **内存池是高频小分配的标准答案**：减少 `cudaMalloc`/`cudaFree` 调用开销和碎片。
3. **异步和流是 GPU 内存管理的核心语境**：分配、释放、传输都要考虑异步执行上下文。
4. **主机端和设备端内存需要协同设计**：不能只优化一边。

---

## 16.3 模式一：RAII 包装器

### 直觉理解

`cudaMalloc` 和 `cudaFree` 是 C 风格 API，容易忘记释放或异常时泄漏。RAII（Resource Acquisition Is Initialization）把资源生命周期绑定到对象生命周期。

### 实现示例

```cpp
template <typename T>
class DeviceBuffer {
    T* ptr_ = nullptr;
    size_t size_ = 0;
public:
    DeviceBuffer(size_t n) : size_(n) {
        cudaMalloc(&ptr_, n * sizeof(T));
    }
    ~DeviceBuffer() {
        if (ptr_) cudaFree(ptr_);
    }
    // 禁止拷贝，允许移动
    DeviceBuffer(const DeviceBuffer&) = delete;
    DeviceBuffer& operator=(const DeviceBuffer&) = delete;
    DeviceBuffer(DeviceBuffer&& other) noexcept { /* ... */ }
    
    T* data() const { return ptr_; }
    size_t size() const { return size_; }
};
```

### 好处

- 异常安全：栈展开时自动释放
- 代码清晰：所有权明确
- 可组合：可以放入标准容器

---

## 16.4 模式二：内存池（Memory Pool）

### 直觉理解

如果程序频繁分配释放不同大小的 buffer，直接调用 `cudaMalloc`/`cudaFree` 会很慢。内存池预先分配一大块，内部管理分配和回收。

### 简单实现思路

```cpp
class MemoryPool {
    void* pool_base_;
    size_t pool_size_;
    // 空闲块列表，按大小或地址排序
    std::vector<Block> free_blocks_;
public:
    void* allocate(size_t size);
    void deallocate(void* ptr);
};
```

### 工程要点

- 对齐要求：GPU 分配通常需要 256 字节对齐
- 合并空闲块：回收时检查相邻块是否可以合并
- 大小分级：小对象和大对象用不同策略
- 线程安全：多线程访问需要同步

---

## 16.5 模式三：环形缓冲区（Ring Buffer）

### 适用场景

流式数据处理，比如：
- 视频处理帧队列
- 传感器数据流
- 流水线阶段的中间结果

### 直觉理解

固定分配 N 个 buffer，生产者用完一个就推进到下一个，消费者按顺序消费。当到达末尾时绕回开头，形成环形。

### 好处

- 无动态分配：所有 buffer 预先分配
- 可预测显存：总显存 = N * buffer_size
- 适合异步流水线：生产者和消费者可以不同步

---

## 16.6 模式四：对象池（Object Pool）

### 适用场景

需要频繁创建和销毁的 GPU 端对象，比如：
- CUDA graph 对象
- 纹理对象
- 小规模的临时结构

### 与内存池的区别

内存池管理的是原始字节。对象池管理的是有构造/析构语义的对象。

---

## 16.7 模式五：双缓冲/多缓冲

### 直觉理解

GPU 计算时，同时准备下一批数据。两个 buffer 交替使用：
- Buffer A：GPU 正在计算
- Buffer B：CPU 正在准备数据

计算完成后交换角色。

### 对显存的影响

需要 2x 的显存来存储输入/输出 buffer，但隐藏了传输延迟。

### 扩展：三缓冲

加入一个"待显示/待消费"的 buffer，进一步解耦生产和消费节奏。

---

## 16.8 模式六：延迟释放（Deferred Deallocation）

### 直觉理解

GPU 执行是异步的。一个 buffer 在 CPU 端"逻辑上"已经没用了，但 GPU kernel 可能还在读它。

延迟释放就是：不立即释放，而是记录"这个 buffer 在 stream 的某个点之后可以释放"，等 GPU 执行到那个点再真正释放。

### 实现方式

```cpp
// 在 stream 上插入一个 callback，在 callback 中释放
cudaLaunchHostFunc(stream, [](void* ptr) {
    cudaFree(ptr);
}, buffer_ptr);
```

或使用 CUDA graph 的依赖管理。

---

## 16.9 主机端与设备端的协同设计

### 原则 1：避免过度 pinned memory

Pinned memory 是稀缺资源，过多会影响 OS 内存管理。只 pin 真正需要频繁传输的 buffer。

### 原则 2：异步传输需要 pinned buffer

`cudaMemcpyAsync` 需要主机端内存是 pinned 的。如果 buffer 不是 pinned，CUDA 驱动会：
- 先临时分配 pinned buffer
- 把数据拷到 pinned buffer
- 再启动异步传输

这增加了延迟和额外开销。

### 原则 3：考虑统一内存的适用场景

对于 CPU/GPU 访问模式不确定、数据结构复杂的程序，UVM 可以简化开发。但要注意 page fault 和迁移开销。

---

## 16.10 常见误区

### 误区 1：GPU 上也能用 std::vector

`std::vector` 管理的是主机内存。GPU 端没有标准库容器，需要自己实现或使用 thrust 等库。

### 误区 2：`cudaFree` 后立即可重用指针

`cudaFree` 是异步的（某些情况下）。如果立即在另一个流中分配，可能得到相同的地址，但数据不一定已清空。

### 误区 3：所有数据都应该放在 GPU

CPU 内存更大、更便宜。只有频繁访问、并行计算的数据才需要常驻 GPU。

### 误区 4：忽略纹理内存和常量内存

对于特定访问模式（如 2D 空间局部性、只读广播），纹理缓存和常量缓存比全局内存快得多。

---

## 16.11 工程建议

### 建议 1：优先使用现有库

如 thrust、cuDF、RMM 等已经实现了高效的内存管理，除非有特殊需求，否则不要自己写。

### 建议 2：显存分配要有日志和统计

在 debug 模式下记录所有分配/释放，帮助定位泄漏和异常峰值。

### 建议 3：设计时考虑对齐和合并访问

buffer 大小按 warp 大小对齐，数据布局按 coalescing 友好方式组织。

### 建议 4：区分"常驻"和"临时"

常驻数据（如模型权重）和临时数据（如中间结果）用不同的管理策略。

---

## 16.12 本章小结

这一章最重要的结论有八条。

1. RAII 是管理 GPU 内存的基本工具，应包装所有原始 CUDA 分配 API。
2. 内存池是高频动态分配的标准解决方案。
3. 环形缓冲区适合流式、流水线场景。
4. 双缓冲/多缓冲用空间换时间，隐藏传输延迟。
5. 延迟释放是处理 GPU 异步执行安全释放的关键模式。
6. 主机端 pinned memory 是稀缺资源，应谨慎使用。
7. 统一内存可以简化开发，但需要理解其隐性开销。
8. 通用 CUDA 开发的内存设计要考虑：生命周期、异步性、对齐、访问模式和主机-设备协同。

---

# 第 17 章 图形与渲染场景中的 GPU 内存管理

## 17.1 本章目标

GPU 最初就是为图形设计的。图形渲染中的内存管理有其独特的术语、约束和优化方向。

本章目标是：

1. 介绍图形渲染中的核心 GPU 内存对象。
2. 讲解 framebuffer、texture、vertex buffer 的管理特点。
3. 说明渲染管线中的显存流动。
4. 建立图形开发者视角的 GPU 内存认知。

---

## 17.2 核心结论

1. **图形渲染中的显存对象类型更多**：texture、buffer、render target、depth stencil 等各有特点。
2. **带宽和延迟敏感度高**：实时渲染要求 60-120 FPS，每帧只有 8-16ms。
3. **显存容量直接影响画质**：纹理分辨率、阴影质量、后处理效果都受显存限制。
4. **现代图形 API 给开发者更多显存控制权**：Vulkan、DX12 的手动管理 vs OpenGL 的自动管理。

---

## 17.3 直觉理解：渲染一帧需要哪些显存

想象一个 3D 游戏画面：

1. **模型数据**：顶点位置、法线、UV 坐标 → Vertex Buffer
2. **纹理**：角色皮肤、地面贴图、天空盒 → Texture
3. **渲染目标**：当前帧正在画的画面 → Framebuffer / Render Target
4. **深度缓冲**：每个像素的深度值，用于遮挡判断 → Depth Buffer
5. **后处理中间结果**：Bloom、SSAO、色调映射等 → Intermediate Textures
6. **常量数据**：相机矩阵、灯光参数 → Uniform Buffer / Constant Buffer

这些对象共同构成一帧的显存 footprint。

---

## 17.4 正式定义：核心图形显存对象

### Vertex Buffer / Index Buffer

存储模型几何数据。通常是只读的，在 GPU 顶点着色阶段读取。

### Texture / Texture Array / Cube Map

图像数据，支持硬件采样和过滤。有 1D/2D/3D 多种形式。

### Render Target（Color Attachment）

渲染输出目标，像素着色器写入的地方。可以是屏幕 framebuffer 或 offscreen texture。

### Depth/Stencil Buffer

存储每个像素的深度值和模板值，用于深度测试和模板测试。

### Uniform/Constant Buffer

存储每帧/每对象不变的常量数据，如变换矩阵、材质参数。

### Shader Storage Buffer (SSBO)

通用读写 buffer，计算着色器常用。

---

## 17.5 底层机制：现代图形 API 的内存模型

### OpenGL 风格：驱动管理

```
应用程序 -> API 调用 -> 驱动决定显存放置 -> GPU 使用
```

驱动自动管理：
- 对象创建时的显存分配
- 对象不在使用时的迁移或释放
- 多 GPU 时的复制

优点：简单。缺点：控制力弱，难以优化。

### Vulkan / DX12 风格：显式管理

```
应用程序 -> 显式分配内存 -> 显式绑定到对象 -> GPU 使用
```

开发者需要：
- 查询内存类型和堆
- 显式分配 `DeviceMemory`
- 把 buffer/image 绑定到具体的内存块
- 自己管理对象生命周期和复用

优点：完全控制，可极致优化。缺点：复杂，容易出错。

### Memory Heap 和 Memory Type

Vulkan 中，物理设备暴露多个 memory heap：

- **Device Local**：GPU 专属显存，带宽最高
- **Host Visible**：CPU 可访问，适合动态更新
- **Host Cached**：CPU 可访问且缓存，适合 CPU 读取 GPU 结果

一个 heap 可以有多个 memory type，表示不同的访问属性组合。

---

## 17.6 典型例子：一帧的显存预算

### 场景：1080p 游戏，60 FPS

| 对象类型 | 估算大小 | 说明 |
|---|---|---|
| 颜色 Buffer | 8 MB | 1920x1080 x 4 bytes (RGBA8) |
| 深度 Buffer | 8 MB | 1920x1080 x 4 bytes (D32F) |
| 后处理纹理 x3 | 24 MB | 多个中间结果 |
| 阴影贴图 | 16-64 MB | 取决于级联数量和分辨率 |
| G-Buffer (Deferred) | 32-64 MB | 多通道几何信息 |
| 纹理资源 | 500 MB-2 GB | 场景纹理总和 |
| 顶点数据 | 50-200 MB | 模型几何 |
| 总计 | 650 MB - 2.5 GB | |

这只是一个粗略估算，实际游戏可能更多或更少。

---

## 17.7 场景延伸

### 场景 1：移动设备渲染

- 显存极度受限（几百 MB 到几 GB）
- 需要 aggressive 的纹理压缩（ETC2、ASTC）
- 后处理效果受限
- 需要精细的 LOD（Level of Detail）管理

### 场景 2：VR/AR 渲染

- 需要渲染两只眼睛（stereo rendering），显存和带宽压力翻倍
- 低延迟要求极高，不能有明显卡顿
- 通常需要 foveated rendering（注视点渲染）来节省资源

### 场景 3：光线追踪

- 需要额外的加速结构（BVH、TLAS/BLAS）
- 显存占用大，构建和更新有开销
- 通常需要 denoising，增加后处理 pass

### 场景 4：引擎编辑器

- 运行时和编辑器共享 GPU
- 需要处理大量资源预览、实时编辑
- 显存碎片化更严重

---

## 17.8 常见误区

### 误区 1：纹理大小只和分辨率有关

格式影响很大：
- RGBA32F：每个像素 16 字节
- RGBA8：每个像素 4 字节
- BC7 压缩：每个像素约 1 字节

同样分辨率的纹理，显存占用可以差 16 倍。

### 误区 2：释放对象后显存立即回收

和 CUDA 类似，图形 API 的对象释放也可能是延迟的。驱动可能保留一段时间以备复用。

### 误区 3：所有 buffer 都应该在 device local heap

Host visible 内存适合需要 CPU 每帧更新的数据（如粒子系统、骨骼动画）。盲目追求 device local 反而可能因频繁迁移更慢。

### 误区 4：忽略 MSAA 的显存开销

4x MSAA 会让 color/depth buffer 的显存占用增加约 4 倍。在显存紧张时需要谨慎使用。

---

## 17.9 工程建议

### 建议 1：纹理压缩是显存优化的第一优先

ASTC、BC7 等格式在质量和压缩比上都很优秀，应作为默认选择。

### 建议 2：实施纹理流送（Texture Streaming）

不一次性加载所有纹理，根据距离和可见性动态加载和卸载 mipmap 层级。

### 建议 3：复用 render target

不同后处理阶段可以复用同一个 texture 作为输入和输出（只要依赖允许），减少总分配量。

### 建议 4：使用 transient attachment

Vulkan 的 transient attachment 允许实现把某些 buffer 放在 tile memory 中（on-chip），不占用全局显存。

### 建议 5：监控每帧显存波动

使用 RenderDoc、NVIDIA Nsight Graphics 等工具捕获帧，分析显存使用。

---

## 17.10 本章小结

这一章最重要的结论有八条。

1. 图形渲染中的显存对象类型丰富：vertex buffer、texture、render target、depth buffer 等各有特点。
2. 实时渲染对带宽和延迟极度敏感，显存管理直接影响帧率。
3. 现代 API（Vulkan、DX12）提供显式显存管理，给予开发者更大控制权但也带来更多责任。
4. Memory heap 和 memory type 的概念决定了对象应该放在哪种物理内存中。
5. 纹理压缩是图形显存优化中最有效的手段之一。
6. 纹理流送、render target 复用、transient attachment 是降低显存占用的关键策略。
7. 不是所有 buffer 都应该放在 device local memory，需要 CPU 频繁更新的数据适合 host visible。
8. 图形和计算（CUDA）可以共享 GPU，但需要注意资源竞争和上下文切换开销。

# 第 18 章 多 GPU、多进程、多租户环境

## 18.1 本章目标

前面的章节主要讨论单 GPU、单进程场景。但生产环境中，多 GPU、多进程、多租户是常态。

本章目标是：

1. 讲解多 GPU 环境下的显存管理特点。
2. 说明多进程共享 GPU 时的资源竞争和隔离问题。
3. 介绍 MIG（Multi-Instance GPU）等硬件隔离机制。
4. 建立多实体共存时的显存管理认知。

---

## 18.2 核心结论

1. **多 GPU 不等于显存简单相加**：模型并行、流水线并行会引入通信 buffer 和冗余存储。
2. **多进程共享 GPU 是危险的**：没有隔离时，一个进程可能 OOM 导致其他进程失败。
3. **MIG 提供了硬件级隔离**：把一块物理 GPU 切分成多个独立实例，各有独立的显存和计算单元。
4. **显存调度在虚拟化/云环境中变得更加复杂**：需要超售、抢占、迁移等机制。

---

## 18.3 多 GPU 显存管理

### 数据并行（Data Parallelism）

每个 GPU 持有完整的模型副本，处理不同的数据批次。

**显存特征**：
- 每卡显存占用 = 模型参数 + 该卡的激活/梯度 + 优化器状态
- 没有跨卡存储冗余（参数相同但每卡独立持有）
- 通信主要是梯度同步（all-reduce）

### 模型并行（Model Parallelism / Tensor Parallelism）

模型被切分到多个 GPU 上，每个 GPU 只持有部分参数。

**显存特征**：
- 单卡参数减少，但中间结果需要跨卡传输
- 需要通信 buffer 存储激活的切片
- 每卡可能同时持有本地参数和接收到的远程激活

### 流水线并行（Pipeline Parallelism）

模型按层切分，不同 GPU 负责不同阶段，像流水线一样传递中间结果。

**显存特征**：
- 需要存储多个 micro-batch 的中间激活（pipeline bubble）
- 显存占用与 pipeline 深度和 micro-batch 数量相关
- 可以通过 activation checkpointing 减少

### 混合并行

实际大模型训练通常组合多种并行：
- 节点内用 tensor parallelism（高带宽 NVLink）
- 节点间用 pipeline parallelism（中等带宽网络）
- 最外层用 data parallelism（扩展性最好）

**显存影响**：
- 组合策略决定了每卡实际存储哪些参数、激活和状态
- 通信 buffer 数量和大小增加
- 需要仔细计算每卡的显存预算

---

## 18.4 多进程共享 GPU

### 默认行为：无隔离

多个进程可以同时在同一块 GPU 上执行 CUDA 操作。CUDA 驱动会尝试调度，但：
- 没有显存隔离：一个进程分配过多，其他进程可能 OOM
- 没有性能隔离：一个进程占满计算单元，其他进程饥饿
- 一个进程崩溃可能影响整个 GPU 上下文

### 显存竞争的表现

- 进程 A 分配了大块显存，进程 B 的 `cudaMalloc` 失败
- 总显存超过物理容量，驱动或系统开始交换/驱逐，性能骤降
- 某个进程泄漏，最终拖垮所有共享该 GPU 的进程

### 软件级限制：CUDA_VISIBLE_DEVICES

```bash
CUDA_VISIBLE_DEVICES=0 python task1.py &
CUDA_VISIBLE_DEVICES=0 python task2.py &
```

两个进程都看到 GPU 0，但没有隔离，只是指定了用哪块卡。

### 软件级限制：显存配额（如 PyTorch max_memory_allocated）

框架层面的限制，但只是"自我约束"，无法阻止其他进程。

---

## 18.5 MIG：硬件级 GPU 虚拟化

### 核心思想

把一块物理 GPU（如 A100 80GB）切分成多个独立的 GPU 实例。

每个实例有：
- 专属的计算单元（SM）
- 专属的显存（带带宽保证）
- 独立的错误隔离

### 切分粒度

A100 支持切成 1、2、3、4、7 个实例（不同配置）。

例如：
- 1g.10gb：1/7 计算 + 10GB 显存
- 2g.20gb：2/7 计算 + 20GB 显存
- 3g.40gb：3/7 计算 + 40GB 显存
- 7g.80gb：整卡

### 对显存管理的意义

1. **真正的隔离**：一个 MIG 实例 OOM 不影响其他实例
2. **可预测的性能**：显存带宽有保证，不受邻居干扰
3. **云环境友好**：可以把一块 A100 租给 7 个用户

### 限制

- 只能用于计算（CUDA），不能用于图形（OpenGL/Vulkan）
- 切分后不能动态调整，需要重启配置
- 不是所有 GPU 都支持（需要 Ampere 及以后的专业卡）

---

## 18.6 多租户云环境

### 显存超售

和 CPU 内存超售类似，云平台可能：
- 把总显存卖给多个用户，总和超过物理显存
- 赌不是所有用户同时用满

风险：如果同时满载，会发生 OOM 或性能严重下降。

### 显存抢占和迁移

某些虚拟化方案支持：
- 暂停一个 VM/容器，把其显存状态保存到主机内存
- 把 GPU 分配给另一个租户
- 之后恢复时把状态拷回

代价：暂停和恢复有显著延迟，不适合延迟敏感任务。

### Kubernetes 中的 GPU 调度

- NVIDIA GPU Operator 提供设备插件
- 可以按整卡、MIG 实例、甚至时间切片（time-slicing）调度
- 时间切片：多个容器共享 GPU，驱动快速切换上下文

---

## 18.7 典型例子：8 卡 A100 训练集群

### 配置

- 8x A100 80GB
- NVLink + NVSwitch 全互联

### 显存计算（混合并行）

假设一个 175B 模型，用 FP16：

**纯数据并行**：
- 每卡参数：175B * 2B = 350GB → 超过单卡容量，不可行

**Tensor Parallel (TP=8) + Data Parallel (DP=1)**：
- 每卡参数：350GB / 8 = 43.75GB
- 加上激活、梯度、优化器状态，接近 80GB 上限

**TP=4 + PP=2 + DP=1**：
- TP group 内每卡参数：350GB / 4 = 87.5GB → 仍然超过

**需要 ZeRO 或更细切分**

这个例子说明：大模型训练需要精心设计的并行策略来控制每卡显存。

---

## 18.8 常见误区

### 误区 1：多 GPU 总显存 = 单卡显存 x 卡数

不是简单相加。模型并行和通信 buffer 会消耗额外显存。

### 误区 2：MIG 实例和物理 GPU 完全一样

MIG 实例有独立显存和计算，但：
- 不能跨实例直接通信（需要经过主机）
- 没有 NVLink 互联（实例间）
- 图形 API 不支持

### 误区 3：时间切片等于真正的并行

时间切片是快速切换，不是同时执行。多个时间切片任务的总吞吐通常低于单独执行。

### 误区 4：多进程安全只要框架处理好就行

框架层面的安全无法阻止其他进程或裸 CUDA 程序的竞争。

---

## 18.9 工程建议

### 建议 1：生产环境优先使用 MIG 或整卡分配

避免无隔离的多进程共享，除非能完全信任所有进程。

### 建议 2：设计并行策略时，先算显存账

不要先定并行策略再算显存，而是先算每卡能放下什么，再决定怎么并行。

### 建议 3：监控每卡显存，不只是总和

多卡场景中，一张卡 OOM 就会导致整个任务失败，即使其他卡还有余量。

### 建议 4：预留通信 buffer 显存

all-reduce、all-gather 等集合通信需要临时 buffer，要在显存预算中预留。

### 建议 5：考虑 NCCL 的显存使用

NCCL 内部也会分配 buffer 用于通信，大集群中这部分不可忽视。

---

## 18.10 本章小结

这一章最重要的结论有八条。

1. 多 GPU 并行策略（数据、模型、流水线）对每卡显存占用有根本性影响。
2. 多进程默认共享 GPU 没有隔离，一个进程的 OOM 可能影响所有进程。
3. MIG 提供硬件级隔离，是多租户环境中最安全的共享方式。
4. 显存超售在云环境中常见，但有满载时性能崩溃的风险。
5. 时间切片是软件级共享，不是真正的并行，总吞吐会下降。
6. 设计分布式训练时，应先计算每卡显存预算，再确定并行策略。
7. 通信 buffer（NCCL、自定义）需要在显存规划中预留。
8. 多 GPU 显存管理的核心是：隔离、可预测性、和每卡预算的精确计算。

---

# 第 19 章 常见故障与误区

## 19.1 本章目标

本章收集 GPU 内存管理中最常见、最迷惑人的故障和误区。

目标是：

1. 列出高频故障现象和根因。
2. 解释为什么某些"看起来对"的做法实际上是错的。
3. 帮助读者在遇到问题时快速定位方向。

---

## 19.2 核心结论

1. **OOM 不一定是因为"真的没内存"**：可能是碎片、最大分配限制、或地址空间耗尽。
2. **显存泄漏和缓存保留看起来很像**：但前者是 bug，后者是策略。
3. **异步执行让问题更难调试**：错误可能在远离根因的地方暴露。
4. **工具看到的显存数字不一定是你以为的意思**：reserved、allocated、active 是不同概念。

---

## 19.3 故障一：CUDA out of memory

### 现象

```
RuntimeError: CUDA out of memory. Tried to allocate X GB
```

### 可能原因（按频率排序）

1. **真的没内存了**：活跃对象总和超过物理容量
2. **碎片导致分配失败**：空闲总量够，但没有连续大块
3. **单次分配超过限制**：某些系统有单块分配上限
4. **地址空间耗尽**：32 位进程或某些驱动的限制
5. **其他进程抢了显存**：多进程共享 GPU 时

### 排查方向

- 先用 `nvidia-smi` 看总体使用情况
- 再查当前进程的显存分配明细
- 检查是否有异常大的 tensor 或临时对象
- 看是否可复用的对象没有被释放

---

## 19.4 故障二：显存泄漏

### 现象

程序运行时间越长，显存占用越高，最终 OOM。

### 与缓存保留的区别

| 特征 | 显存泄漏 | 缓存保留 |
|---|---|---|
| 趋势 | 持续增长 | 增长到上限后稳定 |
| 释放后 | 仍占用 | 仍在池中，可复用 |
| 对象引用 | 存在未释放引用 | 框架/池持有引用 |
| 重启后 | 恢复初始 | 恢复初始 |

### 常见泄漏原因

1. **循环中创建 tensor 没有释放引用**：Python 循环变量持有引用
2. **异常路径没有释放**：C++ 中 `cudaFree` 在异常前未执行
3. **回调闭包捕获了大对象**：事件回调持有不必要的引用
4. **全局缓存无限增长**：没有 eviction 策略的自定义缓存

---

## 19.5 故障三：非法内存访问（Illegal Memory Access）

### 现象

```
RuntimeError: CUDA error: an illegal memory access was encountered
```

### 根因

- Kernel 越界访问：索引计算错误，访问了未分配的地址
- Use-after-free：对象已释放，但 kernel 还在使用
- 主机端指针传到设备端使用
- 多 GPU 时张量放在卡 A，kernel 在卡 B 执行

### 为什么难调试

因为 GPU 执行是异步的。错误可能在 kernel 启动很久后才报出来，调用栈指向的是后来的某个 CUDA 操作。

### 调试方法

- 设置 `CUDA_LAUNCH_BLOCKING=1` 强制同步，定位具体 kernel
- 使用 `cuda-memcheck` 或 compute-sanitizer
- 检查所有指针和索引计算

---

## 19.6 故障四：数据损坏（Silent Data Corruption）

### 现象

程序不崩溃，但结果错误。或者偶尔错误，难以复现。

### 可能原因

- 内存竞争：多个流同时读写同一块内存
- Use-after-free：释放后重用，数据被覆盖
- 越界写：写到了相邻对象，破坏了其他数据
- 精度问题：FP16 溢出、下溢、或非确定性计算

### 为什么危险

不崩溃的错误最难发现，可能在生产环境中潜伏很久。

---

## 19.7 误区一：`cudaFree` 后立即可用

### 错误认知

`cudaFree(ptr)` 后，`ptr` 指向的内存已经可以被重新分配和使用。

### 真相

`cudaFree` 是异步的（在默认流中）。如果其他流正在使用这块内存，或者驱动内部有延迟释放机制，立即重用可能导致未定义行为。

### 正确做法

- 确保所有使用该内存的 kernel 已完成
- 或在释放前同步相关流

---

## 19.8 误区二：显存占用高就是内存泄漏

### 错误认知

看到 `nvidia-smi` 中进程显存高，就认为有泄漏。

### 真相

很多框架（PyTorch、TensorFlow）会缓存释放的显存以便快速复用。这些显存在框架视角是"空闲"的，但在系统视角仍显示为进程占用。

### 正确做法

- 区分 reserved 和 allocated
- 看趋势：持续增长才是泄漏，稳定在高平台是缓存
- 使用框架工具查看池内空闲量

---

## 19.9 误区三：统一内存可以当普通内存用

### 错误认知

用了 `cudaMallocManaged` 就不需要关心 CPU/GPU 数据传输了。

### 真相

UVM 自动管理迁移，但：
- page fault 有开销
- 频繁在 CPU/GPU 间来回访问的数据会很慢
- 预取和提示仍然需要手动优化

### 正确做法

- UVM 适合访问模式不确定或开发阶段
- 性能关键路径仍应显式管理数据传输

---

## 19.10 误区四：零拷贝内存一定更快

### 错误认知

Zero-copy（mapped pinned memory）省去了显式拷贝，所以更快。

### 真相

零拷贝内存通过 PCIe 直接访问，每次访问都是 PCIe 事务。对于频繁访问的数据，这比一次性拷贝到显存慢得多。

### 正确做法

- 零拷贝适合：只读一次、或低频访问的数据
- 频繁访问的数据应该显式拷贝到 device memory

---

## 19.11 误区五：batch size 越大效率越高

### 错误认知

训练或推理时，batch size 越大，GPU 利用率越高，效率越好。

### 真相

- batch size 增大确实提高计算效率（amortize 固定开销）
- 但显存占用也增大，可能导致 OOM
- 某些算法在过大 batch 时收敛性变差（训练）
- 推理时 batch size 增大增加延迟

### 正确做法

- 找到计算效率和显存容量的平衡点
- 使用 gradient accumulation 在有限显存内模拟大 batch

---

## 19.12 本章小结

这一章最重要的结论有十条。

1. OOM 有多种原因，不只是"没内存"，碎片和限制也是常见根因。
2. 显存泄漏和缓存保留的区别在于趋势和可复用性。
3. 非法内存访问通常由越界、use-after-free、或设备/主机指针混淆引起。
4. 异步执行让 GPU 错误难以定位，需要同步工具辅助调试。
5. 静默数据损坏是最危险的错误，不崩溃但结果错误。
6. `cudaFree` 不保证立即物理释放，异步场景需要同步。
7. 高显存占用不等于泄漏，框架缓存是正常行为。
8. 统一内存方便但不免费，频繁迁移有显著开销。
9. 零拷贝内存适合低频访问，高频访问应显式拷贝到设备。
10. batch size 不是越大越好，需要在效率、显存和延迟间平衡。

---


# 第 20 章 排障方法与工具

## 20.1 本章目标

遇到 GPU 内存问题时，知道用什么工具、按什么顺序排查，比盲目猜测高效得多。

本章目标是：

1. 介绍 GPU 内存问题排查的系统方法。
2. 讲解常用工具的使用场景和输出解读。
3. 建立从现象到根因的排查路径。

---

## 20.2 核心结论

1. **先确认现象，再定位根因**：是 OOM？泄漏？越界？还是性能问题？
2. **工具分层使用**：系统级工具（nvidia-smi）→ 框架级工具（PyTorch memory stats）→ 专用工具（Nsight、compute-sanitizer）
3. **同步是调试异步 GPU 的钥匙**：`CUDA_LAUNCH_BLOCKING=1` 能把异步错误定位到具体调用。

---

## 20.3 排查流程图

```
遇到问题
  │
  ├─ OOM ───────────────────────────────┐
  │   ├─ nvidia-smi 看总体占用           │
  │   ├─ 看是否多进程竞争                │
  │   ├─ 框架工具看 allocated vs reserved│
  │   ├─ 检查是否有异常大对象            │
  │   └─ 怀疑碎片 → 减小分配粒度或整理   │
  │
  ├─ 疑似泄漏 ──────────────────────────┤
  │   ├─ 看趋势：持续增长 vs 稳定平台    │
  │   ├─ 框架工具看 active 对象          │
  │   ├─ 检查循环、闭包、全局缓存        │
  │   └─ 对比重启前后的基线              │
  │
  ├─ 非法访问 / 崩溃 ───────────────────┤
  │   ├─ CUDA_LAUNCH_BLOCKING=1          │
  │   ├─ compute-sanitizer               │
  │   ├─ 检查指针、索引、设备一致性      │
  │   └─ 检查 use-after-free             │
  │
  └─ 性能问题 ──────────────────────────┘
      ├─ Nsight Systems / Nsight Compute
      ├─ 看内存带宽利用率
      ├─ 看是否有不必要的传输
      └─ 看 coalescing 和 bank conflict
```

---

## 20.4 系统级工具

### nvidia-smi

最基础的工具，看 GPU 整体状态。

```bash
nvidia-smi
nvidia-smi dmon  # 持续监控
nvidia-smi pmon  # 按进程监控
```

关键字段：
- `Memory-Usage`：已用 / 总量
- `Volatile GPU-Util`：计算利用率（不是显存利用率！）
- `Processes`：每个进程的显存占用

### 局限

- 只能看到进程级粒度，看不到进程内部分配细节
- 看不到框架缓存和实际活跃对象的区别

---

## 20.5 框架级工具（以 PyTorch 为例）

### torch.cuda.memory_summary()

```python
print(torch.cuda.memory_summary(device=0))
```

输出包括：
- `Allocated memory`：实际张量占用
- `Reserved memory`：缓存池总大小
- `Active memory`：当前活跃（有引用）的张量
- `Pool fragmentation`：池内碎片比例

### torch.cuda.memory_stats()

更详细的字典格式，适合程序化分析。

### torch.profiler

```python
with torch.profiler.profile(
    activities=[torch.profiler.ProfilerActivity.CPU,
                torch.profiler.ProfilerActivity.CUDA],
    profile_memory=True,
) as prof:
    # 你的代码
    pass
print(prof.key_averages().table(sort_by="cuda_memory_usage"))
```

可以看到每个算子的显存分配情况。

---

## 20.6 CUDA 专用工具

### compute-sanitizer

前身是 `cuda-memcheck`，检测：
- `memcheck`：越界访问、未初始化访问
- `racecheck`：数据竞争
- `initcheck`：未初始化访问
- `synccheck`：同步错误

```bash
compute-sanitizer --tool memcheck ./your_program
```

### Nsight Systems

系统级性能分析，看时间线：
- kernel 执行
- 内存传输（H2D、D2H、D2D）
- 同步点
- API 调用

适合回答"时间都去哪了"。

### Nsight Compute

kernel 级微观分析：
- 内存带宽利用率
- coalescing 效率
- cache hit rate
- 指令级分析

适合回答"这个 kernel 为什么不够快"。

---

## 20.7 调试技巧

### 技巧 1：CUDA_LAUNCH_BLOCKING=1

强制所有 CUDA 调用同步执行，错误会立即在调用点抛出，而不是延迟到后来的某个同步点。

代价：性能大幅下降，仅用于调试。

### 技巧 2：逐步缩小范围

如果程序复杂，用二分法：
- 注释掉一半代码，看问题是否还在
- 逐步缩小到最小复现单元

### 技巧 3：对比基线

- 小模型 vs 大模型
- 小 batch vs 大 batch
- 旧版本代码 vs 新版本代码

### 技巧 4：记录分配日志

在 debug 模式下，记录每次 `cudaMalloc`/`cudaFree` 的地址、大小、调用栈。

## 20.8 典型排查案例

### 案例 1：PyTorch OOM，但 nvidia-smi 显示还有空闲

可能原因：
- 碎片：空闲总量够，但没有连续大块
- 其他进程抢了显存
- 单次分配请求超过剩余最大连续块

排查：
```python
print(torch.cuda.memory_summary())
# 看 reserved - allocated 是否很大（缓存多）
# 看是否有大块分配失败记录
```

### 案例 2：训练过程中显存缓慢增长

可能原因：
- 泄漏：某些张量没有被释放
- 缓存策略：框架保留更多缓存
- 数据变长：序列长度逐渐增加

排查：
- 记录每步的 `torch.cuda.memory_allocated()`
- 看增长趋势是线性、阶梯式、还是突发
- 检查数据加载器是否产生越来越大的 batch

### 案例 3：非法内存访问，但调用栈指向无辜代码

原因：异步执行导致错误延迟暴露。

排查：
```bash
CUDA_LAUNCH_BLOCKING=1 python train.py
```

错误现在会指向真正的 culprit kernel。

## 20.9 工程建议

### 建议 1：建立显存监控基线

记录正常情况下的：
- 启动后基线显存
- 每步活跃显存
- 峰值显存
- 缓存比例

异常时对比基线。

### 建议 2：自动化测试中加入显存检查

```python
# 测试结束后检查是否有异常增长
final_mem = torch.cuda.memory_allocated()
assert final_mem - initial_mem < threshold
```

### 建议 3：生产环境暴露显存指标

通过 Prometheus/Grafana 等监控：
- GPU 显存使用率
- 每进程显存占用
- OOM 事件次数

---

## 20.10 本章小结

这一章最重要的结论有八条。

1. GPU 内存排查要分层：系统级 → 框架级 → 专用工具。
2. `nvidia-smi` 是基础，但看不到进程内细节。
3. PyTorch 的 `memory_summary` 和 `profiler` 是训练场景的首选工具。
4. `compute-sanitizer` 是检测越界和竞争的金标准。
5. `CUDA_LAUNCH_BLOCKING=1` 是定位异步错误的关键技巧。
6. 排查时先确认现象类别（OOM、泄漏、崩溃、性能），再选工具。
7. 建立正常基线，异常时对比，能大幅缩短定位时间。
8. 生产环境应持续监控显存指标，而不是只在出问题后查看。

# 第 21 章 实战优化清单

## 21.1 本章目标

前面章节讲了很多原理和方法。本章把它们压缩成可直接对照执行的清单。

目标是：

1. 提供训练场景的显存优化 checklist。
2. 提供推理场景的显存优化 checklist。
3. 提供通用 CUDA 开发的显存优化 checklist。
4. 让清单可以直接用于代码审查或设计评审。

---

## 21.2 核心结论

1. **优化是有优先级的**：先做高收益低代价的，再做低收益高代价的。
2. **测量先于优化**：不知道瓶颈在哪，优化可能是无效的甚至负面的。
3. **没有放之四海而皆优的策略**：根据场景选择组合。

---

## 21.3 训练场景优化清单

### 第一阶段：最高优先级（必做）

- [ ] 启用混合精度训练（FP16/BF16）
- [ ] 检查 batch size 是否合理（不是越大越好）
- [ ] 确认没有意外的大张量或广播膨胀
- [ ] 使用框架推荐的默认内存池配置

### 第二阶段：高优先级（通常要做）

- [ ] 启用 activation checkpointing（如果模型深、序列长）
- [ ] 评估 optimizer state sharding（ZeRO/FSDP，大模型必做）
- [ ] 检查数据加载器是否产生过大或不均匀的 batch
- [ ] 清理不必要的中间变量和梯度保留

### 第三阶段：中优先级（按需做）

- [ ] 考虑 gradient accumulation（在有限显存内模拟大 batch）
- [ ] 评估 offload（CPU/NVMe）是否值得
- [ ] 检查通信 buffer 大小（分布式场景）
- [ ] 启用 CUDA graph 捕获（如果图形态稳定）

### 第四阶段：低优先级（深度优化）

- [ ] 自定义内存池策略
- [ ] 算子融合减少中间激活
- [ ] 手动管理某些大对象的生命周期
- [ ] 考虑模型并行或流水线并行的重新划分

---

## 21.4 推理场景优化清单

### 第一阶段：最高优先级（必做）

- [ ] 明确 KV cache 的显存预算和公式估算
- [ ] 检查 batch size 与延迟/显存的平衡点
- [ ] 确认模型权重精度是否可接受（FP16 vs INT8 vs INT4）

### 第二阶段：高优先级（通常要做）

- [ ] 评估 KV cache 量化
- [ ] 使用 PagedAttention 或等效技术（流式服务）
- [ ] 启用连续批处理（在线服务）
- [ ] 检查输入长度分布，设置合理的最大长度截断

### 第三阶段：中优先级（按需做）

- [ ] 实现请求级别的显存配额和拒绝策略
- [ ] 优化 prefill/decode 的调度策略
- [ ] 评估 speculative decoding（用小模型草稿加速）
- [ ] 长文本场景的 KV cache 压缩或驱逐策略

### 第四阶段：低优先级（深度优化）

- [ ] 自定义 kernel 减少 workspace
- [ ] 模型结构层面的 KV cache 优化（如 GQA/MQA）
- [ ] 多 GPU tensor parallelism 的显存优化
- [ ] 专用硬件（如 TPUs 的显存优化策略）

---

## 21.5 通用 CUDA/C++ 开发优化清单

- [ ] 所有显存分配使用 RAII 包装
- [ ] 高频小分配使用内存池
- [ ] 主机端 pinned memory 按需使用，不过度
- [ ] 数据传输与计算重叠（异步、流）
- [ ] 检查内存对齐（256 字节或更大）
- [ ] 检查 coalescing 友好的数据布局
- [ ] 区分常驻数据和临时数据的管理策略
- [ ] 延迟释放确保异步安全
- [ ] 流式数据使用环形缓冲区
- [ ] 监控分配/释放的日志和统计

---

## 21.6 图形渲染优化清单

- [ ] 纹理使用压缩格式（ASTC/BC7/ETC2）
- [ ] 实施纹理流送和 LOD
- [ ] 复用 render target 和中间 buffer
- [ ] 使用 transient attachment 减少全局显存
- [ ] 检查 MSAA 级别是否过度
- [ ] 按需选择 memory heap（device local vs host visible）
- [ ] 监控每帧显存波动和峰值
- [ ] 移动平台特别 aggressive 地控制显存

---

## 21.7 优化优先级通用原则

### 收益/代价矩阵

| 策略 | 收益 | 代价 | 优先级 |
|---|---|---|---|
| 混合精度 | 高 | 低 | 必做 |
| checkpointing | 高 | 中 | 高 |
| 量化 | 高 | 中 | 高 |
| PagedAttention | 高 | 中 | 高 |
| ZeRO/FSDP | 高 | 中 | 高 |
| offload | 中 | 高 | 中 |
| 自定义内存池 | 中 | 高 | 低 |
| 算子融合 | 中 | 高 | 低 |

### 原则

1. 先做"高收益低代价"
2. 测量验证每次优化的实际效果
3. 不要同时做多个大改动，否则无法归因
4. 某些优化有兼容性代价（如 INT4 需要特定 kernel），要评估是否值得

---

## 21.8 本章小结

这一章最重要的结论有六条。

1. 优化应按优先级分阶段进行，不要一次性做所有事。
2. 训练场景最高优先级：混合精度、checkpointing、optimizer sharding。
3. 推理场景最高优先级：KV cache 预算、量化、分页缓存、连续批处理。
4. 通用 CUDA 开发的核心：RAII、内存池、异步、对齐、coalescing。
5. 图形渲染的核心：纹理压缩、流送、复用、transient attachment。
6. 测量先于优化，验证每次改动的实际收益。

# 第 22 章 统一心智模型：一切从"账单"出发

## 22.1 本章目标

这是最后一章。目标是把前面 21 章的内容压缩成一个统一的心智模型，让你在面对任何 GPU 内存问题时，都能快速找到分析角度。

---

## 22.2 核心结论

**GPU 内存管理的一切问题，都可以从"账单"角度理解。**

知道：
1. 谁（什么对象）
2. 占了多少（容量）
3. 占多久（生命周期）
4. 放在哪（物理位置）
5. 怎么访问（访问模式）

就能回答绝大多数 GPU 内存问题。

---

## 22.3 心智模型：五维账单

### 维度 1：对象（Who）

GPU 显存中的对象类型：
- 模型参数
- 梯度
- 优化器状态
- 激活值 / KV cache
- 临时 workspace
- 通信 buffer
- 框架/系统开销

**问题**：能不能减少某类对象？能不能让某类对象更小？

### 维度 2：容量（How Much）

每个对象占多少字节：
- 元素个数 x 位宽 / 8
- 对齐填充
- 分配器粒度浪费

**问题**：位宽能不能降低？对齐能不能优化？分配粒度能不能更细？

### 维度 3：生命周期（How Long）

对象从创建到销毁的时间：
- 常驻（整个程序生命周期）
- 阶段级（一个 epoch、一个请求）
- 临时（一个算子、一个 kernel）

**问题**：能不能缩短生命周期？能不能让不重叠的对象复用同一块内存？

### 维度 4：物理位置（Where）

对象存在哪里：
- GPU 全局内存（device memory）
- GPU 共享内存（shared memory）
- 主机内存（pageable / pinned）
- 统一内存（managed）
- 其他存储（NVMe、网络）

**问题**：能不能把对象移到更便宜/更大的存储？迁移开销是否值得？

### 维度 5：访问模式（How）

对象怎么被访问：
- 顺序 / 随机
- 读多写少 / 读写均衡
- 合并访问 / 分散访问
- 只读 / 读写

**问题**：访问模式是否匹配存储类型的优势？能不能重构数据布局提高带宽利用率？

---

## 22.4 用这个模型分析任何问题

### 例子 1：训练 OOM

- 对象：激活值太大
- 容量：batch size x 序列长度 x 隐藏维度 x 层数
- 生命周期：前向传播期间必须存在
- 位置：GPU 显存
- 访问模式：顺序读写

优化方向：
- 减少容量：减小 batch size、混合精度
- 缩短生命周期：activation checkpointing
- 改变位置：offload 到 CPU

### 例子 2：推理服务显存不足

- 对象：KV cache
- 容量：与并发请求数 x 平均序列长度成正比
- 生命周期：请求开始到结束
- 位置：GPU 显存
- 访问模式：追加写、随机读（注意力计算）

优化方向：
- 减少容量：KV cache 量化、GQA/MQA
- 缩短生命周期：请求完成立即释放（分页缓存）
- 提高分配效率：PagedAttention 减少碎片和过度预留

### 例子 3：CUDA 程序带宽低

- 对象：输入数据
- 容量：很大
- 生命周期：kernel 执行期间
- 位置：GPU 全局内存
- 访问模式：分散访问，coalescing 不好

优化方向：
- 改变访问模式：重构数据布局，让线程按合并模式访问
- 改变位置：使用共享内存缓存
- 减少容量：压缩数据

---

## 22.5 从五维账单到决策树

```
遇到显存问题
  │
  ├─ 先问：什么对象占最多？
  │   └─ 用工具 profiling 找出大头
  │
  ├─ 再问：能不能减少它的容量？
  │   └─ 量化、压缩、减小 batch、截断长度
  │
  ├─ 再问：能不能缩短它的生命周期？
  │   └─ checkpointing、及时释放、复用
  │
  ├─ 再问：能不能放到更便宜的位置？
  │   └─ offload、统一内存、梯度/状态 sharding
  │
  └─ 最后问：访问模式是否最优？
      └─ coalescing、共享内存、缓存友好布局
```

---

## 22.6 一些永恒的权衡

GPU 内存管理中没有免费午餐，常见的权衡：

| 你想要 | 你可能需要付出 |
|---|---|
| 更少显存 | 更多计算、更多通信、或更低精度 |
| 更快分配 | 更多碎片、或更多预留 |
| 更少碎片 | 更复杂的分配策略、或更多内部浪费 |
| 更大容量 | 更慢的存储层级、或更多传输 |
| 更高带宽 | 更严格的数据布局约束、或更多共享内存管理 |
| 编程简单 | 运行时自动管理的隐性开销 |

理解这些权衡，就不会追求"完美"，而是追求"适合当前场景的平衡"。

---

## 22.7 给不同角色的建议

### 算法研究员

- 理解显存账单结构，在设计模型时考虑推理效率
- 知道不同精度、不同注意力结构的显存影响
- 能和工程团队用共同语言讨论资源约束

### 系统工程师

- 掌握分配器原理、并行策略、和监控工具
- 能定位 OOM、泄漏、和性能瓶颈
- 理解框架底层行为和系统行为的差异

### 图形开发者

- 精通现代图形 API 的显存模型
- 掌握纹理压缩、流送、和 render target 复用
- 能在画质和性能间做可控的 trade-off

### 全栈开发者

- 建立"五维账单"的分析习惯
- 知道什么时候该深入，什么时候该用现有框架
- 理解从硬件到应用的完整链条

---

## 22.8 本章小结

这一章最重要的结论有六条。

1. GPU 内存管理的一切问题都可以从"五维账单"理解：对象、容量、生命周期、位置、访问模式。
2. 遇到问题时，按"谁占最多 → 能不能减少 → 能不能缩短 → 能不能换位置 → 访问是否最优"的顺序分析。
3. 所有优化都是权衡：省显存通常要换计算、通信、精度、或复杂度。
4. 不同角色需要不同深度的知识，但"账单思维"是通用的。
5. 工具和方法会演进，但底层的权衡和维度是稳定的。
6. 最好的优化是：在正确的时间，对正确的对象，做正确的决策。

# 附录 A：术语表

| 术语 | 英文 | 解释 |
|---|---|---|
| 显存 | VRAM / Video Memory | GPU 专属的高速内存，用于存储模型参数、激活值、纹理等 |
| 全局内存 | Global Memory | GPU 上最大的内存空间，所有线程可访问，延迟较高 |
| 共享内存 | Shared Memory | SM 上的高速片上内存，同 block 线程共享，需手动管理 |
| 寄存器 | Register | 每个线程私有的最快存储，容量极小 |
| 常量内存 | Constant Memory | 只读广播内存，适合同 warp 线程访问相同地址 |
| 纹理内存 | Texture Memory | 带缓存的只读内存，适合 2D 空间局部性访问 |
| 页锁定内存 | Pinned Memory / Page-Locked Memory | 主机端不能被换出的内存，支持异步 DMA 传输 |
| 统一内存 | Unified Memory / Managed Memory | CPU/GPU 共享的虚拟地址空间，自动迁移 |
| UVA | Unified Virtual Addressing | CUDA 4.0 引入的统一虚拟地址空间 |
| UVM | Unified Virtual Memory | 带自动页迁移的统一内存，需要 Pascal+ |
| 零拷贝 | Zero-Copy | 主机内存映射到设备地址空间，直接通过 PCIe 访问 |
| 合并访问 | Coalesced Access | warp 内线程访问连续地址，合并为最少事务 |
| bank conflict | Bank Conflict | 共享内存中多个线程访问同一 bank 的不同地址，导致串行化 |
| OOM | Out of Memory | 显存不足，分配失败 |
| 碎片 | Fragmentation | 空闲内存不连续，导致大块分配失败 |
| 内部碎片 | Internal Fragmentation | 已分配块内的未使用空间 |
| 外部碎片 | External Fragmentation | 分散的小空闲块无法合并满足大块请求 |
| 内存池 | Memory Pool | 预先分配大块内存，内部管理分配和回收 |
| RAII | Resource Acquisition Is Initialization | 资源获取即初始化，用对象生命周期管理资源 |
